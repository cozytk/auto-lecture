# VCS 비교: 왜 Git인가?

| 구분 | 파일 복사 | SVN (중앙집중) | **Git (분산)** |
|------|----------|---------------|---------------|
| 이력 관리 | 파일명 구분 | 중앙 서버 기록 | **로컬+원격** |
| 협업 | 충돌 빈발 | 서버 필수 | **오프라인 OK** |
| 속도 | - | 느림 | **빠름** |
| 복구 | 불가능 | 서버 의존 | **완전 복제** |

<v-clicks>

<div class="mt-4 p-3 bg-blue-900/30 rounded-lg">
<strong>핵심</strong>: Git은 로컬에 전체 이력을 가지므로 네트워크 없이도 커밋, 브랜치, 로그 확인이 가능
</div>

<div class="mt-2 p-3 bg-amber-900/30 rounded-lg">
💡 <strong>Git ≠ GitHub</strong>: Git은 도구, GitHub는 호스팅 서비스. Git 없이 GitHub 불가, GitHub 없이 Git 가능
</div>

</v-clicks>

<!--
[스크립트]
방금 게임 세이브 비유로 버전관리의 원리를 이해했습니다. 그런데 버전관리 도구가 Git만 있는 건 아닙니다. 어떤 선택지가 있고, 왜 Git을 쓰는지 비교해 보겠습니다.

화면에 표가 보입니다. 세 가지 방식을 네 가지 기준으로 비교하고 있습니다.

첫 번째 열, 파일 복사 방식입니다. 방금 본 `report_최종_수정3.docx` 방식이에요. 이력 관리는 파일명에 의존하고, 협업하면 충돌이 빈번하며, 복구는 사실상 불가능합니다.

두 번째 열, SVN이라고 불리는 중앙집중형 VCS입니다. 중앙 서버 하나에 이력을 기록합니다. 파일 복사보다는 훨씬 나은데, 문제가 있습니다. 서버에 연결이 안 되면 커밋도 못 하고 이력도 못 봅니다. 서버가 날아가면 이력도 함께 사라질 수 있습니다.

세 번째 열, Git입니다. 볼드로 강조되어 있죠. 이력을 로컬과 원격 모두에 기록합니다. 오프라인에서도 커밋할 수 있고, 속도가 빠르며, 모든 복제본이 전체 이력을 가지고 있어서 복구도 안전합니다.

[click] 파란 박스를 보십시오. 핵심 포인트입니다. Git은 로컬에 전체 이력을 가지므로 네트워크 없이도 커밋, 브랜치, 로그 확인이 가능합니다. 비행기 안에서도 코드 작업을 하고 커밋을 남길 수 있다는 뜻입니다. SVN에서는 불가능한 일이에요.

Q: SVN을 쓰던 팀인데 Git으로 바꿔야 하나요?
A: 현재 업계 표준은 Git입니다. SVN에서 Git으로 전환하는 가장 큰 이유는 두 가지예요. 첫째, 오프라인 작업이 가능합니다. 둘째, 브랜치를 만드는 비용이 거의 없습니다. SVN에서 브랜치 하나 만들려면 서버에 전체 복사본을 만들어야 하는데, Git은 포인터 하나 만드는 수준이라 1초도 안 걸립니다. 이 차이가 개발 워크플로우를 완전히 바꿔 놓습니다.

Q: "Git은 백업 도구다"라고 생각해도 되나요?
A: 아닙니다. Git은 백업이 아니라 변경 이력 추적 도구입니다. 물론 원격 저장소에 올리면 백업 효과도 있지만, Git의 핵심 가치는 "언제, 누가, 왜 바꿨는지"를 추적하는 것입니다. 단순 백업이 목적이라면 클라우드 스토리지로 충분합니다. Git은 그 이상의 도구입니다.
