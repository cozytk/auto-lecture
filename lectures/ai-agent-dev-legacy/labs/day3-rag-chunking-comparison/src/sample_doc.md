# RAG 완전 가이드

RAG(Retrieval-Augmented Generation)는 대규모 언어 모델의 한계를 보완하는 핵심 기술이다. LLM은 학습 데이터 이후의 정보를 알지 못하고, 환각(Hallucination)을 생성하는 경향이 있다. RAG는 이 두 가지 문제를 외부 지식 검색으로 해결한다.

## RAG의 핵심 구성 요소

RAG 시스템은 크게 두 가지 파이프라인으로 구성된다. 첫 번째는 인덱싱 파이프라인이고, 두 번째는 검색-생성 파이프라인이다.

인덱싱 파이프라인에서는 문서를 Chunk로 분할하고, 각 Chunk를 임베딩 벡터로 변환한 뒤 벡터 데이터베이스에 저장한다. 이 과정은 사전에 오프라인으로 수행된다.

검색-생성 파이프라인에서는 사용자 쿼리를 임베딩하고, 벡터 DB에서 유사한 Chunk를 검색한 뒤, 검색된 Chunk를 컨텍스트로 LLM에 전달하여 최종 답변을 생성한다.

## Chunking 전략

Chunking은 대용량 문서를 검색 가능한 단위로 분할하는 과정이다. Chunk의 크기와 방식은 검색 품질에 직접적인 영향을 미친다.

### Fixed-size Chunking

Fixed-size Chunking은 텍스트를 고정된 크기(토큰 수 또는 문자 수)로 분할한다. 구현이 단순하고 모든 Chunk의 크기가 일정하다는 장점이 있다.

그러나 문장이나 단락의 중간에서 분할이 발생할 수 있어 맥락이 끊기는 문제가 있다. 이를 보완하기 위해 인접한 Chunk 간에 일부 내용을 겹치게 하는 Overlap을 적용한다. 일반적으로 50~100 토큰의 Overlap을 권장한다.

### Semantic Chunking

Semantic Chunking은 의미적 경계(문장, 단락)에서 텍스트를 분할한다. 자연스러운 맥락 단위로 분할되어 검색 품질이 높다.

분할 우선순위는 단락 구분("\n\n") > 줄 바꿈("\n") > 문장 끝(".", "!", "?") 순서로 설정한다. 이 방식은 Chunk 크기가 일정하지 않지만, 의미 완결성이 높다.

### Document-aware Chunking

Document-aware Chunking은 문서의 구조(헤더, 섹션, 표 등)를 인식하여 분할한다. Markdown이나 HTML 문서에 적합하다.

헤더(#, ##, ###) 단위로 분할하면 각 Chunk가 하나의 완결된 섹션을 담게 된다. 이 방식은 구조화된 문서에서 가장 높은 검색 정확도를 보인다.

### Hierarchical Chunking

Hierarchical Chunking은 여러 레벨의 Chunk를 동시에 유지한다. 대단원 수준의 큰 Chunk는 개요 검색에 사용하고, 소단원 수준의 작은 Chunk는 세부 검색에 사용한다.

쿼리의 종류(개요 질문 vs 세부 질문)에 따라 적절한 레벨의 Chunk를 선택하므로 가장 유연하다. 다만 구현 복잡도가 높고 저장 공간이 많이 필요하다.

## Embedding 모델

Embedding은 텍스트를 고차원 벡터 공간으로 변환하는 과정이다. 의미적으로 유사한 텍스트는 벡터 공간에서 가까운 위치에 놓인다.

### 모델 선택 기준

2026년 기준 주요 Embedding 모델은 다음과 같다. OpenAI의 text-embedding-3-large는 3072차원으로 영어 성능이 우수하다. 한국어 서비스에는 multilingual-e5-large나 bge-m3가 적합하다. Claude와 함께 사용할 때는 Anthropic이 권장하는 voyage-3가 최적화되어 있다.

차원 수가 높을수록 정확도가 높지만 저장 비용도 증가한다. Matryoshka Representation Learning(MRL)을 지원하는 모델은 차원을 동적으로 조정할 수 있어 비용과 정확도의 균형을 맞출 수 있다.

## Retrieval 튜닝

검색 품질은 세 가지 파라미터로 조정한다: Top-k, Similarity Threshold, Re-ranking.

### Top-k 설정

Top-k는 검색 결과로 반환할 Chunk의 수다. k가 작으면 정확하지만 관련 정보를 놓칠 수 있고, k가 크면 커버리지가 높지만 노이즈가 증가한다.

일반적인 QA 시스템에서는 k=5~10이 적절한 균형점이다. 포괄적인 요약이 필요한 경우 k=20 이상을 사용하지만, LLM 컨텍스트 비용이 증가한다.

### Similarity Threshold

Similarity Threshold는 검색 결과를 필터링하는 유사도 기준값이다. 이 값 미만의 결과는 제거한다.

코사인 유사도 기준으로 0.65~0.75가 일반적인 권장 범위다. 도메인이나 임베딩 모델에 따라 최적값이 달라지므로 실험을 통해 조정해야 한다.

### Re-ranking

Re-ranking은 Bi-encoder로 검색한 결과를 Cross-encoder로 재정렬하는 과정이다. Bi-encoder는 빠르지만 쿼리와 문서의 상호작용을 포착하지 못한다. Cross-encoder는 느리지만 쿼리-문서 쌍을 함께 평가하여 더 정확한 관련도를 계산한다.

일반적인 패턴은 Top-100을 Bi-encoder로 빠르게 검색한 후, Cross-encoder로 Top-5로 좁히는 것이다. 이 방식은 속도와 정확도의 균형을 맞춘다.

## Hallucination 최소화

RAG를 사용해도 Hallucination이 완전히 사라지지 않는다. 관련 문서를 찾지 못했거나, 노이즈 문서가 컨텍스트에 포함되거나, 프롬프트 설계가 잘못된 경우 발생한다.

Hallucination을 줄이는 가장 효과적인 방법은 프롬프트 설계다. "제공된 문서에 없는 내용은 '확인할 수 없다'고 명시하라"는 지침을 System Prompt에 포함한다. 또한 답변에 출처 문서 번호를 표시하도록 요구하면 모델이 근거 없는 답변을 생성하기 어려워진다.

신뢰도 기반 필터링도 효과적이다. 검색 결과의 최대 유사도 점수가 0.6 미만이면 답변을 아예 거부하고 "신뢰할 수 있는 정보를 찾을 수 없습니다"라고 안내하는 것이 잘못된 답변을 제공하는 것보다 낫다.
