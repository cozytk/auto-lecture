### 1-3. 데이터 익명화 방침

**방법**:
- Prod DB에서 최근 30일 로그를 주 1회 추출한다.

**담당자**: 데이터 엔지니어링 팀 (자동화 파이프라인, 매주 월요일 오전 3시 실행)

## 2. Scaling 전략

### 2-1. 병목 분석

Agent 서비스의 주요 처리 시간은 LLM API 응답 대기(평균 2~4초)다.
서버 CPU는 경량 Proxy 역할이므로 병목이 아니다.
→ **수평 Scaling이 최적 전략**이다.

### 2-2. Stateless 설계

서버가 상태를 갖지 않도록 다음 항목을 외부화한다.

| 상태 항목 | 저장 위치 | TTL |
|---|---|---|
| 대화 히스토리 | Redis | 30분 |
| 세션 메타데이터 (user_id, 시작 시간) | Redis | 30분 |
| Semantic Cache | Redis | 60분 |
| 평가 로그 (Trace) | PostgreSQL | 90일 |

### 2-3. Auto-scaling 트리거

| 트리거 | 증가 기준 | 감소 기준 |
|---|---|---|
| CPU 사용률 | > 70% (2분 지속) | < 20% (10분 지속) |
| 요청 큐 길이 | > 100건 | < 10건 |
| 최소 인스턴스 | 2개 (가용성 보장) | — |
| 최대 인스턴스 | 20개 | — |

**주의**: LLM API 호출이 외부 서비스이므로 인스턴스 수보다 **동시 연결 수 제한**이 더 중요하다.
인스턴스당 최대 50개 동시 LLM 요청으로 제한한다.

## 3. 모델 계층화 계획

### 3-1. Task 유형 정의

| Task 유형 | 설명 | 판단 기준 | 비율 |
|---|---|---|---|
| 분류/라우팅 | 질문 카테고리 분류, 메타데이터 추출 | 모든 요청의 1단계 | 100% |
| 단순 답변 | 단답형, FAQ, 규정 확인 | 컨텍스트 길이 < 2,000 토큰 | 65% |
| 일반 RAG 응답 | 문서 기반 서술형 답변 | 컨텍스트 길이 2,000~8,000 토큰 | 30% |
| 복잡 추론 | 다단계 분석, 정책 해석 | 명시적 "분석", "비교" 요청 | 5% |

### 3-2. 모델 배정

| Task 유형 | 모델 | 선택 근거 |
|---|---|---|
| 분류/라우팅 | gpt-4o-mini | 정확도 충분, 비용 최소화 |
| 단순 답변 | gpt-4o-mini | 단답형은 gpt-4o-mini로 충분 |
| 일반 RAG 응답 | gpt-4o | 문서 근거 응답에 정확도 중요 |
| 복잡 추론 | gpt-4o | o3 대비 비용 효율 우선 (필요 시 전환) |

### 3-3. 분류 로직

```python
def classify_task(query: str, context_length: int) -> TaskType:
    # 복잡 추론 키워드
    complex_keywords = ["분석해줘", "비교해줘", "장단점", "정책 해석"]
    if any(kw in query for kw in complex_keywords):
        return TaskType.COMPLEX

    # 컨텍스트 길이 기반
    if context_length > 2000:
        return TaskType.RAG_RESPONSE

    # 기본: 단순 답변
    return TaskType.SIMPLE
```

## 4. 월간 비용 추정

### 4-1. 계층화 전 (gpt-4o 100%)

```
일 50,000 요청 × 30일 = 1,500,000 요청/월

입력 토큰: 1,500,000 × 1,500 / 1,000 × $0.005 = $11,250
출력 토큰: 1,500,000 × 300 / 1,000 × $0.015 = $6,750
합계: $18,000/월  ← 목표 $5,000 대비 3.6배 초과
```

### 4-2. 계층화 후

| Task | 비율 | 일 요청 | 입력 단가 | 출력 단가 | 월 비용 |
|---|---|---|---|---|---|
| 분류/라우팅 (mini) | 100% | 50,000 | $0.00015 | $0.0006 | $225 |
| 단순 답변 (mini) | 65% | 32,500 | $0.00015 | $0.0006 | $1,102 |
| 일반 RAG (4o) | 30% | 15,000 | $0.005 | $0.015 | $1,800 |
| 복잡 추론 (4o) | 5% | 2,500 | $0.005 | $0.015 | $300 |
| **합계** | | | | | **$3,427** |

### 4-3. Semantic Cache 적용 후

반복 쿼리 비율 약 30% 가정 (FAQ 특성상):
```
절감: $3,427 × 30% = $1,028
Cache 적용 후: $3,427 - $1,028 = $2,399/월
```

### 4-4. 최종

| 단계 | 월 비용 | 절감률 |
|---|---|---|
| 계층화 전 | $18,000 | — |
| 계층화 후 | $3,427 | 81% |
| Cache 적용 후 | $2,399 | 87% |
| **목표 $5,000 달성** | **예** | — |

## 5. 카나리 배포 계획

### 5-1. 단계별 계획

| 단계 | 트래픽 비율 | 관찰 시간 | 승격 기준 | 롤백 기준 |
|---|---|---|---|---|
| 1단계 | 5% | 1시간 | 오류율 < 1%, P95 < 5s | 오류율 > 2% 또는 P95 > 8s |
| 2단계 | 25% | 2시간 | 오류율 < 1%, P95 < 5s | 오류율 > 1.5% 또는 P95 > 7s |
| 3단계 | 100% | — | — | 동일 기준 (자동 롤백) |

### 5-2. 자동 롤백 설정

```yaml
# 카나리 자동 롤백 기준
canary:
  analysis:
    interval: 1m
    threshold: 5   # 5번 연속 기준 초과 시 롤백
  metrics:
    - name: error_rate
      threshold: 0.02   # 2%
    - name: latency_p95
      threshold: 8000   # 8,000ms
```

### 5-3. 배포 일정

```
Day 0: CI 통과 + Golden Test Set 통과 확인
Day 1 AM: Dev 배포 → 내부 테스트
Day 1 PM: Staging 배포 → 통합 테스트
Day 2 AM: 카나리 5% 배포
Day 2 PM: 카나리 25% 승격 (기준 통과 시)
```

## 6. 비상 대응 계획

### 6-1. P0 장애 롤백 절차

1. PagerDuty 알림 수신 (On-call 담당자)
2. 현재 오류율·응답 시간 대시보드 확인 (2분)
3. 최근 배포 이력 확인 — 배포 직후 장애면 롤백 우선 (3분)
4. 롤백 명령 실행:
   ```bash
   just rollback --version v1.2.3
   ```
5. 롤백 후 오류율 정상화 확인 (5분)
6. Slack `#ops-p0` 채널에 상황 공유
7. 원인 분석 시작 (별도 Runbook 참조)

**목표 롤백 시간**: 장애 감지 후 15분 이내

### 6-2. 에스컬레이션 체계

| 수준 | 조건 | 연락 대상 | 방법 |
|---|---|---|---|
| L1 | 오류율 > 1% | On-call 개발자 | PagerDuty |
| L2 | 15분 내 미해결 | 팀 리드 | 전화 |
| L3 | 30분 내 미해결, 고객 영향 | CTO | 전화 |

### 6-3. 사후 검토 (Post-mortem) 프로세스

- P0 장애 24시간 내 타임라인 작성
- 72시간 내 근본 원인 분석 (5-Why 적용)
- 1주 내 재발 방지 조치 완료 및 확인
- 전사 공유 (동일 실수 반복 방지)
