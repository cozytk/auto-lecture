## Session 1: CLI 방식 복습

### 왜 이것이 중요한가

AI 도구(Claude, GPT 등)가 외부 시스템과 상호작용하려면 "도구"가 필요하다.
가장 전통적인 방법은 CLI 명령어를 실행하는 것이다.
`subprocess.run()`, `exec()`, 쉘 스크립트 호출 — 개발자라면 이미 익숙한 패턴이다.

### 핵심 원리

CLI 방식의 도구 연동은 **텍스트 입출력 파이프라인**이다.

```
AI 모델 → "git status 실행해줘" → 쉘 명령어 실행 → stdout 캡처 → AI에게 반환
```

**장점**:
- 기존 CLI 도구를 그대로 재활용 — 새로 만들 것이 없다
- 개발자에게 친숙한 멘탈 모델
- 디버깅이 쉽다 (터미널에서 직접 실행 가능)

**단점**:
- 출력이 비정형 텍스트 → AI가 파싱에 실패할 수 있다
- 도구 목록이 하드코딩 → 새 도구 추가 시 코드 수정 필요
- 보안: 쉘 인젝션 위험
- 상태 관리가 어렵다 (각 명령이 독립적)

### 주의사항과 흔한 오해

> **오해 1**: "CLI로 못하는 건 없으니까 MCP가 필요 없다"
> → CLI로 **할 수 있는 것**과 **잘 할 수 있는 것**은 다르다. AI가 CLI 출력을 해석하는 과정에서 오류가 빈번하다.

> **오해 2**: "MCP가 CLI를 대체한다"
> → MCP는 CLI를 대체하는 것이 아니라, AI-도구 간 통신에 **구조화된 프로토콜**을 추가하는 것이다. 내부적으로 CLI를 호출할 수도 있다.

### Q&A

**Q: AI가 쉘 명령어를 직접 실행하면 보안 문제가 없나요?**
A: 맞습니다. 대표적으로 쉘 인젝션 공격이 가능합니다. AI가 `rm -rf /`를 실행하거나, 사용자 입력을 그대로 쉘에 전달하면 위험합니다. 그래서 대부분의 AI 도구는 샌드박스 안에서 실행하거나, 허용된 명령어 목록(allowlist)을 사용합니다. Claude Code의 경우 사용자에게 명령어 실행 전 승인을 요청하는 방식을 씁니다.

**Q: subprocess로 호출하면 되는데 왜 복잡하게 만드나요? (← CLI 익숙한 개발자 배경)**
A: 단일 도구를 호출할 때는 subprocess로 충분합니다. 하지만 10개, 20개 도구를 AI에게 제공해야 한다면? 각 도구의 입력 스키마, 출력 형식, 에러 처리를 일일이 코딩해야 합니다. MCP는 이 "도구를 AI에게 알려주는 방법"을 표준화한 것입니다.

## Session 2: MCP 등장

### 왜 이것이 중요한가

2024년 11월, Anthropic이 MCP(Model Context Protocol)를 공개했다.
AI 모델이 외부 도구/데이터에 접근하는 방법을 **표준화**하려는 시도다.

문제의 핵심: AI 도구 연동이 **N×M 문제**였다.
- N개의 AI 앱 (Claude, GPT, Cursor, Windsurf...)
- M개의 도구/서비스 (GitHub, Slack, DB, 파일시스템...)
- 각 조합마다 별도의 연동 코드 → N×M개의 어댑터

MCP는 이를 **N+M 문제**로 바꾼다.
- AI 앱은 MCP 클라이언트만 구현
- 도구는 MCP 서버만 구현
- 중간에 MCP 프로토콜이 표준 연결

**비유**: USB가 등장하기 전에는 프린터마다 전용 케이블이 필요했다. USB가 이 문제를 해결한 것처럼, MCP는 AI-도구 연결의 "USB"를 목표로 한다.

### 핵심 원리

MCP의 아키텍처:

```
AI 호스트 앱 (Claude Desktop, IDE)
  └── MCP 클라이언트
        ├── MCP 서버 A (GitHub)
        ├── MCP 서버 B (Slack)
        └── MCP 서버 C (PostgreSQL)
```

**MCP 서버가 제공하는 3가지**:
1. **Tools** — AI가 호출할 수 있는 함수 (예: `search_issues`, `send_message`)
2. **Resources** — AI가 읽을 수 있는 데이터 (예: 파일 내용, DB 스키마)
3. **Prompts** — 미리 정의된 프롬프트 템플릿

**통신 방식**: JSON-RPC 2.0 over stdio 또는 SSE(Server-Sent Events)
- 로컬 서버: stdio (프로세스 간 통신)
- 원격 서버: SSE over HTTP

### 다른 접근법과의 비교

| 구분 | CLI 방식 | Function Calling | MCP |
|------|---------|-----------------|-----|
| 도구 정의 | 코드에 하드코딩 | API 스키마 (JSON) | MCP 서버가 동적 제공 |
| 출력 형식 | 비정형 텍스트 | 정형 JSON | 정형 JSON |
| 도구 검색 | 불가 | 불가 | 가능 (동적 디스커버리) |
| 상태 관리 | 없음 | 없음 | 세션 기반 |
| 보안 | 쉘 인젝션 위험 | API 레벨 | 권한 스코프 지정 |
| 재사용성 | 낮음 (앱별 구현) | 중간 | 높음 (표준 프로토콜) |

### 주의사항과 흔한 오해

> **오해 1**: "MCP 서버를 만들려면 복잡한 인프라가 필요하다"
> → MCP 서버는 단순한 프로세스다. Python이나 TypeScript로 50줄이면 기본 서버를 만들 수 있다.

> **오해 2**: "MCP는 Anthropic 전용이다"
> → MCP는 오픈 프로토콜이다. OpenAI, Google 등도 지원을 시작했다. 특정 벤더에 종속되지 않는다.

> **오해 3**: "MCP가 Function Calling을 대체한다"
> → Function Calling은 AI 모델이 함수를 호출하는 메커니즘이고, MCP는 그 함수(도구)를 AI에게 알려주는 프로토콜이다. 레이어가 다르다. MCP를 통해 발견된 도구가 Function Calling으로 실행된다.

### Q&A

**Q: MCP 서버는 항상 로컬에서 실행해야 하나요?**
A: 아닙니다. 로컬(stdio)과 원격(SSE/HTTP) 두 방식 모두 지원합니다. 파일시스템 같은 로컬 리소스는 stdio로, SaaS API 같은 원격 서비스는 SSE로 연결합니다. 2025년부터는 OAuth 기반 원격 MCP 서버가 본격화되고 있습니다.

**Q: 기존에 만든 CLI 도구를 MCP 서버로 감싸는 게 가능한가요? (← CLI 익숙한 개발자 배경)**
A: 네, 가장 흔한 패턴입니다. MCP 서버 내부에서 기존 CLI 도구를 subprocess로 호출하고, 출력을 정형화된 JSON으로 변환하여 반환합니다. 기존 자산을 버리는 게 아니라 "AI 친화적 래퍼"를 씌우는 것입니다.

**Q: MCP가 아직 초기 단계인데 프로덕션에 써도 되나요?**
A: 프로토콜 자체는 안정적이지만, 에코시스템은 빠르게 변하고 있습니다. 내부 도구나 개발 워크플로에는 적극 도입해도 좋지만, 고객 대면 서비스에 적용할 때는 버전 호환성과 보안 검토를 신중히 하십시오.

## Session 3: 선택 기준

### 언제 CLI를 쓰는가
- 단순한 1회성 명령 실행
- 이미 잘 동작하는 CLI 도구가 있고, AI 연동이 부차적일 때
- 빠른 프로토타이핑

### 언제 MCP를 쓰는가
- AI 앱에 여러 도구를 체계적으로 제공해야 할 때
- 도구의 입출력을 정형화하고 싶을 때
- 여러 AI 앱에서 동일한 도구를 재사용하고 싶을 때
- 도구 목록이 동적으로 변할 때

### 핵심 결론

> CLI와 MCP는 경쟁이 아니라 **레이어가 다른 보완 관계**다.
> CLI는 "명령을 실행하는 방법"이고, MCP는 "AI에게 도구를 알려주는 방법"이다.
> MCP 서버 내부에서 CLI를 호출하는 것이 가장 흔한 실무 패턴이다.
