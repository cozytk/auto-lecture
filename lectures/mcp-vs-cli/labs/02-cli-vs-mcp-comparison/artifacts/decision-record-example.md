# 시나리오별 의사결정 기록서 — 모범 답안 예시

아래는 5개 시나리오 중 2개의 모범 답안 예시입니다.
나머지 3개는 학생이 직접 작성해야 합니다.

---

## 시나리오 1: 개인 CLI 자동화

### 상황 분석
- 도구 개수: 3~5개 (ls, mv, cp 등 기본 명령어)
- AI 앱 개수: 1개 (개인 터미널)
- 출력 정형화 필요성: 낮음 (파일 목록 정도는 AI가 잘 파싱)
- 상태 관리 필요성: 없음
- 보안 요구 수준: 낮음 (개인 환경)
- 재사용성 요구: 낮음 (나만 사용)

### 판단
- 선택: CLI
- 핵심 근거: 도구가 소수이고 1개 앱에서만 사용하므로, MCP 서버 구축의 오버헤드가 이점보다 크다. CLI의 토큰 효율성(TES 202)도 이 규모에서 유리하다.

### 트레이드오프
- 이 선택의 장점: 즉시 구현 가능, subprocess 한 줄이면 끝, 디버깅 쉬움
- 이 선택의 단점: 다른 AI 앱으로 전환 시 재구현 필요
- 반대 선택을 했다면 생길 문제: MCP SDK 학습 + 서버 설정 시간이 실제 자동화 시간보다 길어짐

---

## 시나리오 2: 팀 내부 도구 표준화

### 상황 분석
- 도구 개수: 10개 (GitHub, Slack, 내부 API 등)
- AI 앱 개수: 3개 (Claude Desktop, Cursor, VS Code Copilot)
- 출력 정형화 필요성: 높음 (10개 도구의 다양한 출력을 AI가 정확히 파싱해야 함)
- 상태 관리 필요성: 있음 (인증 토큰 등 세션 관리)
- 보안 요구 수준: 중간 (팀 내부지만 API 키 관리 필요)
- 재사용성 요구: 높음 (3개 앱 x 10개 도구 = 30개 조합)

### 판단
- 선택: MCP
- 핵심 근거: 3개 AI 앱 x 10개 도구 = N x M 문제의 전형적 사례. CLI로 구현하면 앱마다 별도 연동 코드(30개)가 필요하지만, MCP로는 서버 10개 + 클라이언트 설정 3개 = 13개로 해결된다.

### 트레이드오프
- 이 선택의 장점: 한 번 만들면 모든 AI 앱에서 재사용, 도구 추가/수정이 서버에서만 발생
- 이 선택의 단점: 초기 구축 비용 높음, 팀원 MCP 학습 필요, 토큰 효율성은 CLI보다 낮음
- 반대 선택을 했다면 생길 문제: 3개 앱 각각에 10개 도구를 연동하는 코드 유지보수 지옥
