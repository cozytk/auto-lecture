## I DO: 시연 관찰 (약 8분)

강사가 "리서치 -> 검증 -> 보고서" 그래프를 예시로 아래 순서로 경계를 나누는 과정을 시연한다.

1. 각 Node가 읽는 필드와 쓰는 필드를 매트릭스로 적는다.
2. 함께 자주 움직이는 Node를 한 그룹으로 묶는다.
3. 외부에 공개해야 할 State와 내부 전용 State를 구분한다.
4. 체크포인트가 필요한 지점을 선택한다.

### 관찰 포인트

- 분리 기준은 파일 수가 아니라 State 의존성이다.
- 내부 구현 상세를 부모 그래프에 노출하면 리팩토링 효과가 떨어진다.
- checkpointer는 "긴 작업"과 "재개 가치가 큰 작업"에 먼저 붙인다.

## 검증 방법

- Node별 읽기/쓰기 필드가 최소 3개 이상 기록되었는가
- 부모 공개 키와 내부 전용 키가 분리되어 있는가
- 인터페이스 계약이 "입력 / 출력 / 계약 메모" 3항목을 모두 포함하는가
- 체크포인트 전략이 재개, 메모리, thread 분리 관점으로 설명되는가

## 트러블슈팅

| 증상 | 원인 | 해결 방법 |
|------|------|----------|
| 어디서 끊어야 할지 모름 | 읽기/쓰기 필드를 안 적음 | 먼저 의존성 매트릭스를 작성한다 |
| 부모 State가 너무 커짐 | 내부 전용 필드까지 공개함 | 재시도 횟수, 임시 캐시 등은 내부로 숨긴다 |
| 서브그래프를 나눠도 테스트가 어려움 | 계약이 모호함 | 입력 키와 출력 키를 표로 고정한다 |
