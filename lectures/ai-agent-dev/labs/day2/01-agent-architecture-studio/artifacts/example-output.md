# 예시 산출물: Agent Architecture Studio

## Agent 이름
- 고객 문의 해결 Agent

## Goal
- 고객 문의를 접수한 뒤 FAQ 검색, 추가 정보 확인, 에스컬레이션 판단까지 마쳐 사용자가 다음 행동을 알 수 있게 한다.

## Memory

| 구분 | 저장 내용 | 왜 필요한가 |
|------|----------|-------------|
| 대화 히스토리 | 최근 고객 문의와 추가 질문 | 같은 질문 반복을 막기 위해 |
| 작업 상태 | 현재 단계(classify / search / escalate) | 루프와 종료 조건을 제어하기 위해 |
| Tool 결과 | FAQ 검색 결과, 담당자 라우팅 결과 | 같은 Tool 재호출을 줄이고 근거를 남기기 위해 |

## Tool

| Tool | 읽기/쓰기 | 위험 수준 | 호출 조건 |
|------|----------|----------|----------|
| `search_faq` | 읽기 | low | 질문이 FAQ로 해결 가능한지 확인할 때 |
| `lookup_ticket_policy` | 읽기 | low | 환불/장애처럼 정책 확인이 필요할 때 |
| `create_escalation_ticket` | 쓰기 | high | FAQ와 정책으로 해결되지 않았고 사람 개입이 필요할 때 |

## Control Logic

- 시작 조건: 고객 메시지 수신
- 반복 조건: 추가 확인이 필요하고 최대 검색 횟수를 넘지 않았을 때
- 종료 조건: 답변 생성 또는 에스컬레이션 티켓 생성 완료
- 실패 시 fallback: 최대 검색 횟수 초과 시 사람 상담으로 라우팅

## Sub-task 분해표

| ID | 설명 | depends_on | 전략 | 완료 기준 |
|----|------|------------|------|----------|
| T1 | 문의 유형 분류 | - | sequential | 기술/환불/일반으로 분류됨 |
| T2 | FAQ 검색 | T1 | sequential | 관련 답변 또는 미해결 상태 확보 |
| T3 | 정책 확인 | T1 | parallel | 환불/장애 정책을 읽음 |
| T4 | 해결 가능 여부 판단 | T2, T3 | sequential | 답변 생성 또는 에스컬레이션 결정 |
| T5 | 티켓 생성 | T4 | sequential | 담당자 티켓이 생성됨 |

## 병렬 가능 지점

- T2와 T3는 T1 이후 서로 직접 의존하지 않으므로 병렬 후보로 둘 수 있다.

## Re-planning이 필요한 상황

- FAQ 검색 결과가 비어 있고 정책 문서에도 직접 답이 없을 때
- 티켓 생성 Tool이 실패해 다른 라우팅 채널로 바꿔야 할 때

## 구조 판단

- 선택: Multi-step

### 선택 이유
1. FAQ 검색 결과에 따라 다음 Tool 호출이 달라진다.
2. 실패 시 에스컬레이션이라는 대체 경로가 필요하다.
3. 검색 횟수와 현재 단계 같은 상태를 누적 관리해야 한다.

### 배제한 대안
- 배제한 구조: Single-step
- 배제 이유: 한 번의 호출로는 FAQ 검색, 정책 확인, 티켓 생성 여부를 모두 안전하게 처리하기 어렵다.
