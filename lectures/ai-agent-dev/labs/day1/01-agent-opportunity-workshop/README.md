# Day 1 실습 01: Agent Opportunity Workshop

## 실습 목적

Session 1의 핵심 내용을 실제 업무 문제에 적용하여, Agent 후보를 도출하고
RAG / Agent / 스크립트 중 어떤 구조가 적합한지 논리적으로 판단한다.

- **연관 세션**: Session 1 - Agent 문제 정의와 과제 도출
- **실습 형식**: README 중심
- **난이도**: 기초 ~ 중급
- **예상 소요 시간**: 75분 (I DO 15분 / WE DO 25분 / YOU DO 35분)

## 산출물

- Agent 후보 발굴표 3개
- 최종 선정 후보 2개
- 후보별 아키텍처 판단표 2개
- 업무 유형별 패턴 매핑 요약 1개

## 사전 준비

- 본인 업무 또는 만들고 싶은 Agent 아이디어 3개 이상 떠올려 둔다.
- Session 1의 Pain-Task-Skill-Tool 프레임워크 내용을 다시 확인한다.

## I DO: 시연 관찰 (약 15분)

강사가 "주간 보고서 자동화"와 "사내 FAQ 챗봇"을 예시로 아래 과정을 시연한다.

1. Pain를 수치화한다.
2. Task를 동사 단위로 분해한다.
3. Skill과 Tool을 구분한다.
4. Agent / RAG / 스크립트 적합성을 3가지 조건으로 판정한다.
5. 최종 후보 2개만 남긴다.

### 관찰 포인트

- Pain가 구체적일수록 Tool 선택이 쉬워진다.
- "Tool을 쓸 수 있다"와 "Agent가 필요하다"는 다르다.
- RAG는 검색 중심, Agent는 판단 + 행동 중심, 스크립트는 고정 흐름 중심이다.

## 검증 방법

- 후보가 3개 이상 작성되었는가
- 최종 후보가 2개로 압축되었는가
- 각 후보에 대해 Pain, Task, Skill, Tool이 모두 분리되어 있는가
- 구조 선택 이유가 "좋아 보여서"가 아니라 판단 축으로 설명되는가

## 트러블슈팅

| 증상 | 원인 | 해결 방법 |
|------|------|----------|
| 후보가 너무 추상적임 | Pain가 수치화되지 않음 | 빈도, 시간, 실패 비용을 먼저 적는다 |
| 모든 후보가 Agent처럼 보임 | Tool 중심으로 생각함 | "판단이 필요한가?"를 다시 체크한다 |
| RAG와 Agent를 구분 못함 | 검색과 행동을 혼동함 | 문서 검색만 하면 RAG, 외부 쓰기/행동이면 Agent를 우선 검토 |
