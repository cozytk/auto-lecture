# 에이전트 생성 테스트 품질 체크리스트

에이전트(Claude, Cursor 등)에게 "이 코드에 테스트를 작성해줘"라고 요청한 후,
생성된 테스트의 품질을 아래 기준으로 평가한다.

---

## 1. 동어반복(Tautological) 테스트 여부

> **문제**: 테스트가 구현 코드를 그대로 복사하거나, 값이 자기 자신과 같다고 단언하는 경우

- [ ] 테스트가 함수의 반환값을 단순히 다시 호출해서 비교하지 않는가?
- [ ] `assert result == function(input)` 형태의 무의미한 단언이 없는가?
- [ ] 예상값(expected value)이 하드코딩된 수학적 정답인가? (구현 코드 재사용 아님)

**예시: 나쁜 테스트 (동어반복)**
```python
def test_average():
    values = [1, 2, 3]
    result = compute_average(values)
    assert result == compute_average(values)  # 항상 통과 — 의미 없음
```

**예시: 좋은 테스트**
```python
def test_average():
    assert compute_average([1, 2, 3]) == 2.0  # 수학적으로 검증된 예상값
```

## 2. 엣지 케이스 커버리지

- [ ] 빈 입력(빈 리스트, 빈 파일)을 테스트하는가?
- [ ] 결측값/None/빈 문자열을 처리하는 테스트가 있는가?
- [ ] 경계값(boundary value)을 테스트하는가? (예: 등급 임계값 정확히 90점)
- [ ] 비정상 입력(음수 값, 문자열 등)에 대한 테스트가 있는가?
- [ ] 단일 요소 입력을 테스트하는가?

---

## 3. 테스트 독립성

- [ ] 각 테스트가 다른 테스트의 실행 순서에 의존하지 않는가?
- [ ] 전역 상태를 변경하는 경우 테스트 후 정리(teardown)하는가?
- [ ] 테스트가 실제 파일시스템/네트워크에 의존하는 경우 임시 파일/Mock을 사용하는가?
- [ ] 한 테스트 실패가 다른 테스트에 영향을 주지 않는가?

---

## 4. Assert 메시지 명확성

- [ ] assert 실패 시 왜 실패했는지 메시지가 표시되는가?
- [ ] 예상값과 실제값이 오류 메시지에 포함되는가?

**예시: 나쁜 assert**
```python
assert result == 5.0
```

**예시: 좋은 assert**
```python
assert result == 5.0, f"평균이 5.0이어야 하지만 {result}임"
```

---

## 5. 실제 요구사항 검증

- [ ] 테스트가 비즈니스 규칙을 검증하는가? (예: "90점 이상은 A등급")
- [ ] 테스트 이름만 봐도 어떤 동작을 검증하는지 알 수 있는가?
- [ ] 기술적 구현이 아닌 외부에서 관찰 가능한 동작을 테스트하는가?
- [ ] 단순히 코드가 "예외 없이 실행된다"만 검증하는 테스트가 없는가?

---

## 6. 테스트 구조 (AAA 패턴)

- [ ] Arrange(준비) → Act(실행) → Assert(검증) 구조를 따르는가?
- [ ] 하나의 테스트가 하나의 동작만 검증하는가?
- [ ] 테스트 함수명이 `test_[기능]_[조건]_[예상결과]` 형태로 설명적인가?

## 에이전트 테스트의 흔한 패턴

에이전트가 테스트를 생성할 때 자주 나타나는 문제들:

1. **구현 복사**: 구현 코드의 로직을 테스트에 그대로 옮겨 쓰는 경우
2. **단순 실행 확인**: `assert result is not None` 수준의 테스트만 생성
3. **해피 패스만 테스트**: 정상 입력만 다루고 예외/엣지 케이스 누락
4. **의미없는 이름**: `test_function1`, `test_method_works` 같은 이름
5. **테스트끼리 의존**: 이전 테스트가 만든 파일/상태를 다음 테스트가 사용

이 체크리스트를 활용해 에이전트가 생성한 테스트를 직접 평가해보고,
부족한 부분을 직접 추가해보자.
