# 평가 루브릭

**Lab 04: 바이브 코딩 — 처음부터 앱 만들기**
**총점: 100점**

---

## 1. 동작 여부 (30점)

핵심 기능이 실제로 동작하는가를 평가한다.

| 점수 | 기준 |
|------|------|
| 28~30점 | 시나리오의 필수 기능 3개 모두 동작. 에러 없이 정상 실행. |
| 20~27점 | 필수 기능 2개 동작, 나머지 1개는 부분 동작 또는 에러 있음. |
| 10~19점 | 필수 기능 1개만 동작하거나 전체가 부분적으로 동작. |
| 1~9점 | 코드는 존재하지만 실행 시 오류. 일부 흐름만 동작. |
| 0점 | 실행 불가 또는 아무것도 동작하지 않음. |

**평가 방법**: 발표 시간에 직접 터미널에서 실행. 사전 준비된 스크린샷은 인정하지 않음.

---

## 2. 코드 품질 (20점)

에이전트가 만든 코드가 AGENTS.md의 규칙을 따르고, 테스트가 존재하는가를 평가한다.

| 점수 | 기준 |
|------|------|
| 18~20점 | AGENTS.md 존재, 규칙 준수, 테스트 5개 이상 통과. |
| 13~17점 | AGENTS.md 존재, 테스트 2~4개 통과. 규칙 일부 미준수. |
| 7~12점 | AGENTS.md 존재하지만 내용 부실, 테스트 1개 통과. |
| 1~6점 | AGENTS.md 없거나 테스트 전혀 없음. 코드만 존재. |
| 0점 | 코드 구조 없음. |

**세부 항목**:
- AGENTS.md 존재 및 내용 충실도 (5점)
- 테스트 존재 및 통과 여부 (10점)
- 코드 가독성 및 구조 (5점)

---

## 3. 프롬프트 전략 (20점)

체계적으로 프롬프트를 작성하고 진행했는가를 평가한다.

| 점수 | 기준 |
|------|------|
| 18~20점 | 단계별 프롬프트 전략이 명확. 효과적인 프롬프트를 발표 시 공유하고 이유를 설명. |
| 13~17점 | 대체로 체계적으로 진행. 프롬프트 전략을 발표 시 언급. |
| 7~12점 | 다소 무작위적이었지만 결과는 만들어냄. 프롬프트 전략 설명이 빈약. |
| 1~6점 | 프롬프트 전략 없이 진행. 에이전트에 모든 것을 한 번에 요청. |
| 0점 | 에이전트를 거의 사용하지 않거나 코드를 직접 작성. |

**평가 방법**: 발표 시 핵심 프롬프트 공유 단계에서 평가.

---

## 4. 발표/시연 (15점)

앱을 소개하고 과정을 설명하는 능력을 평가한다.

| 점수 | 기준 |
|------|------|
| 14~15점 | 발표 포맷 5단계 모두 충실. 시연이 막힘 없이 진행. 배운 점 인사이트 있음. |
| 10~13점 | 발표 포맷 대부분 준수. 시연 성공. 배운 점 언급. |
| 5~9점 | 발표 포맷 일부 누락. 시연 부분 실패. |
| 1~4점 | 발표 미흡. 시연 실패 또는 미진행. |
| 0점 | 발표 없음. |

**발표 포맷 체크리스트**:
- [ ] 시나리오 소개
- [ ] AGENTS.md 설명
- [ ] 동작 시연
- [ ] 핵심 프롬프트 공유
- [ ] 배운 점

---

## 5. 도전 정신 (15점)

주어진 것 이상을 시도했는가를 평가한다.

| 점수 | 기준 |
|------|------|
| 14~15점 | 선택 기능 2개 이상 구현 또는 창의적 추가 기능. 기술적 도전이 명확. |
| 10~13점 | 선택 기능 1개 구현 또는 필수 기능의 완성도가 특별히 높음. |
| 5~9점 | 필수 기능을 충실히 구현했지만 추가 시도 없음. |
| 1~4점 | 필수 기능 일부만 구현. 추가 시도 없음. |
| 0점 | 최소 요건에도 미치지 못함. |

**가산점 사례**:
- 40분 안에 예상보다 훨씬 많은 기능 완성
- 특히 창의적인 아이디어(자유 주제)
- 발표 시 특별히 영리한 프롬프트 기법 공유
- 팀원 모두가 적극적으로 기여

## 강사용 메모

- 동작 여부는 반드시 **발표 자리에서 직접 실행**으로 확인
- 코드 직접 작성 여부는 git 히스토리 또는 프롬프트 기록으로 확인 가능
- 발표 시간이 부족하면 동작 여부(30점) 우선 확인 후 코드 품질은 별도 제출 허용
