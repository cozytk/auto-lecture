## 1. LLM 챗봇과 코드 에이전트의 차이

### 왜 이것이 중요한가

여러분은 이미 ChatGPT, Claude 같은 LLM 채팅을 사용해 본 적이 있을 것이다.
코드 관련 질문을 하면 꽤 괜찮은 답변이 돌아온다.

그런데 한 가지 불편한 점이 있다.
코드를 **복사-붙여넣기**해야 한다.

"이 함수에 버그가 있는데 고쳐줘" → 코드를 복사해서 채팅창에 붙인다.
답변을 받으면 → 다시 복사해서 에디터에 붙인다.
실행해보니 안 된다 → 에러 메시지를 다시 복사해서 붙인다.

이 **복사-붙여넣기 루프**가 코드 에이전트가 해결하는 핵심 문제다.
코드 에이전트는 **직접** 파일을 읽고, 수정하고, 실행하고, 결과를 확인한다.

---

### 핵심 원리

**비유로 이해하기**: 식당을 떠올려 보십시오.

→ LLM 채팅 = **전화 상담 셰프**. 요리법을 알려주지만, 직접 요리하지 않는다. 재료가 있는지도 모른다.
→ 코드 에이전트 = **파견 셰프**. 부엌에 와서 냉장고를 열어보고, 재료를 확인하고, 직접 요리한다. 맛도 본다.

**실제 동작**:

| 구분 | LLM 채팅 (ChatGPT, Claude 웹) | 코드 에이전트 (CLI Agent) |
|------|-------------------------------|--------------------------|
| 상호작용 | 질문 → 답변 (1회) | 요청 → **자율적 다단계 실행** |
| 파일 접근 | 없음 (복사-붙여넣기) | 직접 읽기·쓰기·검색 |
| 명령 실행 | 없음 | 셸 명령 **직접 실행** |
| 검증 | 사용자가 직접 | 에이전트가 테스트 실행·결과 확인 |
| 컨텍스트 | 대화 창에 붙여넣은 것만 | **전체 코드베이스** 탐색 가능 |

**비유의 한계**: 다만 파견 셰프와 달리, 코드 에이전트는 **사람의 감각**이 없다.
맛을 보는 것이 아니라 **테스트를 실행**하여 결과를 판단한다.
테스트가 없으면 에이전트도 자신의 코드가 맞는지 확인할 수 없다.

---

### 실무에서의 의미

[Anthropic의 2026 Agentic Coding Trends Report](https://resources.anthropic.com/hubfs/2026%20Agentic%20Coding%20Trends%20Report.pdf)에 따르면, 에이전틱 코딩은 이미 산업 전반에 채택되었다.
TELUS는 13,000개 AI 솔루션을 만들며 코드 배포 속도를 30% 높였다.

실무에서 코드 에이전트는 다음과 같은 작업에 쓰인다:
- 버그 수정: 에러 로그를 주면 원인을 찾아 코드를 수정하고 테스트까지 실행
- 리팩토링: "이 모듈을 TypeScript로 마이그레이션해" 같은 대규모 변경
- 테스트 작성: 기존 코드를 읽고 테스트 코드를 자동 생성

---

### 다른 접근법과의 비교

| 구분 | LLM 채팅 | IDE 자동완성 (Copilot) | 코드 에이전트 (CLI) |
|------|----------|----------------------|-------------------|
| **작업 단위** | 질문 1개 | 코드 한 줄~함수 | **전체 작업 흐름** |
| **자율성** | 없음 | 없음 (사용자가 수락) | **높음** (자율 실행) |
| **파일 접근** | 없음 | 열린 파일만 | **전체 프로젝트** |
| **셸 실행** | 불가 | 제한적 | **네이티브** |
| **적합한 작업** | 개념 질문 | 코드 조각 완성 | 복잡한 멀티스텝 작업 |

---

### 주의사항과 흔한 오해

> **오해 1**: "코드 에이전트가 개발자를 대체한다"

→ **실제**: Addy Osmani의 [80% 문제](https://addyo.substack.com/p/the-80-problem-in-agentic-coding)에 따르면, 에이전트가 코드의 80%를 생성하더라도 **검증·아키텍처·의사결정**은 사람의 몫이다. 코드 리뷰 시간은 오히려 91% 증가했다. 에이전트는 도구이지, 대체재가 아니다.

> **오해 2**: "코드 에이전트는 마법이다. 아무 말이나 하면 알아서 해준다"

→ **실제**: 에이전트는 **제한된 도구를 조합**하여 작업한다. 파일을 읽고, 수정하고, 명령을 실행하는 것이 전부다. 도구를 이해하면 에이전트의 능력과 한계를 정확히 파악할 수 있다. 모호한 요청은 모호한 결과를 낳는다.

이 주제에 대해 커뮤니티에서 의견이 나뉜다.
- **긍정 진영**: "에이전틱 코딩은 이미 주류. 적응하지 않으면 뒤처진다" — [InfoWorld](https://www.infoworld.com/article/4143101/pity-the-developers-who-resist-agentic-coding.html)
- **회의 진영**: "코드 품질이 낮고, 기대 대비 실망스럽다" — [Max Woolf](https://minimaxir.com/2026/02/ai-agent-coding/)

→ 더 깊이 알고 싶다면 **"Agentic Coding 80% Problem"**을 검색해 보십시오.

### 코드 예제

LLM 채팅과 코드 에이전트의 차이를 코드로 표현하면:

**LLM 채팅 방식** (사람이 중간에 개입):
```
[사용자] "add_todo 함수에 validation 추가해줘"
[LLM]    "다음과 같이 수정하면 됩니다: ..."  (텍스트로 코드 제시)
[사용자]  코드 복사 → 에디터에 붙여넣기 → 저장 → 실행 → 에러 발생
[사용자] "이런 에러가 나는데?"  (에러 복사-붙여넣기)
[LLM]    "아, 그러면 이렇게 고치세요: ..."
```

**코드 에이전트 방식** (에이전트가 자율 실행):
```
[사용자] "add_todo 함수에 validation 추가해. test_app.py에 테스트도 추가하고 실행해서 통과 확인해"
[에이전트] Read: todo_app.py → 코드 분석
[에이전트] Read: test_app.py → 기존 테스트 확인
[에이전트] Edit: todo_app.py → validation 로직 추가
[에이전트] Edit: test_app.py → 테스트 케이스 추가
[에이전트] Bash: python -m pytest test_app.py → 실행
[에이전트] "4개 테스트 모두 통과했습니다. 변경 사항을 요약하면..."
```

이 차이가 바로 **에이전틱 루프**다. 다음 섹션에서 이 루프의 구조를 자세히 살펴본다.

## 2. 에이전틱 루프 (Agentic Loop)

### 왜 이것이 중요한가

앞서 코드 에이전트가 "자율적으로 다단계 실행"한다고 했다.
그런데 정확히 **어떤 단계**를 밟는 것인가?

에이전틱 루프는 코드 에이전트의 **심장**이다.
이 루프를 이해하면, 에이전트가 왜 특정 순서로 작업하는지 예측할 수 있다.
예측할 수 있으면, 더 나은 지시를 내릴 수 있다.

루프를 모르면 에이전트는 "블랙박스"로 남는다.
블랙박스는 신뢰하기 어렵고, 문제가 생겼을 때 디버깅할 수 없다.

→ 참고: [Claude Code 아키텍처 — 에이전틱 루프](https://code.claude.com/docs/ko/overview)

---

### 핵심 원리

**비유로 이해하기**: 숙련된 수리 기사를 떠올려 보십시오.

→ 고장 신고를 받으면 → **현장을 확인**하고(Observe)
→ 원인을 **분석**하고(Think)
→ 수리를 **실행**한다(Act)
→ 수리 후 **작동을 확인**하고, 안 되면 다시 원인을 분석한다

이것이 **Observe → Think → Act** 루프다.
코드 에이전트도 정확히 이 패턴을 따른다.

**실제 동작**:

```
사용자 요청
    ↓
[Observe] 코드 읽기 (Read, Grep, Glob)
    ↓
[Think]   계획 수립 (무엇을 어떻게 수정할지)
    ↓
[Act]     코드 수정 (Edit, Write)
    ↓
[Act]     명령 실행 (Bash — 테스트, 빌드)
    ↓
[Observe] 결과 확인
    ↓
  성공? → 완료
  실패? → [Think] 원인 분석 → [Act] 재수정 → 반복
```

이것이 **ReAct 패턴**이다.
"생각하고(Reason), 행동한다(Act)"의 줄임말이다.

**비유의 한계**: 수리 기사는 경험과 감각으로 판단한다.
에이전트는 **도구의 출력**으로만 판단한다.
테스트가 없으면 "수리가 잘 되었는지" 확인할 방법이 없다.

→ 이것이 "**검증 수단 제공**"이 중요한 이유다. 2교시에서 자세히 다룬다.

---

### 실무에서의 의미

실제 프로젝트에서 에이전트에게 "로그인 버그 수정해"라고 요청하면:

1. `Grep`으로 "login" 관련 파일을 검색
2. `Read`로 해당 파일들을 읽어 구조 파악
3. `Edit`으로 버그 수정
4. `Bash`로 테스트 실행
5. 테스트 실패 시 → 에러 분석 → 재수정 → 재실행

이 과정이 **사람의 개입 없이** 자동으로 진행된다.
한 사이클에서 해결되지 않으면 여러 사이클을 반복한다.

### 실제 사례: @explore 서브에이전트의 에이전틱 루프

Lab 00에서 `@explore`를 실행하면, 서브에이전트가 다음과 같은 에이전틱 루프를 수행한다:

```
[Observe] Glob "*" → 파일 5개 발견 (note.py, storage.py, ...)
[Think]   "package.json이 없네? TypeScript가 아니라 Python이구나"
[Observe] Read note.py → CLI 구조 파악
[Observe] Read storage.py → JSON 저장소 패턴 확인
[Observe] Read formatter.py, config.py, test_note.py
[Think]   "진입점을 확인해야겠다"
[Act]     Grep "if __name__" → note.py:106이 메인 진입점
[Act]     Grep "import argparse|unittest" → 기술 스택 종합
[Act]     탐색 리포트 생성
```

→ **50초 동안 14번의 도구 호출**로 코드베이스를 분석했다.
→ Glob(넓게 훑기) → Read(깊게 읽기) → Grep(패턴 확인)이 전형적인 탐색 패턴이다.

> 실제 세션 로그는 Lab 00의 [README.feedback.md](../labs/00-codebase-exploration/README.feedback.md)에서 확인할 수 있다.

### 다른 접근법과의 비교

| 구분 | 단일 추론 (LLM) | 에이전틱 루프 | 파이프라인 (CI/CD) |
|------|-----------------|-------------|-------------------|
| **반복** | 없음 (1회 응답) | **동적 반복** | 고정 단계 반복 |
| **판단** | 없음 | LLM이 상황 판단 | 규칙 기반 |
| **도구 사용** | 없음 | **다양한 도구 조합** | 사전 정의된 스크립트 |
| **유연성** | 낮음 | **높음** | 낮음 |

---

### 주의사항과 흔한 오해

> **오해 1**: "에이전트가 무한 루프에 빠지면 어떡하나?"

→ **실제**: 모든 코드 에이전트에는 **최대 반복 횟수**가 설정되어 있다. 일정 횟수 이상 실패하면 사용자에게 보고하고 멈춘다. 하지만 비용은 누적되므로, 명확한 요청과 검증 수단을 제공하는 것이 중요하다.

> **오해 2**: "에이전틱 루프 = 단순한 while 문 아닌가?"

→ **실제**: 핵심은 **LLM이 다음 행동을 동적으로 결정**한다는 점이다. `if-else`로 분기가 사전 정의된 것이 아니라, 중간 결과를 보고 "다음에 뭘 할지" 판단한다. 이것이 전통 자동화와의 근본적 차이다.

## 3. 도구 (Tools) — 에이전트의 손과 발

### 왜 이것이 중요한가

앞서 에이전틱 루프에서 에이전트가 "도구를 사용한다"고 했다.
그런데 에이전트가 사용하는 도구는 정확히 무엇인가?

이 도구를 이해하면 에이전트의 **능력 범위**를 정확히 파악할 수 있다.
"이건 에이전트가 할 수 있어"와 "이건 못 해"를 구분할 수 있게 된다.

모든 CLI 코드 에이전트는 **거의 동일한 도구 세트**를 제공한다.
OpenCode에서 배운 도구 개념은 Claude Code, Codex에서도 그대로 적용된다.

→ 참고: [Claude Code 기능 개요 — 내장 도구](https://code.claude.com/docs/ko/features-overview)

---

### 핵심 원리

**비유로 이해하기**: 목수의 도구 상자를 떠올려 보십시오.

→ 톱(Bash) = 무언가를 **실행**한다
→ 자(Read) = 무언가를 **읽고 측정**한다
→ 연필(Edit/Write) = 무언가를 **수정하거나 새로 만든다**
→ 돋보기(Grep/Glob) = 무언가를 **찾는다**

목수가 모든 작업을 망치 하나로 하지 않듯, 에이전트도 상황에 맞는 도구를 선택한다.

**실제 동작 — 도구 목록**:

| 기능 | OpenCode | Claude Code | 하는 일 |
|------|----------|-------------|---------|
| 파일 읽기 | `read` | `Read` | 코드 파일의 내용을 읽어 컨텍스트에 추가 |
| 파일 수정 | `edit`, `write` | `Edit`, `Write` | 코드를 직접 수정하거나 새 파일 생성 |
| 파일 검색 | `grep`, `glob` | `Grep`, `Glob` | 코드베이스에서 패턴·파일명 검색 |
| 셸 실행 | `bash` | `Bash` | 빌드, 테스트, git 등 터미널 명령 실행 |
| 웹 검색 | `websearch` | `WebSearch` | 문서·API 레퍼런스 등 외부 정보 검색 |
| 웹 가져오기 | `webfetch` | `WebFetch` | URL의 내용을 직접 가져오기 |
| 사용자 질문 | `question` | `AskUserQuestion` | 모호한 상황에서 사용자에게 확인 |
| 서브에이전트 | `task` | `Agent` | 독립적인 하위 작업을 별도 에이전트에 위임 |
| 스킬 로딩 | `skill` | `Skill` | 미리 정의된 절차/지식을 필요할 때 로드 |

**비유의 한계**: 목수는 새로운 도구를 상점에서 사온다.
에이전트는 **MCP**를 통해 새 도구를 추가할 수 있다. 이것은 4교시에서 다룬다.

---

### 실무에서의 의미

실제 프로젝트에서 에이전트가 도구를 사용하는 전형적인 흐름:

```
[Glob] "*.py" → 프로젝트의 Python 파일 목록 확인
[Read] src/auth/login.py → 로그인 로직 파악
[Grep] "token" → 토큰 관련 코드 위치 찾기
[Edit] src/auth/login.py → 버그 수정
[Bash] python -m pytest tests/ → 테스트 실행
[Bash] git diff → 변경사항 확인
```

에이전트는 이 도구들을 **조합**하여 작업한다.
하나의 도구가 아닌, 도구의 **조합 순서**가 작업의 품질을 결정한다.

**실제 사례 — @explore 서브에이전트의 도구 사용:**

```
[Glob] "*"                    → 최상위 파일 목록 (5개 Python 파일)
[Glob] "**/package.json"      → JS/TS 프로젝트? (없음 → Python으로 판단)
[Read] note.py                → CLI 진입점, argparse 명령어 구조
[Read] storage.py             → JSON 파일 기반 CRUD
[Read] test_note.py           → unittest 기반 테스트 24개
[Grep] "if __name__"          → 진입점: note.py:106
[Grep] "import argparse|..."  → 기술 스택: argparse, unittest, json
```

에이전트는 5번의 Glob으로 구조를 파악하고, 7번의 Read로 핵심 파일을 읽고, 2번의 Grep으로 패턴을 확인했다. **도구의 조합 순서**가 작업의 효율을 결정한다.

---

### WebSearch/WebFetch는 단순한 curl인가?

자주 묻는 질문이다.

| 구분 | curl / fetch | WebSearch/WebFetch |
|------|-------------|-------------------|
| 반환 형식 | 원시 HTML | 모델이 읽기 쉬운 **정제된 텍스트** |
| 검색 기능 | 없음 | 검색 엔진 통합 (키워드 → 결과) |
| 컨텍스트 최적화 | 없음 | 불필요한 HTML 태그, 광고, 네비게이션 제거 |
| 토큰 효율 | 낮음 (HTML 전체) | 높음 (**핵심 콘텐츠만**) |

에이전트가 `curl`로 웹페이지를 읽으면 HTML 태그가 컨텍스트를 낭비한다.
WebFetch는 **핵심 텍스트만 추출**하여 토큰을 절약한다.

→ 이 "토큰 절약"이 왜 중요한지는 2교시 **컨텍스트 엔지니어링**에서 다룬다.

### 주의사항과 흔한 오해

> **오해 1**: "도구가 많으면 에이전트가 더 잘한다"

→ **실제**: 도구가 많으면 에이전트가 **어떤 도구를 선택할지** 혼란스러워할 수 있다. 또한 도구 정의 자체가 컨텍스트를 소비한다. MCP 도구 정의만으로 컨텍스트의 40~50%를 차지하는 경우도 있다. **꼭 필요한 도구만** 연결하는 것이 좋다.

> **오해 2**: "Bash 도구가 있으니 뭐든 할 수 있다"

→ **실제**: Bash 도구는 강력하지만, 그만큼 **위험**하기도 하다. `rm -rf /`도 실행할 수 있다. 이것이 4교시에서 다룰 **권한과 샌드박스**가 중요한 이유다.

## 4. 코드 에이전트 생태계 개관

### 왜 이것이 중요한가

앞서 도구를 배웠다. 그런데 이 도구를 제공하는 **코드 에이전트**는 여러 종류가 있다.
어떤 것이 있는지, 어떤 차이가 있는지 알아야 자신에게 맞는 도구를 선택할 수 있다.

---

### 핵심 원리: 세 가지 유형

**① VSCode 플러그인형**: GitHub Copilot, Continue
→ IDE 안에서 자동완성·채팅 제공
→ 열린 파일 중심, 전체 프로젝트 접근 제한적

**② IDE 통합형**: Cursor, Windsurf
→ 에디터 자체가 AI 기능을 내장
→ 코드 편집과 AI 대화가 하나의 인터페이스

**③ CLI 코드 에이전트** (오늘의 주제): Claude Code, Codex, OpenCode, Gemini CLI, Aider
→ 터미널에서 실행, 프로젝트 **전체**에 접근
→ 셸 명령 직접 실행, 자동화·파이프라인 통합 가능

---

### CLI 에이전트의 장점 — 왜 터미널인가

| 장점 | 설명 |
|------|------|
| **전체 파일시스템 접근** | IDE는 열린 프로젝트만, CLI는 전체 시스템 |
| **셸 네이티브 실행** | 빌드, 테스트, git, docker 등을 직접 실행 |
| **Unix 파이프 활용** | `cat log \| agent "분석해"` 등 |
| **자동화 통합** | 비대화형 모드로 CI/CD, cron에 통합 |
| **서버 환경** | SSH를 통해 원격 서버에서 직접 실행 |
| **리소스 효율** | IDE 없이 가볍게 실행, 여러 인스턴스 병렬 가능 |

→ 참고: [Why CLIs Are Better for AI Coding Agents Than IDEs](https://www.firecrawl.dev/blog/why-clis-are-better-for-agents)

하지만 CLI와 IDE는 **상호 보완적**이다.
CLI 에이전트로 대규모 리팩토링을, IDE 도구로 빠른 자동완성을 사용하는 하이브리드 접근이 실용적이다.

### Q&A

**Q: 코드 에이전트가 내 코드를 외부로 보내나요? 보안은 괜찮은가요?**
A: 코드 에이전트는 사용자의 코드를 LLM API에 전송하여 처리한다. 이것은 LLM 채팅에 코드를 붙여넣는 것과 동일한 수준의 데이터 전송이다. 대부분의 주요 에이전트(Claude Code, Codex)는 학습에 사용자 데이터를 사용하지 않겠다고 명시하고 있다. 기업 환경에서는 온프레미스 모델이나 VPN을 고려할 수 있다. 더 깊이 알고 싶다면 **"LLM data privacy policy"**를 검색해 보십시오.

**Q: 무료 모델로도 충분한가요? (← 주니어 개발자 배경)**
A: 학습 목적으로는 충분하다. OpenCode Zen이나 OpenRouter의 무료 모델로 에이전틱 루프, 도구 사용, 프로젝트 규칙 등 핵심 개념을 모두 실습할 수 있다. 다만 무료 모델은 복잡한 리팩토링이나 대규모 코드베이스 작업에서는 유료 모델에 비해 정확도가 떨어질 수 있다. 개념을 익힌 후 실무에서는 유료 모델을 권장한다. 더 깊이 알고 싶다면 **"OpenRouter free models comparison"**을 검색해 보십시오.

**Q: Cursor 같은 IDE 에이전트와 CLI 에이전트, 뭘 써야 하나요?**
A: 둘 다 쓰는 것이 가장 실용적이다. Cursor/Windsurf는 코드 편집 중 빠른 자동완성과 인라인 채팅에 강하다. CLI 에이전트는 멀티파일 리팩토링, 셸 명령 실행, 자동화에 강하다. 실무에서는 "에디터에서 코드 작성 + CLI 에이전트로 대규모 작업"을 병행하는 개발자가 많다. 더 깊이 알고 싶다면 **"CLI vs IDE coding agent"**를 검색해 보십시오.

**Q: 에이전트가 코드를 수정하다가 내 작업을 망치면 어떡하나요?**
A: Git을 사용한다면 언제든 `git diff`로 변경사항을 확인하고 `git stash` 또는 `git checkout -- .`로 되돌릴 수 있다. 코드 에이전트를 사용할 때는 **항상 Git 커밋 상태를 깨끗하게 유지**하는 것이 좋다. 작업 전에 커밋하고, 에이전트 작업 후 diff를 검토하라. 4교시에서 권한과 샌드박스를 다룰 때 더 자세히 배운다. 더 깊이 알고 싶다면 **"git safety for AI coding agents"**를 검색해 보십시오.

**Q: 에이전틱 루프가 전통적인 CI/CD 파이프라인과 뭐가 다른 건가요? (← 개발자 배경)**
A: CI/CD 파이프라인은 **사전 정의된 고정 단계**를 순서대로 실행한다. "빌드 → 테스트 → 배포"처럼 분기가 미리 정해져 있다. 에이전틱 루프는 **LLM이 중간 결과를 보고 다음 행동을 동적으로 결정**한다. 테스트가 실패하면 에러를 분석하고, 코드를 수정하고, 다시 테스트하는 판단을 스스로 한다. CI/CD는 "무엇을 할지"가 고정, 에이전틱 루프는 "무엇을 할지"가 유동적이다. 더 깊이 알고 싶다면 **"ReAct pattern LLM agent"**를 검색해 보십시오.

<details>
<summary>💡 힌트</summary>
	도서관에서 책을 찾는 과정을 생각해 보십시오. 목록 검색과 실제 책 읽기의 차이는?
</details>

<details>
<summary>✅ 정답</summary>
	**Grep**은 코드베이스에서 **패턴을 검색**하여 어떤 파일의 몇 번째 줄에 해당 내용이 있는지 찾는다. **Read**는 특정 파일의 **전체 내용을 읽어** 컨텍스트에 추가한다.
	**설명:** Grep은 "어디에 있는지 찾기", Read는 "그 내용을 자세히 보기"다. 에이전트는 보통 Grep으로 위치를 찾고, Read로 해당 파일을 읽는 순서로 작업한다.
</details>

**Q4. 다음 시나리오에서 에이전트가 사용할 도구의 순서를 예측하시오: "test_utils.py에서 실패하는 테스트를 찾아 원인을 분석하고 수정해줘"**

A) Edit → Bash → Read
B) Bash → Read → Grep → Edit → Bash
C) Read → Edit → Bash
D) Grep → Read → Edit → Read

<details>
<summary>💡 힌트</summary>
	에이전트는 먼저 "어떤 테스트가 실패하는지" 확인해야 한다. 확인하려면 어떤 도구를 쓸까?
</details>

<details>
<summary>✅ 정답</summary>
	**B) Bash → Read → Grep → Edit → Bash**
	**설명:** ① Bash로 테스트를 실행하여 어떤 테스트가 실패하는지 확인 → ② Read로 실패한 테스트 파일 읽기 → ③ Grep으로 관련 소스 코드 검색 → ④ Edit으로 코드 수정 → ⑤ Bash로 테스트 재실행하여 통과 확인. 이것이 전형적인 에이전틱 루프다.
</details>

**Q5. "에이전트에게 도구가 많을수록 좋다"는 주장에 동의하는가? 근거를 포함하여 답하시오.**

<details>
<summary>💡 힌트</summary>
	도구 정의 자체가 에이전트의 컨텍스트 윈도우에서 차지하는 비중을 생각해 보십시오.
</details>

<summary>✅ 정답</summary>

</details>

<details>
<summary>✅ 정답</summary>
	정형화된 정답은 없다. 핵심은 **에이전트가 도구를 조합하여 작업하는 패턴을 관찰하는 것**이다.
	일반적인 패턴:
	- 파일 목록 확인 → `bash` (ls) 또는 `glob`
	- 파일 생성 → `write`
	- 파일 실행 → `bash` (python)
	- 파일 수정 → `read` → `edit`
	- 테스트 작성+실행 → `read` → `write` → `bash` → (실패 시) `read` → `edit` → `bash`
</details>

## 핵심 정리

- **코드 에이전트 ≠ LLM 채팅**: 에이전트는 파일을 직접 읽고·수정하고·실행한다
- **에이전틱 루프**: Observe → Think → Act 를 반복하며 자율적으로 문제를 해결한다
- **도구**: 모든 CLI 에이전트는 유사한 도구 세트를 제공한다 (Read, Edit, Bash, Grep 등)
- **CLI 에이전트의 장점**: 전체 파일시스템 접근, 셸 실행, 자동화 통합
- **오늘의 실습 도구**: OpenCode — 무료 모델로 핵심 개념을 모두 학습 가능
