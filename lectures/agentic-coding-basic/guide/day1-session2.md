## 1. 에이전틱 루프 심화 — 도구 호출 흐름

### 왜 이것이 중요한가

1교시에서 에이전틱 루프의 **개념**을 배웠다.
Observe → Think → Act 가 반복된다는 것을 알았다.

이번에는 한 단계 더 들어간다.
에이전트가 **실제로 어떤 도구를 어떤 순서로 호출**하는지 관찰한다.

왜 이것이 중요한가?
도구 호출 순서를 읽을 수 있으면, 에이전트가 **왜 실패했는지** 진단할 수 있다.
"잘못된 파일을 읽었구나", "테스트를 실행하지 않았구나" — 이런 판단이 가능해진다.

→ 이것이 "에이전트를 디버깅하는 능력"이다.

---

### 핵심 원리

에이전트가 "TODO 앱에 삭제 기능을 추가해"라는 요청을 받으면:

**전형적인 도구 호출 순서:**

```
[1] Glob: *.py → 프로젝트의 Python 파일 목록 확인
[2] Read: todo_app.py → 기존 코드 구조 파악
[3] Read: test_app.py → 기존 테스트 확인
[4] Think: "delete_todo 함수를 추가하고, 테스트도 작성해야겠다"
[5] Edit: todo_app.py → delete_todo() 함수 추가
[6] Edit: test_app.py → test_delete_todo() 추가
[7] Bash: python -m pytest test_app.py → 테스트 실행
[8] Observe: "1 failed" → 에러 분석
[9] Edit: todo_app.py → 버그 수정
[10] Bash: python -m pytest test_app.py → 재실행
[11] Observe: "all passed" → 완료
```

**핵심 패턴**: `탐색(1~3) → 계획(4) → 구현(5~6) → 검증(7) → 피드백(8~10) → 확인(11)`

이 패턴을 알면, 에이전트에게 더 나은 지시를 내릴 수 있다.
예를 들어, "test_app.py에 테스트도 추가하고 실행해서 통과하는지 확인해"라고 말하면 에이전트가 검증 단계를 건너뛰지 않는다.

---

### 실무에서의 의미

실제 프로젝트에서는 도구 호출이 10~20회 이상 이어지기도 한다.
복잡한 버그 수정의 경우:

1. `Grep`으로 에러 메시지 검색 → 관련 파일 5개 발견
2. `Read`로 5개 파일 모두 읽기
3. `Think`로 원인 분석
4. `Edit`으로 2개 파일 수정
5. `Bash`로 테스트 실행 → 실패
6. `Read`로 에러 로그 분석 → 다른 파일에도 영향 발견
7. `Edit`으로 추가 수정
8. `Bash`로 재실행 → 성공

이 전체 과정이 **사람의 개입 없이** 진행된다.

---

### 주의사항과 흔한 오해

> **오해 1**: "에이전트가 항상 최적의 도구 순서를 선택한다"

→ **실제**: 에이전트도 실수한다. 불필요한 파일을 읽거나, 검색 없이 바로 수정에 뛰어드는 경우가 있다. 이럴 때 **"먼저 관련 파일을 검색하고"** 같은 지시를 추가하면 도움이 된다.

> **오해 2**: "도구 호출 로그를 볼 필요 없다"

→ **실제**: 도구 호출 로그는 에이전트의 **사고 과정**을 보여준다. 에이전트가 왜 이상한 결과를 내는지 이해하려면 로그를 읽어야 한다. OpenCode TUI의 하단 로그, Claude Code의 도구 호출 표시를 항상 확인하라.

## 2. 컨텍스트 엔지니어링 — 에이전트의 기억력을 설계하다

### 왜 이것이 중요한가

1교시에서 에이전틱 루프와 도구를 배웠다.
에이전트가 도구를 사용하며 자율적으로 작업한다는 것을 알았다.

그런데 여기서 자연스러운 질문이 생긴다.
**"에이전트가 읽은 파일, 실행한 결과, 대화 이력 — 이 모든 것이 어디에 저장되는가?"**

그 답이 **컨텍스트 윈도우**다.
컨텍스트 윈도우는 에이전트의 **작업 메모리**와 같다.

2026년, 프롬프트 엔지니어링은 **컨텍스트 엔지니어링**으로 진화했다.
단순히 "질문을 잘 쓰는 것"이 아니라, 에이전트에게 제공하는 **전체 정보 환경**을 설계하는 것이 핵심이다.

→ 참고: [Context Engineering for AI Agents (Anthropic)](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)

---

### 핵심 원리

**비유로 이해하기**: 칠판이 있는 회의실을 떠올려 보십시오.

→ **칠판** = 컨텍스트 윈도우. 크기가 고정되어 있다.
→ 회의가 진행될수록 칠판이 **가득 찬다** (대화, 파일 내용, 실행 결과가 쌓임).
→ 칠판이 가득 차면 **이전 내용을 지우고** 요약을 써야 한다 (compaction).
→ 지울 때 **중요한 내용이 함께 지워질 수 있다**.

**실제 동작 — 컨텍스트 윈도우의 구조**:

[Claude Platform 공식 문서](https://platform.claude.com/docs/en/build-with-claude/context-windows)에 따르면, 컨텍스트 윈도우는 다음과 같이 구성된다:

```
┌─────────────────────────────────────┐
│        컨텍스트 윈도우 (1M 토큰)        │
│                                     │
│  ┌─ 입력 (Input) ──────────────┐   │
│  │  시스템 프롬프트              │   │
│  │  도구 정의 (MCP 포함)        │   │
│  │  프로젝트 규칙 (AGENTS.md)   │   │
│  │  이전 대화 히스토리          │   │
│  │  현재 사용자 메시지          │   │
│  └─────────────────────────────┘   │
│                                     │
│  ┌─ 출력 (Output) ─────────────┐   │
│  │  에이전트 응답                │   │
│  │  도구 호출 요청              │   │
│  └─────────────────────────────┘   │
│                                     │
│  → 다음 턴에서:                      │
│    이전 출력이 입력에 추가됨          │
│    → 컨텍스트가 선형적으로 증가       │
└─────────────────────────────────────┘
```

> **핵심**: 대화가 진행될수록 컨텍스트가 **선형적으로 증가**한다. 이전 턴의 모든 내용이 다음 턴의 입력에 포함된다.

**비유의 한계**: 칠판은 내용을 선택적으로 지울 수 있지만, 컨텍스트 윈도우의 compaction은 **LLM이 자동으로 요약**한다. 사람이 "이건 남기고 저건 지워"라고 세밀하게 제어하기 어렵다.

---

### Claude Opus 4.6의 1M 컨텍스트 — 왜 중요한가

Claude Opus 4.6이 **1M 토큰** 컨텍스트로 확장되었다.
이것은 코드 에이전트 사용자들이 오래 기다려온 업데이트다.

→ 참고: [Claude Opus 4.6 — 1M Context GA](https://claude.com/blog/1m-context-ga)

**왜 필요했는가?**
- 대규모 코드베이스를 한 세션에서 탐색하려면 기존 200K 토큰은 부족했다
- 에이전틱 루프가 반복될수록 컨텍스트가 빠르게 소진되는 문제
- 서브에이전트에 위임하지 않고도 복잡한 작업을 한 세션에서 완료 가능

**Claude 모델의 컨텍스트 인지 (Context Awareness)**:

[공식 문서](https://platform.claude.com/docs/en/build-with-claude/context-windows)에 따르면, Claude Sonnet 4.6 이상 모델은 **컨텍스트 인지** 기능을 갖추고 있다.

세션 시작 시:
```xml
<budget:token_budget>1000000</budget:token_budget>
```

도구 호출 후:
```xml
<system_warning>Token usage: 35000/1000000; 965000 remaining</system_warning>
```

→ 에이전트가 **남은 공간을 인지**하고, 작업을 효율적으로 수행할 수 있다.

---

### 컨텍스트가 차면 성능이 떨어지는가? — 실전 논쟁

> 이 주제는 커뮤니티에서 활발히 논쟁 중이다. 양쪽 시각을 모두 다룬다.

**"/compact로 충분하다" 진영**:
- 자동 압축(compaction)이 핵심 정보를 유지하면서 공간을 확보한다
- Claude Code는 자동으로 컨텍스트를 요약·압축하는 메커니즘이 있다
- `/compact` 후 작업을 계속하면 충분히 효과적이라는 의견

**"새 세션을 시작하는 게 낫다" 진영**:
- **"Lost in the Middle" 현상**: LLM은 긴 컨텍스트의 **중간 부분** 정보를 놓치는 경향이 있다
  - 초반·후반 정보 정확도: 85~95%
  - 중간 정보 정확도: 76~82%
- **컨텍스트 로트(Context Rot)**: [Chroma Research 2025](https://research.trychroma.com/context-rot)에서 18개 최신 모델(GPT-4.1, Claude Opus 4, Gemini 2.5 포함)을 대상으로 **194,480회** LLM 호출 실험을 수행한 결과, **모든** 모델에서 컨텍스트가 길어질수록 성능 하락을 확인했다
  - 200K 윈도우 모델이 50K 토큰부터 성능 하락 시작
  - NIAH(바늘 찾기) 벤치마크에서는 모든 모델이 거의 완벽한 점수를 받지만, 이것은 **단순 어휘 검색**만 측정한다. 실제 의미적 작업에서는 입력 길이에 따라 비균일하게 성능이 저하된다
  - 이것은 학습이 아닌 **트랜스포머 아키텍처의 구조적 한계**
  - > "LLM은 10,000번째 토큰을 100번째 토큰만큼 신뢰성 있게 처리하지 못한다" — Chroma Research

→ 참고: [Context Rot (Chroma Research 원문)](https://research.trychroma.com/context-rot)
→ 참고: [Context Rot: Why AI Gets Worse the Longer You Chat](https://www.producttalk.org/context-rot/)
→ 참고: [Understanding LLM performance degradation (Stefano Demiliani)](https://demiliani.com/2025/11/02/understanding-llm-performance-degradation-a-deep-dive-into-context-window-limits/)

### 컨텍스트 로트 — 왜 발생하는가?

컨텍스트 로트의 두 가지 핵심 메커니즘:

**① Distractor 누적 효과**
- 관련 없지만 **비슷한 정보**가 컨텍스트에 쌓일수록 성능이 급격히 떨어진다
- 모델마다 반응이 다르다:
  - **Claude 계열**: 불확실하면 **응답을 거부**한다 (낮은 환각율)
  - **GPT 계열**: 틀린 답을 **자신 있게 제시**한다 (높은 환각율)
- 에이전트 맥락: 한 세션에 이전 작업 컨텍스트가 쌓여 있으면 새 작업의 정확도가 저하된다

**② 의미적 유사도 함정**
- 질문과 찾아야 할 정보의 **키워드가 다를수록** 성능이 더 빠르게 하락한다
- NIAH 벤치마크: 정확한 키워드 매칭 → 높은 점수
- 실무: 의미적 이해 필요 → **실제 성능 저하가 더 심함**
- 벤치마크 점수를 실무 성능으로 착각하지 말 것

**에이전트에서의 실제 영향:**
- 버그 수정 후 같은 세션에서 새 기능 추가 → 이전 디버깅 맥락이 Distractor로 작용
- 10개 파일을 통째로 컨텍스트에 로드 → 필요한 3개 함수만 넣는 것보다 성능 저하
- "이 코드 개선해줘" (키워드 불일치) → "validate 함수의 null 체크 추가"보다 성능 낮음

→ 이 실험은 단어 따라 쓰기 같은 **단순 태스크** 기반이다. 코드 수정, 요약 같은 실제 에이전트 작업에서는 성능 저하가 **더 심하다**.

### 실전 컨텍스트 관리 전략

| 전략 | 설명 | 언제 사용 |
|------|------|----------|
| **하나의 작업 = 하나의 세션** | 세션을 작게 유지 | 항상 (기본 원칙) |
| **`/clear`** | 컨텍스트 초기화 | 무관한 작업 간 전환 시 |
| **`/compact`** | 이전 내용 요약·압축 | 장기 작업 중간 |
| **서브에이전트 위임** | 탐색 작업을 별도 컨텍스트에서 실행 | 메인 컨텍스트 보호 시 |
| **새 세션** | 완전히 새로 시작 | 2회 이상 같은 실패 반복 시 |

> **"주방 싱크" 지양**: 하나의 세션에 버그 수정, 리팩토링, 새 기능 추가를 모두 섞지 마라. 각각 별도 세션으로 분리한다.

---

### 다른 접근법과의 비교

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|------|-------------------|-------------------|
| **범위** | 사용자 메시지 텍스트 | 시스템 프롬프트 + 도구 정의 + 대화 이력 + 파일 내용 **전체** |
| **관심사** | "질문을 어떻게 쓸까" | "에이전트의 전체 정보 환경을 어떻게 설계할까" |
| **제어 가능 요소** | 프롬프트 텍스트 | 프로젝트 규칙, 도구 선택, 세션 관리, 메모리 |
| **시대** | 2023~2024 | **2025~2026** |

---

### 주의사항과 흔한 오해

> **오해 1**: "1M 토큰이니까 컨텍스트 관리를 신경 쓸 필요 없다"

→ **실제**: 컨텍스트 로트는 **토큰 수와 무관하게** 발생한다. 1M 토큰이 있어도 50K 토큰부터 성능이 떨어질 수 있다. 큰 컨텍스트는 "더 많이 담을 수 있다"는 뜻이지, "무한히 담아도 괜찮다"는 뜻이 아니다.

> **오해 2**: "프롬프트를 잘 쓰면 컨텍스트 관리는 필요 없다"

→ **실제**: 아무리 좋은 프롬프트를 쓰더라도, 시스템 프롬프트에 불필요한 도구 정의가 40%를 차지하면 효과가 반감된다. 컨텍스트 엔지니어링은 프롬프트를 **포함하면서** 더 넓은 범위를 다룬다.

→ 더 깊이 알고 싶다면 **"Context Rot LLM"**을 검색해 보십시오.

## 3. 효과적인 프롬프팅 — 에이전트에게 좋은 지시 내리기

### 왜 이것이 중요한가

앞서 컨텍스트 엔지니어링의 "큰 그림"을 배웠다.
이번에는 그 안에서 **사용자가 직접 제어하는 부분** — 프롬프트 작성법을 다룬다.

같은 작업을 요청하더라도, **어떻게 요청하느냐**에 따라 결과가 극적으로 달라진다.
이것은 에이전트가 "멍청해서"가 아니라, **모호한 입력에는 모호한 출력**이 나오기 때문이다.

→ 참고: [Claude Code 모범 사례](https://code.claude.com/docs/ko/best-practices)

---

### 핵심 원리: 6가지 프롬프팅 원칙

| 원칙 | 설명 |
|------|------|
| **① 검증 수단 제공** | 테스트, 예상 출력을 포함해 에이전트가 스스로 확인하게 한다 |
| **② 탐색 → 계획 → 구현** | 바로 코딩하지 않고, 먼저 코드를 읽고 계획을 세운다 (Plan 모드) |
| **③ 구체적 컨텍스트** | 파일, 제약사항, 기존 패턴을 명시한다 |
| **④ 작업 분해** | 큰 작업을 독립적인 작은 단위로 나눈다 |
| **⑤ TDD + 에이전트** | 테스트를 먼저 작성하면 에이전트가 스펙으로 활용한다 |
| **⑥ 검토 후 적용** | 에이전트 결과를 맹신하지 않고 diff를 검토한다 |

### 나쁜 예 vs 좋은 예

<columns>
	<column>
		**❌ 나쁜 프롬프트**
		```
		로그인 버그 수정해줘
		```
		→ 어떤 버그?
		→ 어떤 파일?
		→ 어떻게 확인?
	</column>
	<column>
		**✅ 좋은 프롬프트**
		```

		src/auth/ 토큰 리프레시 로직 확인.
		재현 테스트 작성 후 수정해줘.
		```

		→ 검증 수단
	</column>

**차이가 생기는 이유:**
- 나쁜 프롬프트: 에이전트가 "로그인"이라는 키워드로 **추측**해야 한다
- 좋은 프롬프트: 에이전트가 정확한 **위치와 검증 수단**을 갖고 시작한다

### `@` 파일 참조 — 파일명을 "쓰는 것"과 "첨부하는 것"의 차이

프롬프트에 파일명을 텍스트로 쓰면, 에이전트는 그 파일을 **직접 찾아서 읽어야** 한다.
`@`를 붙여서 파일을 참조하면, 해당 파일의 **내용이 컨텍스트에 즉시 첨부**된다.

| 방식 | 예시 | 에이전트의 동작 |
|------|------|----------------|
| **텍스트로 파일명 작성** | `src/auth.py 수정해줘` | 에이전트가 Glob/Read 도구로 파일을 **찾아서 읽는** 단계 필요 |
| **`@` 파일 참조** | `@src/auth.py 수정해줘` | 파일 내용이 **즉시 컨텍스트에 포함** → 바로 작업 시작 |

**왜 이것이 중요한가?**

1. **도구 호출 절약**: 에이전트가 파일을 찾고 읽는 1~2회의 도구 호출을 건너뛸 수 있다
2. **정확성 향상**: 에이전트가 비슷한 이름의 다른 파일을 잘못 선택할 가능성이 없다
3. **컨텍스트 확보**: 대화 시작부터 파일 내용이 포함되므로, 에이전트의 첫 응답 품질이 높아진다

**사용법 (도구별):**

```bash
# OpenCode — 입력 창에서 @와 파일 경로 입력
@src/auth.py 이 파일의 토큰 리프레시 로직을 분석해줘

# Claude Code — 동일한 문법
@src/auth.py @tests/test_auth.py 두 파일을 비교해서 빠진 테스트 케이스 알려줘

# 여러 파일을 동시에 첨부 가능
@src/models.py @src/routes.py @tests/test_routes.py 이 세 파일 기반으로 리팩토링해줘
```

> **팁**: `@`를 입력하면 **자동 완성**이 뜬다. 파일 경로를 외울 필요 없이, 탐색하며 선택할 수 있다.

> **주의**: 큰 파일을 여러 개 첨부하면 컨텍스트를 빠르게 소비한다. 필요한 파일만 선택적으로 첨부하라.

### Plan 모드 — "먼저 생각하고, 나중에 행동하라"

대부분의 CLI 에이전트는 **Plan 모드**를 지원한다.
코드를 수정하지 않고, **읽기 전용으로 분석·계획만** 수행하는 모드다.

| 도구 | Plan 모드 진입 |
|------|--------------|
| OpenCode | `Tab` → Plan 에이전트 |
| Claude Code | `Shift+Tab` 또는 `/plan` |
| Codex | `Shift+Tab` 또는 `/plan` |

**추천 워크플로우**:
1. **Plan 모드**로 코드를 읽고 계획 수립
2. 계획을 검토한 후 **Build 모드**로 전환
3. Build 모드에서 구현

→ 바로 코딩에 뛰어들면 에이전트가 잘못된 방향으로 갈 확률이 높다.

---

### Build vs Plan: 무엇이 다른가?

| | **Build 모드** | **Plan 모드** |
|---|---|---|
| **역할** | 코드 작성·수정 | 분석·설계 |
| **도구 접근** | 모든 도구 활성화 | 읽기 전용 (edit/bash 비활성화) |
| **위험도** | 파일을 직접 변경 | 변경 없음 (안전) |
| **기본값** | ✅ 기본 모드 | `Tab` / `Shift+Tab` / `/plan` |

**판단 기준:**
- **Build로 바로 가도 되는 경우**: "README.md에 설치 방법 추가해" — 단일 파일, 저위험, 명확한 범위
- **Plan을 먼저 거쳐야 하는 경우**: "인증을 JWT에서 OAuth2로 마이그레이션해" — 다수 파일, 고위험, 영향 범위 불확실

→ "이 작업이 잘못되면 얼마나 큰일인가?" — 답이 "꽤 큰일"이면 Plan 모드부터 시작하라.

### 주의사항과 흔한 오해

> **오해 1**: "자세하게 쓸수록 좋다"

→ **실제**: 핵심은 **관련 있는 구체적 정보**를 제공하는 것이다. 불필요한 배경 설명이 길어지면 오히려 핵심이 묻힌다. "무엇을 하고, 어떤 파일에서, 어떻게 확인할 것인지" 세 가지에 집중하라.

> **오해 2**: "에이전트에게 계획을 세우라고 하면 시간 낭비다"

→ **실제**: Plan 모드는 5분을 "낭비"하여 30분의 삽질을 방지한다. 특히 복잡한 작업에서 Plan 모드로 먼저 구조를 파악하면, 에이전트가 잘못된 방향으로 가는 것을 사전에 막을 수 있다.

### Q&A

**Q: /compact를 하면 정보가 손실되지 않나요?**
A: 일부 손실이 발생할 수 있다. compaction은 LLM이 이전 대화를 **요약**하는 과정이다. 요약 과정에서 미묘한 맥락(이전 시도와 실패 이유, 코드의 의도)이 빠질 수 있다. 그래서 "하나의 작업 = 하나의 세션" 원칙이 중요하다. 작업이 명확하면 요약해도 핵심이 보존된다. 더 깊이 알고 싶다면 **"LLM compaction information loss"**를 검색해 보십시오.

**Q: 1M 토큰이면 코드 파일 몇 개 정도 담을 수 있나요? (← 개발자 배경)**
A: 평균적인 코드 파일이 200~500줄(약 1,000~3,000 토큰)이라고 가정하면, 1M 토큰에 약 **300~1,000개 파일**을 담을 수 있다. 하지만 실제로는 시스템 프롬프트, 도구 정의, 대화 이력도 토큰을 소비한다. 또한 컨텍스트 로트로 인해 전부 담는 것보다 **필요한 것만** 담는 것이 효과적이다. 더 깊이 알고 싶다면 **"token estimation code files"**를 검색해 보십시오.

**Q: 프롬프트 엔지니어링을 열심히 배웠는데, 이제 무용한 건가요?**
A: 전혀 아니다. 컨텍스트 엔지니어링은 프롬프트 엔지니어링을 **포함**하는 상위 개념이다. 좋은 프롬프트 작성 능력은 여전히 핵심이다. 다만 이제는 프롬프트 **외에도** 시스템 프롬프트, 도구 선택, 세션 관리, 메모리 설계까지 고려해야 한다는 것이다. 더 깊이 알고 싶다면 **"context engineering vs prompt engineering"**을 검색해 보십시오.

**Q: 컨텍스트 로트가 트랜스포머의 구조적 한계라면, 해결할 수 없는 건가요?**
A: 현재까지는 완전한 해결책은 없다. Chroma 연구에 따르면 18개 최신 모델 **모두** 이 현상을 보인다. 하지만 모델 아키텍처 개선(더 효율적인 어텐션 메커니즘)과 하네스 수준의 대응(서브에이전트 위임, 세션 분리, 요약 전략)으로 **영향을 최소화**할 수 있다. 실전에서는 "세션을 작게 유지"하는 것이 가장 효과적인 대응이다. 더 깊이 알고 싶다면 **"transformer attention context rot"**을 검색해 보십시오.

**Q: TDD + 에이전트 조합이 정말 효과적인가요? (← 개발자 배경)**
A: 매우 효과적이다. 테스트를 먼저 작성하면 에이전트에게 **명확한 스펙**이 생긴다. "이 테스트를 통과하는 코드를 작성해"는 "이 기능을 구현해"보다 훨씬 구체적이다. 에이전트가 테스트를 실행하고 결과를 확인할 수 있으므로, **자체 검증 루프**가 완성된다. Addy Osmani도 "테스트를 먼저 작성하고 에이전트에게 반복시키라"고 권장한다. 더 깊이 알고 싶다면 **"TDD with AI coding agents"**를 검색해 보십시오.

**Q5. "Plan 모드"를 사용해야 하는 상황과 바로 "Build 모드"로 시작해도 되는 상황을 각각 1개씩 예시를 들어 설명하시오.**

<details>
<summary>💡 힌트</summary>
	작업의 복잡도와 영향 범위를 기준으로 생각해 보십시오.
</details>

<details>
<summary>✅ 정답</summary>
	**Plan 모드가 필요한 상황**: "인증 시스템을 JWT에서 OAuth2로 마이그레이션해" — 여러 파일에 영향을 미치고, 잘못된 방향으로 가면 대규모 롤백이 필요하다. 먼저 영향 범위를 파악하고 계획을 세워야 한다.
	**바로 Build 모드 가능한 상황**: "README.md에 설치 방법 섹션 추가해" — 단일 파일, 단순 작업이므로 계획 없이 바로 실행해도 위험이 적다.
	**설명:** Plan 모드는 "잘못되면 되돌리기 어려운 작업"에 필수적이다. 단순하고 영향 범위가 좁은 작업은 바로 Build 모드로 시작해도 무방하다.
</details>

	|------|:------------------------:|:------------------------:|
	| 토큰 | 446,704 | **299,831** (-33%) |

	| 소요 시간 | 2분 45초 | **2분 9초** |

	- **grep이 사라졌다**: AGENTS.md가 파일 역할을 알려주니 키워드 검색이 불필요해짐

## 핵심 정리

- **에이전틱 루프 심화**: 도구 호출 순서(탐색 → 계획 → 구현 → 검증)를 읽으면 에이전트를 디버깅할 수 있다
- **컨텍스트 엔지니어링**: 프롬프트만이 아니라 에이전트의 **전체 정보 환경**을 설계한다
- **컨텍스트 로트**: 컨텍스트가 길어지면 모든 모델에서 성능이 저하된다 → 세션을 작게 유지
- **효과적인 프롬프팅**: 구체적 위치 + 명확한 스펙 + 검증 수단이 핵심
- **Plan 모드**: 복잡한 작업은 먼저 계획을 세운 후 구현한다
