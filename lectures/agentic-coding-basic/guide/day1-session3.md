## 1. 프로젝트 규칙 (AGENTS.md)

### 왜 이것이 중요한가

2교시에서 컨텍스트 엔지니어링을 배웠다.
에이전트에게 제공하는 **전체 정보 환경**을 설계하는 것이 핵심이라고 했다.

그렇다면 가장 먼저 설계해야 할 것은 무엇인가?
바로 **프로젝트 규칙**이다.

프로젝트 규칙은 에이전트에게 "이 프로젝트에서는 이렇게 해"라고 지시하는 **마크다운 파일**이다.
모든 세션 시작 시 시스템 프롬프트에 **자동으로** 로드된다.

규칙이 없으면 에이전트는 **일반적인 관행**으로 추측한다.
추측이 맞을 때도 있지만, 프로젝트마다 고유한 컨벤션이 있다.
"우리 프로젝트는 `uv`로 패키지를 관리한다" — 이건 에이전트가 코드를 읽어서 알 수 없다.

규칙이 있으면 에이전트는 추측 대신 **지시를 따른다**.
결과의 일관성과 품질이 눈에 띄게 달라진다.

→ 참고: [Claude Code 공식 문서 — 메모리](https://code.claude.com/docs/ko/memory)

### 핵심 원리

**비유로 이해하기**: 새로 입사한 개발자를 떠올려 보십시오.

→ 첫 출근 날, 아무런 안내 문서 없이 "코드 짜세요" — 혼란스럽다.
→ 팀 위키에 "코드 스타일, 브랜치 전략, 테스트 방법"이 정리되어 있다 — 바로 적응한다.

프로젝트 규칙은 **에이전트를 위한 온보딩 문서**다.
에이전트도 신입 개발자와 마찬가지로, 프로젝트 고유의 규칙을 알려줘야 잘 따른다.

**실제 동작 — 계층적 설정:**

```
전역 규칙 (모든 프로젝트에 적용)
    ↓ 병합
프로젝트 규칙 (이 프로젝트에 적용)
    ↓ 병합
디렉토리 규칙 (이 디렉토리에 적용)
```

더 가까운 파일이 **우선**한다.
전역에서 "탭 사용"이라고 해도, 프로젝트 규칙에서 "스페이스 사용"이라고 하면 스페이스가 적용된다.

| 도구 | 파일명 | 전역 위치 | 프로젝트 위치 |
|------|--------|----------|-------------|
| Claude Code | `CLAUDE.md` | `~/.claude/CLAUDE.md` | 프로젝트 루트 `CLAUDE.md` |
| Codex | `AGENTS.md` | `~/.codex/AGENTS.md` | 프로젝트 루트 `AGENTS.md` |
| OpenCode | `AGENTS.md` | `~/.config/opencode/AGENTS.md` | 프로젝트 루트 `AGENTS.md` |
| Gemini CLI | `GEMINI.md` | - | 프로젝트 루트 `GEMINI.md` |

**비유의 한계**: 신입 개발자는 온보딩 문서를 "대충 읽고 넘기기"도 한다.
에이전트는 규칙을 **거의 100% 따르려 한다**.
이것이 장점이자 위험이다 — **나쁜 규칙도 충실히 따른다**.

→ 참고: [OpenCode 규칙 문서](https://opencode.ai/docs/rules/)

### CLAUDE.md 상세 구조 (Claude Code 공식 문서)

> Claude Code의 CLAUDE.md는 3단계 계층 구조로 동작한다.
> 참고: [Claude Code 공식 문서 — 메모리](https://code.claude.com/docs/ko/memory)

**3단계 계층 구조:**

| 범위 | 위치 | 목적 | 공유 대상 |
|------|------|------|----------|
| **관리 정책** | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md` | IT/DevOps 관리 조직 전체 지침 | 조직의 모든 사용자 (제외 불가) |
| **프로젝트 지침** | `./CLAUDE.md` 또는 `./.claude/CLAUDE.md` | 팀 공유 코딩 표준 | 소스 제어를 통한 팀 멤버 |
| **사용자 지침** | `~/.claude/CLAUDE.md` | 개인 선호도 | 본인만 (모든 프로젝트) |

더 구체적인 위치가 더 광범위한 위치보다 **우선**한다.
작업 디렉토리 위의 CLAUDE.md 파일은 시작 시 전체 로드되고, 하위 디렉토리의 CLAUDE.md는 해당 파일을 읽을 때 **지연 로드**된다.

**`.claude/rules/` — 경로별 규칙:**

더 큰 프로젝트에서는 `.claude/rules/` 디렉토리에 주제별 규칙 파일을 분리할 수 있다:

```
your-project/
├── .claude/
│   ├── CLAUDE.md           # 주 프로젝트 지침
│   └── rules/
│       ├── code-style.md   # 코드 스타일
│       ├── testing.md      # 테스트 규칙
│       └── security.md     # 보안 요구사항
```

YAML frontmatter의 `paths` 필드로 특정 파일 경로에만 규칙을 적용할 수 있다:

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API 개발 규칙
- 모든 API 엔드포인트는 입력 검증 포함
- 표준 오류 응답 형식 사용
```

이 규칙은 `src/api/` 디렉토리의 TypeScript 파일을 읽을 때만 로드된다.
경로 범위가 없는 규칙은 모든 세션에 무조건 로드된다.

**`@import` — 파일 가져오기:**

CLAUDE.md 파일은 `@path/to/import` 구문으로 외부 파일을 가져올 수 있다:

```markdown
프로젝트 개요는 @README를 참조하세요.
npm 명령은 @package.json을 참조하세요.
git 워크플로우는 @docs/git-instructions.md를 따르세요.
```

- 상대 경로는 가져오기를 포함하는 파일 기준으로 해석
- 최대 **5홉** 깊이까지 재귀적 가져오기 가능
- 개인 파일도 가져올 수 있다: `@~/.claude/my-project-instructions.md`

**효과적인 CLAUDE.md 작성 원칙** (공식 문서 기반):

1. **크기**: 파일당 **200줄 이하** 목표 — 길수록 컨텍스트 소비 증가, 준수율 감소
2. **구체성**: 검증 가능할 정도로 구체적으로 — "코드를 제대로 포맷하세요" ❌ → "`2칸 들여쓰기 사용`" ✅
3. **일관성**: 두 규칙이 모순되면 에이전트가 임의 선택 — 주기적으로 충돌 제거
4. **자동 생성 주의**: `/init`은 출발점으로만 사용, CLAUDE.md가 이미 있으면 덮어쓰지 않고 개선 사항 제안

> 모노레포에서 다른 팀의 CLAUDE.md가 로드되면 `claudeMdExcludes` 설정으로 제외할 수 있다:
> ```json
> { "claudeMdExcludes": ["**/monorepo/CLAUDE.md", "/path/to/other-team/.claude/rules/**"] }
> ```

### 자동 메모리

> CLAUDE.md는 사람이 작성하는 지침이다. **자동 메모리**는 Claude가 스스로 작성하는 노트다.

| | CLAUDE.md 파일 | 자동 메모리 |
|---|---------------|-----------|
| **작성자** | 사용자 | Claude |
| **내용** | 지침 및 규칙 | 학습 및 패턴 |
| **범위** | 프로젝트, 사용자 또는 조직 | 작업 트리당 |
| **로드** | 모든 세션 (전체) | 모든 세션 (**처음 200줄**) |
| **용도** | 코딩 표준, 워크플로우, 아키텍처 | 빌드 명령, 디버깅 인사이트, 선호도 |

**저장 위치**: `~/.claude/projects/<project>/memory/`

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 인덱스 — 처음 200줄만 매 세션 로드
├── debugging.md       # 상세 노트 (필요 시 읽기)
├── api-conventions.md # 주제별 파일
└── ...
```

- `MEMORY.md`의 **처음 200줄**만 모든 세션에 로드됨 (CLAUDE.md는 길이 무관 전체 로드)
- 상세 노트는 별도 주제 파일로 관리, 필요 시 Claude가 직접 읽음
- 동일 Git 저장소의 모든 worktree가 **하나의 자동 메모리** 공유
- `/memory` 명령으로 저장된 내용 감사·편집 가능
- "항상 npm 대신 pnpm 사용해"라고 말하면 Claude가 자동 메모리에 저장
- `/compact` 후에도 CLAUDE.md는 디스크에서 다시 읽어 세션에 재주입됨 — 압축에 안전

> 자동 메모리는 기본 활성화. 비활성화: 설정에서 `autoMemoryEnabled: false` 또는 환경 변수 `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`
>
> → 참고: [Claude Code 공식 문서 — 메모리](https://code.claude.com/docs/ko/memory)

### 학술적 근거: ETH Zurich 연구 (2026)

> 프로젝트 규칙의 효과는 학술 연구로 검증되었다 — 그리고 결과는 놀랍다.

논문: ["Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?"](https://arxiv.org/abs/2602.11988)
- 저자: Gloaguen, Mundler, Muller, Raychev, Vechev (ETH Zurich SRI Lab)
- **60,000개 이상**의 리포지토리에서 사용되는 프로젝트 규칙 파일의 첫 대규모 실증 연구

**핵심 발견**:

| 조건 | 성능 변화 | 비용 변화 |
|------|----------|----------|
| **LLM 자동 생성** 규칙 | **-0.5~-2%** 하락 | **+20%** 증가 |
| **인간 작성** 규칙 | **+4%** 향상 | **+19%** 증가 |
| 차이 | **6%p** (인간 우위) | 비슷 |

→ **인간이 작성한 규칙만 효과가 있다.**
→ LLM이 자동 생성한 규칙은 8개 실험 중 5개에서 오히려 성능을 떨어뜨렸다.

**효과적인 규칙 작성 가이드라인** (논문 기반):
1. LLM으로 자동 생성하지 말 것 — 8개 실험 중 5개에서 성능 저하
2. **최소한으로** 작성 — 규칙이 길수록 "불필요한 탐색"을 유발하여 비용만 증가
3. **도구 지시에 집중** — `uv` 사용 지시 시 에이전트의 도구 사용률 **1.6배** 증가
4. 코드베이스 개요(디렉토리 구조)는 쓰지 말 것 — 에이전트가 스스로 탐색하며, 개요가 도움 되지 않음
5. README와 중복하지 말 것

→ 참고: [논문 전문](https://arxiv.org/html/2602.11988v1)

### "순응의 역설" — 규칙의 양날의 검

> 에이전트는 규칙을 **충실히 따른다**. 문제는, **나쁜 규칙도 따른다**는 것이다.

이것이 ETH Zurich 연구가 밝힌 **"순응의 역설(Compliance Paradox)"**이다.

**구체적 시나리오 1**:
- "모든 변경 전에 전체 테스트 스위트를 실행하라"는 규칙을 넣었다
- 에이전트는 충실히 따른다
- 하지만 1줄 수정에도 10분짜리 테스트를 매번 실행한다
- 비용은 20% 증가하는데, 성공률은 올라가지 않는다

**구체적 시나리오 2**:
- "코드 수정 전에 디렉토리 구조를 전부 파악하라"는 규칙
- 에이전트가 불필요한 파일까지 모두 읽는다
- 컨텍스트가 빠르게 소진되고, 정작 중요한 파일을 놓친다

→ **교훈**: 규칙이 불필요한 탐색이나 작업을 유발하면, 비용만 올리고 성공률은 떨어진다.

→ 참고: [관련 논문 — 잘 작성된 규칙은 런타임 28.64% 감소, 출력 토큰 20% 감소](https://arxiv.org/html/2601.20404v1)

### 실무에서의 의미

[Claude Code 공식 문서](https://code.claude.com/docs/ko/memory)와 ETH Zurich 연구를 종합하면, 가장 효과적인 규칙의 공통점:

**포함해야 할 것** ✅ — 에이전트가 추측할 수 없는 정보:
- 빌드 명령: `just build` 또는 `pnpm run build`
- 테스트 방법: `pytest tests/ -v`
- 금지 사항: "절대 `rm -rf`를 실행하지 마라"
- 패키지 매니저: "`npm` 대신 `uv`를 사용한다"

**포함하지 말아야 할 것** ❌ — 에이전트가 스스로 파악할 수 있는 정보:
- 코드베이스 디렉토리 구조 개요 (에이전트가 Glob으로 탐색)
- README와 중복되는 내용
- "깨끗한 코드를 작성하세요" 같은 자명한 관행
- 언어 버전, 프레임워크 버전 (코드를 읽으면 알 수 있음)

### 다른 접근법과의 비교

| 구분 | 규칙 없음 | LLM 자동 생성 (/init) | 인간 직접 작성 (최소한) |
|------|----------|---------------------|---------------------|
| **성능** | 기준선 | **-0.5~-2%** | **+4%** |
| **비용** | 기준선 | **+20%** | **+19%** |
| **노력** | 없음 | 1분 (명령 한 줄) | 30분~1시간 |
| **위험** | 추측 의존 | 불필요한 탐색 유발 | 낮음 |
| **권장** | 비추천 | **비추천** | **권장** |

→ `/init`으로 자동 생성 후 **반드시 검토·수정**하라.
→ 자동 생성된 규칙을 그대로 쓰면 오히려 성능이 떨어진다.

### 주의사항과 흔한 오해

> **오해 1**: "AGENTS.md를 상세하게 쓸수록 좋다"

→ **실제**: ETH Zurich 연구의 핵심 발견은 **"최소한으로 작성하라"**이다. 규칙이 길수록 에이전트가 "불필요한 탐색"을 하게 되어 비용만 증가한다. 200줄 이하를 권장한다. 에이전트가 추측할 수 없는 정보**만** 넣어라.

> **오해 2**: "/init으로 자동 생성하면 충분하다"

→ **실제**: `/init`(또는 LLM 자동 생성)은 8개 실험 중 5개에서 성능을 떨어뜨렸다. 자동 생성은 **출발점**으로만 사용하고, 반드시 인간이 검토·수정해야 한다. 핵심은 "에이전트가 모르는 것"만 남기고 나머지를 삭제하는 것이다.

이 주제에 대해 커뮤니티에서 의견이 나뉜다:
- **상세파**: "규칙이 많을수록 에이전트가 정확히 동작한다. 우리 팀은 500줄 CLAUDE.md를 쓴다"
- **간결파**: "ETH Zurich 논문이 증명했다. 규칙은 최소한일수록 좋다. 50줄이면 충분하다"

→ **실전 권장안**: 도구 지시(빌드, 테스트, 패키지 매니저)와 금지 사항에 집중하라. 도구 지시 한 줄이 에이전트의 도구 사용률을 **1.6배** 높인다 (ETH Zurich 논문).

### 코드 예제

효과적인 AGENTS.md의 실제 예시:

**❌ 나쁜 규칙 (자동 생성, 과도함):**
```markdown
# AGENTS.md

이 프로젝트는 Python 3.12를 사용합니다.
디렉토리 구조는 다음과 같습니다:
- src/: 소스 코드
- tests/: 테스트 코드
- docs/: 문서
코드를 작성할 때는 깨끗하고 읽기 좋은 코드를 작성하세요.
변수명은 명확하게 지어주세요.
함수는 하나의 역할만 해야 합니다.
```

→ 문제: 에이전트가 코드를 읽으면 알 수 있는 정보와 자명한 관행뿐이다.

**✅ 좋은 규칙 (인간 작성, 최소한):**
```markdown
# AGENTS.md

## 빌드 & 테스트
- 패키지 관리: `uv` (npm, pip 사용 금지)
- 테스트: `uv run pytest tests/ -v`
- 린트: `uv run ruff check .`

## 코드 컨벤션
- 모든 함수에 타입 힌트와 한국어 docstring 필수
- 에러 메시지는 한국어로 작성

## 금지 사항
- `rm -rf` 실행 금지
- main 브랜치에 직접 커밋 금지
- `.env` 파일을 Git에 커밋 금지
```

→ 핵심: 에이전트가 **추측할 수 없는** 도구 지시와 금지 사항만 포함한다.

이 규칙 파일의 각 항목이 다음 섹션에서 배울 **커스텀 에이전트**와 결합되면, 에이전트의 동작을 더 정밀하게 제어할 수 있다.

→ 더 깊이 알고 싶다면 **"AGENTS.md best practices ETH Zurich"**를 검색해 보십시오.

## 2. 커스텀 에이전트 vs 서브에이전트

### 왜 이것이 중요한가

앞서 프로젝트 규칙으로 에이전트의 **행동 기준**을 정했다.
그런데 규칙만으로는 부족한 상황이 있다.

"코드 리뷰를 요청했는데, 에이전트가 리뷰 대신 코드를 수정해버렸다."
규칙에 "리뷰만 하세요"라고 써도, 에이전트는 "더 좋은 코드"를 보면 수정하고 싶어 한다.

이럴 때 필요한 것이 **커스텀 에이전트**다.
도구 자체를 제한하여, 리뷰 에이전트는 **물리적으로 코드를 수정할 수 없게** 만든다.

한편, 대규모 코드베이스를 탐색하면 컨텍스트가 빠르게 소진된다.
2교시에서 배운 컨텍스트 로트 문제다.
이럴 때 필요한 것이 **서브에이전트**다.
탐색 작업을 **별도 컨텍스트**에서 수행하고, 결과만 메인에 반환한다.

이 두 개념은 자주 혼동된다.
이번에 명확히 구분하자.

→ 참고: [Claude Code 서브에이전트 문서](https://code.claude.com/docs/ko/sub-agents)

---

### 핵심 원리

**비유로 이해하기**: 회사 조직을 떠올려 보십시오.

→ **커스텀 에이전트** = 팀의 **직책**. "리뷰어"라는 직책은 코드를 수정하지 않고 리뷰만 한다. **상사(사용자)가** "리뷰어, 이것 좀 봐줘"라고 지시한다.

→ **서브에이전트** = 회의 중 **"잠깐 나가서 확인해올 동료"**. 메인 회의(메인 컨텍스트)를 방해하지 않고, 별도 공간에서 확인하고 결과만 가져온다. **회의 참석자(에이전트)가 스스로** "잠깐 확인하고 올게"라고 결정한다.

**커스텀 에이전트 (Custom Agent)** — **"역할을 정의한다"**

역할별로 특화된 에이전트를 정의하여 도구 접근, 모델, 프롬프트를 **분리**한다.
**사용자가 직접 호출**하여 특정 역할을 수행하게 한다.

| 패턴 | 역할 | 도구 제한 | 호출 방법 |
|------|------|----------|----------|
| **Build** | 코드 작성·수정 | 모든 도구 활성화 | 기본 에이전트 |
| **Plan** | 분석·설계 | 읽기 전용 (edit/bash 비활성화) | `Tab` 또는 `@plan` |
| **Reviewer** | 코드 리뷰 | 읽기 전용 | `@reviewer` |
| **Explorer** | 코드베이스 탐색 | 읽기 + 검색만 | `@explorer` |

파일 위치:
- OpenCode: `.opencode/agents/*.md`
- Claude Code: `.claude/agents/*.md`

**서브에이전트 (Sub-agent)** — **"작업을 위임한다"**

에이전트가 **자율적으로** 독립적인 하위 작업을 별도 컨텍스트에서 실행한다.
**사용자가 아니라 에이전트**가 필요할 때 호출한다.

```
메인 에이전트: "10개 파일을 분석해야 하는데, 내 컨텍스트가 부족하다"
    ↓
서브에이전트 A: 파일 1~5 분석 (독립 컨텍스트)
서브에이전트 B: 파일 6~10 분석 (독립 컨텍스트)
    ↓
메인 에이전트: 결과만 받아서 종합
```

**비유의 한계**: 직책은 영구적이지만, 커스텀 에이전트는 세션 중 언제든 전환할 수 있다.
서브에이전트는 작업 완료 후 "자리에 돌아오는" 것이 아니라 **종료**된다.

---

### 핵심 비교 테이블

| 특성 | 커스텀 에이전트 | 서브에이전트 |
|------|---------------|------------|
| **호출 주체** | **사용자** (`@reviewer`) | **에이전트** (자동으로) |
| **목적** | 역할 강제 (리뷰어는 수정 불가) | 컨텍스트 분리 (메인 오염 방지) |
| **컨텍스트** | 메인과 공유 가능 | **독립적** (별도 컨텍스트 윈도우) |
| **수명** | 사용자가 전환할 때까지 | 작업 완료 시 **즉시 종료** |
| **도구 제한** | 가능 (핵심 기능) | 보통 제한 없음 |
| **병렬 실행** | 불가 (한 번에 하나) | **가능** (여러 작업 동시) |
| **비유** | 팀의 **직책** | 잠깐 나가서 **확인해올 동료** |

---

### 실제 사례: @explore — 하이브리드 패턴

`@explore`는 커스텀 에이전트와 서브에이전트의 **경계에 있는 흥미로운 사례**다.

```
사용자 → @explore "이 코드베이스를 분석해줘"     ← 사용자가 호출 (커스텀 에이전트처럼)
         ↓
[Build 에이전트] → task 도구 호출                 ← 에이전트가 서브에이전트를 생성
                    ↓
                [Explore 서브에이전트]             ← 독립 컨텍스트에서 실행
                    → Glob ×5, Read ×7, Grep ×2
                    → 탐색 리포트 생성 (50초)
                    → 종료 (결과만 메인에 반환)     ← 서브에이전트 특성
         ↓
[Build 에이전트] → 결과를 요약하여 사용자에게 전달
```

**왜 하이브리드인가?**
- **사용자가 `@explore`를 호출**한다는 점에서 커스텀 에이전트와 유사하다
- 하지만 내부적으로 Build 에이전트가 **`task` 도구**를 통해 Explore 서브에이전트를 spawn한다
- Explore 서브에이전트는 **독립 컨텍스트**에서 실행되고, **read/grep/glob만 사용 가능**하다
- 실제로 서브에이전트는 "Task 도구를 호출할 수 없다"고 출력에 명시했다 — 도구 제한이 작동하는 증거

> **교훈**: 커스텀 에이전트와 서브에이전트는 칼로 자르듯 구분되지 않을 수 있다.
> 핵심은 **독립 컨텍스트에서 제한된 도구로 실행되는가**이다.

---

### 커스텀 에이전트 흐름 vs 서브에이전트 흐름

**커스텀 에이전트 흐름:**

```
사용자 → @reviewer "이 코드 리뷰해줘"
         ↓
[Reviewer 에이전트] (read/grep만 가능, edit/bash 비활성화)
         ↓
"코드 리뷰 결과: 3가지 개선 사항..."
         ↓
사용자 → @build (기본 에이전트로 전환)
         ↓
[Build 에이전트] (모든 도구 활성화)
         ↓
"리뷰 피드백을 반영하여 수정 완료"
```

**서브에이전트 흐름:**

```
사용자 → "이 프로젝트의 아키텍처를 분석하고 리팩토링해줘"
         ↓
[메인 에이전트] → Task: "src/ 디렉토리 구조 분석해" (서브에이전트에 위임)
                    ↓
                [서브에이전트] (독립 컨텍스트)
                    → glob, read, grep 실행
                    → "분석 결과: 3개 모듈, 12개 파일..."
                    → 종료 (결과만 메인에 반환)
         ↓
[메인 에이전트] → 분석 결과를 바탕으로 리팩토링 실행
```

---

### 실무에서의 의미

**커스텀 에이전트 실전 사례:**

한 팀에서 코드 리뷰 에이전트를 만들었다.
`@reviewer`를 호출하면 코드를 읽고 피드백만 제공한다.
Edit, Write, Bash 도구가 비활성화되어 있으므로, 코드를 수정할 수 없다.

→ 리뷰와 구현을 **물리적으로** 분리하여 실수를 방지한다.

**서브에이전트 실전 사례:**

대규모 리팩토링 시 에이전트가 20개 파일을 분석해야 한다.
모두 메인 컨텍스트에서 읽으면 컨텍스트가 빠르게 소진된다.
서브에이전트에게 "이 5개 파일을 분석하고 요약해줘"를 위임한다.
서브에이전트는 독립 컨텍스트에서 분석하고, **요약만** 메인에 반환한다.

→ 메인 컨텍스트를 보호하면서 대규모 탐색이 가능하다.

**세션 추적으로 내부 동작 분석하기:**

서브에이전트의 실제 동작을 확인하고 싶다면, 세션을 내보내서 분석할 수 있다.
```bash
opencode session list                    # 세션 ID 확인
opencode export <session-id>             # JSON으로 내보내기
```
내보낸 JSON에서 `task` 도구 호출과 자식 세션을 찾으면, 서브에이전트가 어떤 도구를 몇 번 호출했는지 확인할 수 있다.

→ 자세한 방법은 [부록: 세션 추적 가이드](appendix-session-tracking.md)를 참고하라.

---

### 언제 무엇을 쓰나?

| 상황 | 선택 | 이유 |
|------|------|------|
| 코드 리뷰 시 수정을 방지하고 싶다 | **커스텀 에이전트** | 도구 제한으로 역할 강제 |
| 탐색 결과가 메인 컨텍스트를 오염시키지 않게 | **서브에이전트** | 독립 컨텍스트에서 실행 후 결과만 반환 |
| 프로젝트 표준 코드 스타일을 강제하고 싶다 | **커스텀 에이전트 + 규칙** | 도구 제한 + AGENTS.md 결합 |
| 10개 파일을 병렬로 분석하고 싶다 | **서브에이전트** | 병렬 위임으로 속도 향상 |
| 주니어 개발자에게 안전한 환경을 제공하고 싶다 | **커스텀 에이전트** | 읽기 전용 에이전트로 위험 방지 |

---

### 주의사항과 흔한 오해

> **오해 1**: "커스텀 에이전트와 서브에이전트는 같은 것 아닌가?"

→ **실제**: 완전히 다르다. 핵심 차이는 **누가 호출하는가**이다. 커스텀 에이전트는 사용자가 `@reviewer`처럼 명시적으로 호출한다. 서브에이전트는 에이전트가 자율적으로 호출한다. 커스텀 에이전트는 "역할"을 정의하고, 서브에이전트는 "작업"을 위임한다.

> **오해 2**: "서브에이전트를 쓰면 무조건 좋다"

→ **실제**: 서브에이전트도 비용이 발생한다. 각 서브에이전트마다 별도 API 호출이 필요하고, 시스템 프롬프트와 도구 정의가 중복 로드된다. 단순한 작업은 메인 에이전트가 직접 하는 것이 효율적이다. 서브에이전트는 **컨텍스트 보호가 필요할 때**나 **병렬 처리가 필요할 때** 사용한다.

### 코드 예제

커스텀 리뷰 에이전트 정의 파일:

**`.opencode/agents/reviewer.md`** (OpenCode 기준):
```markdown
---
name: reviewer
description: 코드 리뷰 전문 에이전트. 코드를 수정하지 않고 피드백만 제공.
model: default
allowed-tools: read, grep, glob
---

# Code Reviewer

코드를 수정하지 마세요. 읽기만 가능합니다.

## 리뷰 기준
- 로직 결함 및 잠재적 버그
- 타입 힌트 누락 여부
- 테스트 커버리지 부족
- AGENTS.md 규칙 위반

## 출력 형식
각 이슈를 다음 형식으로 보고:
- **파일**: 파일 경로
- **줄**: 줄 번호
- **심각도**: 높음/중간/낮음
- **설명**: 이슈 설명과 개선 제안
```

사용법:
```bash
# OpenCode에서
@reviewer todo_app.py를 리뷰해줘

# Claude Code에서
@reviewer todo_app.py를 리뷰해줘
```

→ `edit`, `write`, `bash` 도구가 `allowed-tools`에 없으므로, 이 에이전트는 **코드를 수정할 수 없다**.

이 커스텀 에이전트 패턴은 다음에 배울 **스킬**과 결합하면 더 강력해진다.

→ 더 깊이 알고 싶다면 **"Claude Code custom agents sub-agents"**를 검색해 보십시오.

## 3. 스킬 (Skills) — 필요할 때 로드하는 매뉴얼

### 왜 이것이 중요한가

앞서 프로젝트 규칙(항상 적용)과 커스텀 에이전트(역할 분리)를 배웠다.
여기서 자연스러운 질문이 생긴다.
**"반복되는 작업 절차를 미리 정의해두고 필요할 때만 불러올 수 없을까?"**

예를 들어, "Docker 이미지를 빌드하고 ECR에 푸시하는 절차"는 매번 필요한 게 아니다.
이것을 AGENTS.md에 넣으면 **항상** 컨텍스트를 소비한다.
그렇다고 매번 프롬프트에 절차를 붙여넣기도 번거롭다.

이럴 때 쓰는 것이 **스킬(Skill)**이다.
스킬은 재사용 가능한 절차/지식을 패키징하여, 에이전트가 **필요할 때만** 로드한다.

→ 참고: [Claude Code 스킬](https://code.claude.com/docs/en/skills)
→ 참고: [OpenCode 스킬](https://opencode.ai/docs/skills/)

---

### 핵심 원리

**비유로 이해하기**: 사무실의 규칙과 매뉴얼을 떠올려 보십시오.

→ **규칙(AGENTS.md)** = 사무실 벽에 붙은 **공지사항**. 모든 직원이 항상 본다.
→ **스킬(SKILL.md)** = 서랍에 있는 **업무 매뉴얼**. 특정 업무를 할 때만 꺼내본다.

**실제 동작 — SKILL.md 파일 구조:**

```markdown
---
name: docker-deploy
description: Docker 이미지를 빌드하고 ECR에 푸시하는 절차
---

# Docker Deploy

## 절차
1. Dockerfile 확인
2. `docker build -t app:latest .`
3. ECR 로그인
4. 태그 후 푸시
...
```

- `name`: 스킬 이름 (호출 시 사용)
- `description`: 에이전트가 이 스킬을 언제 로드할지 판단하는 기준

**스킬 설치 경로:**

| 도구 | 경로 |
|------|------|
| Claude Code | `.claude/skills/{name}/SKILL.md` |
| OpenCode | `.opencode/skills/{name}/SKILL.md` |
| Codex | `.agents/skills/{name}/SKILL.md` |

→ **하나의 스킬 = 하나의 작업**. 범위를 좁게 유지한다.

**비유의 한계**: 매뉴얼은 사람이 직접 찾아서 읽어야 한다.
스킬은 에이전트가 **자동으로** 필요한 시점에 로드하거나, 사용자가 `/skill-name`으로 명시적으로 호출할 수 있다.

### 비슷한 개념들의 종합 비교

| 개념 | 범위 | 언제 적용 | 컨텍스트 소비 | 비유 |
|------|------|----------|-------------|------|
| **프로젝트 규칙** (AGENTS.md) | 항상 로드 | 프로젝트 컨벤션, 금지사항 | 매 세션 소비 | 사무실 공지사항 |
| **스킬** (SKILL.md) | 필요할 때 로드 | 반복되는 워크플로우 | 호출 시만 소비 | 서랍의 매뉴얼 |
| **커스텀 에이전트** | 역할별 분리 | 역할 강제 (리뷰어는 수정 불가) | 전환 시 활성 | 팀의 직책 |

| **MCP** | 외부 연동 | 코드 저장소 밖의 데이터/서비스 | 도구 정의 상시 소비 | USB 포트 |
| **플러그인** | 하네스 확장 | 하네스 자체의 동작 확장 | 최소 (훅이므로) | 브라우저 확장 |
| **CLI 도구** | 로컬 실행 | 빌드, 테스트, 린트 등 | 없음 (결과만) | 도구 상자 |

→ **규칙**은 "항상", **스킬**은 "필요할 때", **커스텀 에이전트**는 "역할별", **서브에이전트**는 "작업별"이다.

### 주의사항과 흔한 오해

> **오해 1**: "규칙과 스킬의 차이가 뭔가요? 둘 다 마크다운 파일인데?"

→ **실제**: 핵심 차이는 **로드 시점**이다. 규칙은 세션 시작 시 **자동으로** 로드되어 항상 컨텍스트를 소비한다. 스킬은 필요할 때만 로드되므로 컨텍스트를 절약한다. 항상 필요한 정보(빌드 명령, 코드 스타일)는 규칙에, 가끔 필요한 절차(배포, 마이그레이션)는 스킬에 넣는다.

> **오해 2**: "스킬은 플러그인과 같은 거 아닌가?"

→ **실제**: 스킬은 **마크다운 문서**다. 에이전트에게 "이 절차를 따라라"라고 알려주는 것이다. 플러그인은 에이전트 런타임 자체의 **동작을 수정**한다 (라이프사이클 훅, 자동 트리거 등). 스킬은 지식을 제공하고, 플러그인은 행동을 수정한다.

### 코드 예제

간단한 스킬 파일 작성 예시:

**`.opencode/skills/add-feature/SKILL.md`**:
```markdown
---
name: add-feature
description: 새 기능을 추가할 때 따라야 할 표준 절차
---

# Add Feature Skill

## 절차
1. 기존 코드를 먼저 읽고 구조를 파악한다
2. 관련 테스트 파일을 확인한다
3. 기능 코드를 작성한다
4. 테스트를 작성한다
5. `uv run pytest tests/ -v`로 테스트를 실행한다
6. 모든 테스트가 통과하면 완료

## 코드 컨벤션
- 모든 함수에 타입 힌트 필수
- docstring은 한국어로 작성
- 에러는 커스텀 예외 클래스를 사용
```

→ 이 스킬은 에이전트가 새 기능을 추가할 때 `/add-feature`로 호출하거나, 에이전트가 자동으로 로드한다.

### 스킬의 품질 관리: skill-creator

> 스킬을 만드는 것은 쉽다. **잘 작동하는 스킬을 유지하는 것**이 어렵다.

스킬을 SKILL.md로 작성하면 끝일까? 현실은 다르다.
모델이 업데이트되면 기존 스킬이 깨질 수 있고, 스킬의 `description`이 부정확하면 엉뚱한 상황에서 트리거되거나, 필요한 상황에서 트리거되지 않는다.

이 문제를 해결하기 위해 2026년 3월, Anthropic이 **skill-creator**를 개선하여 공개했다.

→ 참고: [Improving skill-creator: Test, measure, and refine Agent Skills (Claude Blog, 2026.03)](https://claude.com/blog/improving-skill-creator-test-measure-and-refine-agent-skills)
→ 참고: [skill-creator SKILL.md (GitHub)](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)

---

### 두 종류의 스킬

skill-creator를 이해하려면 먼저 스킬이 **왜 필요한지**를 구분해야 한다.

| 유형 | 설명 | 예시 | 수명 |
|------|------|------|------|
| **Capability Uplift** | 기본 모델이 할 수 없거나 불안정한 작업을 가능하게 | PDF 비채움 양식에 정확한 위치에 텍스트 배치 | 모델 발전 시 **불필요해질 수 있음** |
| **Encoded Preference** | 모델이 이미 할 수 있는 작업을 팀의 워크플로우대로 수행 | NDA 리뷰를 팀 기준에 맞게, 주간 업데이트를 정해진 형식으로 | **지속적으로 유효** |

→ **Capability Uplift** 스킬은 모델이 진화하면 쓸모없어질 수 있다. 따라서 주기적 검증이 필요하다.
→ **Encoded Preference** 스킬은 워크플로우가 바뀌지 않는 한 유효하지만, 절차가 정확히 수행되는지 검증해야 한다.

---

### Eval — 스킬을 테스트하는 자동화된 방법

> Eval은 **"이 스킬이 제대로 작동하는가?"**를 자동으로 검증하는 테스트 케이스다.

소프트웨어에 단위 테스트가 있듯, 스킬에도 테스트가 필요하다. skill-creator는 `evals/evals.json`에 테스트 케이스를 정의한다:

```json
{
  "skill_name": "pdf-form-filler",
  "evals": [
    {
      "id": 1,
      "prompt": "이 PDF 양식의 이름 필드에 '홍길동'을 채워줘",
      "expected_output": "이름 필드의 정확한 좌표에 텍스트가 배치됨",
      "files": ["sample-form.pdf"]
    }
  ]
}
```

**Eval의 두 가지 핵심 용도:**

1. **회귀 감지 (Regression Detection)**: 모델 업데이트 후 기존 스킬이 여전히 작동하는가?
2. **능력 진부화 감지 (Capability Obsolescence)**: 기본 모델이 이미 이 작업을 잘 하게 되어 스킬이 불필요해졌는가?

→ **실제 사례**: PDF 스킬이 비채움 양식의 텍스트 위치 지정에 실패했다. Eval이 이 실패를 정확히 포착했고, "추출된 텍스트 좌표에 앵커링"하는 방식으로 수정하여 해결했다.

![skill-creator PDF evals](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69a237b02128b691d9e8b2af_skillscreator-PDFevals-1920x840-v1.png)

### 벤치마크 모드 — 정량적으로 측정하기

> "측정할 수 없으면 개선할 수 없다."

벤치마크 모드는 **eval 통과율**, **소요 시간**, **토큰 사용량**을 정량적으로 추적한다.

![skill-creator benchmark mode](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69a237f15fbc61e1ccd00a0a_skillscreator-benchmarkmode-1920x1080-v1.png)

**벤치마크 워크플로우:**

```
iteration-1/
├── eval-0-pdf-fill/
│   ├── with_skill/outputs/       # 스킬 적용 결과
│   ├── without_skill/outputs/    # 기본 모델 결과 (베이스라인)
│   ├── eval_metadata.json        # 테스트 정의
│   ├── grading.json              # 자동 채점 결과
│   └── timing.json               # 토큰, 시간 측정
├── benchmark.json                # 종합 벤치마크
└── feedback.json                 # 사용자 피드백
```

이 구조를 통해 **스킬 버전 간 정량 비교**가 가능하다:
- iteration-1: 기존 스킬 vs 베이스라인
- iteration-2: 개선된 스킬 vs 기존 스킬
- iteration-3: 추가 개선 → 수렴할 때까지 반복

---

### 멀티에이전트 평가와 A/B 테스트

순차적으로 eval을 실행하면 느리고, 이전 eval의 컨텍스트가 다음 eval을 오염시킨다.
skill-creator는 **독립 에이전트를 병렬로 생성**하여 이 문제를 해결한다:

- 각 에이전트가 독립 컨텍스트에서 eval을 실행
- 에이전트별 토큰/시간 메트릭이 격리됨
- **Comparator 에이전트**로 A/B 테스트 지원 — 스킬 적용 vs 미적용, 또는 구버전 vs 신버전 비교
- **블라인드 채점**: 채점 에이전트는 어떤 버전을 평가하는지 모른다 → 편향 방지

![skill-creator A/B testing](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69a74e0afa8435f070120ed9_skillscreator-AB-testing-1920x1080-v1.png)

→ 2교시에서 배운 **서브에이전트의 컨텍스트 분리**가 여기서도 핵심 원리로 작동한다.

### 스킬 Description 최적화

> 스킬의 `description`이 부정확하면, 에이전트가 **잘못된 상황에서 스킬을 로드**하거나, **필요한 상황에서 로드하지 않는다**.

skill-creator는 트리거 정확도를 체계적으로 개선한다:

**Step 1**: should-trigger / should-not-trigger 쿼리 20개 생성
```json
[
  {"query": "이 엑셀 파일에서 수익률 컬럼을 추가해줘", "should_trigger": true},
  {"query": "이 코드를 리팩토링해줘", "should_trigger": false}
]
```

**Step 2**: 최적화 루프 실행
```bash
python -m scripts.run_loop \
  --eval-set trigger-eval.json \
  --skill-path ./my-skill \
  --max-iterations 5
```

**Step 3**: 결과 적용 — 최적화된 description으로 교체

**결과**: 공개된 6개 문서 생성 스킬 중 **5개에서 트리거 정확도가 향상**되었다.

![skill-creator description optimization results](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/69a74e1f72940942cb534904_skillscreator-skill-description-optimization-results.png)

→ Description은 스킬의 "이력서"와 같다. 너무 넓으면 엉뚱한 곳에서 호출되고(false positive), 너무 좁으면 필요한 곳에서 놓친다(false negative).

### skill-creator의 전체 워크플로우

```
Create (작성)
  ↓ 인터뷰 → 리서치 → SKILL.md 작성
Test (테스트)
  ↓ evals.json에 테스트 케이스 정의
Measure (측정)
  ↓ 벤치마크 모드로 통과율/시간/토큰 측정
  ↓ A/B 비교 (스킬 적용 vs 미적용)
Refine (개선)
  ↓ 피드백 기반 스킬 수정
  ↓ Description 최적화
  ↓ 다시 Measure → 수렴할 때까지 반복
```

→ **소프트웨어 개발의 TDD(Test-Driven Development)와 같은 패턴**이다.
→ 스킬도 코드와 마찬가지로 "작성 → 테스트 → 측정 → 개선"의 반복적 개발 사이클을 거쳐야 한다.

### 미래 전망: 스킬에서 스펙으로

> 모델이 발전하면, "어떻게(how)" 대신 **"무엇을(what)"**만 기술하면 될 수 있다.

현재 SKILL.md는 **구현 계획(implementation plan)**에 가깝다 — "1단계: 이것을 하고, 2단계: 저것을 하라."
하지만 모델이 충분히 발전하면, 자연어로 **"무엇을 원하는지"**만 기술하면 모델이 "어떻게"를 알아서 결정할 수 있다.

이 시점에서 스킬은 **사양서(specification)**에 가까워진다.
그때까지는 구체적인 절차를 포함한 SKILL.md가 여전히 효과적이다.

→ 이것이 **Capability Uplift** 스킬의 수명이 제한적인 이유이기도 하다.

스킬과 다음 섹션의 **확장 기능**을 조합하면, 에이전트의 활용 범위가 크게 넓어진다.

→ 더 깊이 알고 싶다면 **"Claude Code skills vs plugins"**를 검색해 보십시오.

## 4. 코드 에이전트 확장 기능 (Features Overview)

### 왜 이것이 중요한가

지금까지 규칙, 커스텀 에이전트, 스킬을 배웠다.
이것들은 에이전트의 **지식과 역할**을 설정하는 것이다.

하지만 코드 에이전트에는 더 많은 기능이 있다.
**멀티에이전트**, **훅**, **메모리**, **Plan 모드** 등이다.

이 기능들을 알면 에이전트의 **활용 범위**가 크게 넓어진다.
"이것도 에이전트로 할 수 있었어?"라는 순간이 줄어든다.

→ 참고: [Claude Code 기능 개요](https://code.claude.com/docs/ko/features-overview)

### 핵심 원리

코드 에이전트의 주요 확장 기능을 한눈에 정리한다:

| 기능 | 설명 | 언제 사용? |
|------|------|-----------|
| **멀티에이전트** | 여러 에이전트를 병렬로 실행하여 작업 분담 | 대규모 리팩토링, 리뷰 + 구현 병렬 |
| **Git 워크트리** | 독립된 브랜치에서 병렬 작업 | 서로 다른 접근법 동시 시도 |
| **훅 (Hooks)** | 도구 실행 전후에 커스텀 로직 삽입 | 자동 린팅, 보안 검사, 알림 |
| **메모리** | 프로젝트/사용자 정보를 세션 간 유지 | 사용자 선호도, 프로젝트 컨텍스트 축적 |
| **비대화형 모드** | 스크립트에서 에이전트 실행 | CI/CD, 자동화 파이프라인 |
| **Plan 모드** | 읽기 전용으로 분석·계획만 수행 | 코드 작성 전 전략 수립 |

### 멀티에이전트 — 여러 에이전트를 동시에

하나의 에이전트가 모든 작업을 하면 시간이 오래 걸린다.
멀티에이전트는 여러 에이전트를 **병렬로** 실행하여 작업을 분담한다.

**사용 사례**:
- 에이전트 A: 프론트엔드 리팩토링
- 에이전트 B: 백엔드 API 수정
- 에이전트 C: 테스트 코드 업데이트

→ 각 에이전트가 독립적으로 작업하고, 결과를 합친다.

---

### 훅 (Hooks) — 도구 실행 전후에 개입

에이전트가 도구를 사용할 때, **전후로 커스텀 로직**을 실행할 수 있다.

```
에이전트가 Edit 도구 호출
    ↓ [PreToolUse 훅] 린터 실행 → 규칙 위반 감지 → 경고
    ↓
Edit 실행
    ↓ [PostToolUse 훅] 자동으로 포맷팅 적용
```

**실전 활용 예시**:
- `PreToolUse: Bash` → 위험한 명령어(`rm -rf`) 차단
- `PostToolUse: Edit` → 저장 후 자동 린팅
- `PostToolUse: Bash` → 테스트 실패 시 Slack 알림

→ 참고: [Claude Code 훅](https://code.claude.com/docs/en/hooks)

---

### 메모리 — 세션을 넘어서는 기억

기본적으로 에이전트는 세션이 끝나면 모든 것을 잊는다.
메모리 기능을 사용하면 **세션 간 정보를 유지**할 수 있다.

| 종류 | 저장 위치 | 용도 |
|------|----------|------|
| **프로젝트 메모리** | 프로젝트 디렉토리 | 프로젝트 고유 결정사항, 아키텍처 노트 |
| **사용자 메모리** | 전역 설정 | 사용자 선호도, 코딩 스타일 |

→ "이전 세션에서 TypeScript strict 모드를 쓰기로 결정했다" — 이런 결정을 기억한다.

### 실무에서의 의미

이 확장 기능들을 조합하면 강력한 워크플로우가 만들어진다:

**워크플로우 예시 — Writer-Reviewer 패턴**:
1. **커스텀 에이전트 `@build`**: 코드를 작성 (모든 도구 활성)
2. **커스텀 에이전트 `@reviewer`**: 코드를 리뷰 (읽기 전용)
3. **훅**: 리뷰 결과를 자동으로 기록
4. **메모리**: 리뷰 패턴을 학습하여 다음 세션에 반영

→ 이것이 2교시에서 배운 **Writer-Reviewer 에이전틱 패턴**의 구현이다.

### 주의사항과 흔한 오해

> **오해 1**: "확장 기능을 다 쓸수록 좋다"

→ **실제**: 1교시에서 배웠듯이, 도구가 많으면 컨텍스트를 소비한다. 훅, MCP, 멀티에이전트를 무분별하게 추가하면 오버헤드가 커진다. **현재 필요한 기능만** 활성화하라.

> **오해 2**: "메모리가 있으면 규칙이 필요 없다"

→ **실제**: 메모리는 **개인적 경험**을 축적하고, 규칙은 **팀 표준**을 정의한다. 둘은 보완적이다. "이 프로젝트는 uv를 사용한다"는 규칙이고, "지난번에 이 모듈은 A 패턴이 더 잘 맞았다"는 메모리다.

→ 4교시에서 이 기능들의 실전 활용을 더 자세히 다룬다.

## Q&A

**Q: AGENTS.md를 상세하게 쓸까, 간결하게 쓸까? (← 논쟁적 포인트)**
A: 커뮤니티에서 활발히 논쟁 중인 주제다. ETH Zurich 논문은 "간결하게"를 강력히 지지한다. LLM이 자동 생성한 장문 규칙은 성능을 떨어뜨리고 비용만 올린다. 반면 GitHub Blog의 2,500개 리포 분석은 "구체적인 도구 지시"가 효과적이라고 한다. 결론은 **"짧지만 구체적"**이다. 200줄 이하로, 에이전트가 추측할 수 없는 도구 지시와 금지 사항에 집중하라. "모든 것을 다 적는 것"이 아니라 "에이전트가 모르는 것만 적는 것"이 핵심이다. 더 깊이 알고 싶다면 **"arxiv 2602.11988"**을 검색해 보십시오.

**Q: /init으로 자동 생성한 AGENTS.md를 그대로 써도 되나요? (← 흔한 오해)**
A: 안 된다. ETH Zurich 연구에서 LLM 자동 생성 규칙은 8개 실험 중 5개에서 성능을 오히려 떨어뜨렸다. `/init`은 출발점으로만 사용하고, 반드시 인간이 검토·수정해야 한다. 핵심은 에이전트가 코드를 읽어서 알 수 있는 정보(디렉토리 구조, 언어 버전)를 삭제하고, 알 수 없는 정보(빌드 명령, 금지 사항)만 남기는 것이다. 더 깊이 알고 싶다면 **"AGENTS.md auto-generation pitfalls"**를 검색해 보십시오.

**Q: 커스텀 에이전트와 서브에이전트를 동시에 사용할 수 있나요? (← 개발자 배경)**
A: 가능하다. 예를 들어 커스텀 에이전트 `@build`가 코드를 작성하는 중에, 내부적으로 서브에이전트를 호출하여 다른 파일을 탐색할 수 있다. 커스텀 에이전트는 "역할"을 정의하고, 서브에이전트는 그 역할 안에서 "작업을 위임"한다. 커스텀 에이전트의 도구 제한이 서브에이전트에게도 적용될 수 있으므로, 도구 설정을 주의 깊게 확인해야 한다. 더 깊이 알고 싶다면 **"Claude Code agent composition"**을 검색해 보십시오.

**Q: "순응의 역설"이라면, 결국 규칙을 안 쓰는 게 가장 좋은 건 아닌가요?**
A: 아니다. 규칙을 **안 쓰는 것**과 **잘 쓰는 것**은 다르다. ETH Zurich 연구에서 인간 작성 규칙은 +4% 성능 향상을 보였다. 규칙이 없으면 에이전트가 프로젝트 고유의 빌드 명령, 테스트 방법을 알 수 없어 추측에 의존한다. "순응의 역설"이 경고하는 것은 "규칙을 쓰지 마라"가 아니라 **"나쁜 규칙을 쓰지 마라"**이다. 도구 지시와 금지 사항에 집중하면 안전하다. 더 깊이 알고 싶다면 **"compliance paradox AI agents"**를 검색해 보십시오.

**Q: AGENTS.md를 Git에 커밋해야 하나요? (← 개발자 배경)**
A: 반드시 커밋해야 한다. AGENTS.md는 팀 전체가 공유하는 **팀 컨벤션 문서**다. Git에 커밋하면 모든 팀원의 에이전트가 동일한 규칙을 따른다. 개인 설정은 전역 파일(`~/.config/opencode/AGENTS.md`)에 넣고, 프로젝트 규칙은 프로젝트 루트에 커밋한다. 더 깊이 알고 싶다면 **"AGENTS.md team sharing"**을 검색해 보십시오.

**Q: 스킬을 너무 많이 설치하면 문제가 되나요?**
A: 스킬 자체는 **호출 시에만** 로드되므로, 많이 설치해도 평소에는 컨텍스트를 소비하지 않는다. 다만 스킬의 `description`이 에이전트의 스킬 선택에 영향을 줄 수 있다. 비슷한 이름의 스킬이 여러 개 있으면 에이전트가 잘못된 스킬을 선택할 수 있으므로, 사용하지 않는 스킬은 정리하는 습관이 필요하다. 더 깊이 알고 싶다면 **"agent skills management"**를 검색해 보십시오.

**Q: 큰 프로젝트에서 AGENTS.md를 100줄 이하로 유지할 수 있나요? (← 개발자 배경)**
A: 가능하다. **계층적 구조**를 활용한다. 프로젝트 루트 AGENTS.md에는 전체 컨벤션만 넣고, 각 디렉토리의 AGENTS.md에 해당 모듈의 규칙을 분리한다. `src/api/AGENTS.md`에 API 규칙, `src/db/AGENTS.md`에 DB 규칙을 넣으면, 각 파일은 간결하면서 필요한 규칙은 모두 커버할 수 있다. 더 깊이 알고 싶다면 **"hierarchical AGENTS.md"**를 검색해 보십시오.

<summary>✅ 정답</summary>
	**순응의 역설**: 에이전트는 규칙을 충실히 따르지만, **나쁜 규칙도 충실히 따른다**는 현상이다. 불필요한 탐색이나 과도한 검증을 유발하는 규칙을 넣으면, 에이전트가 이를 그대로 수행하여 비용만 증가하고 성공률은 떨어진다.
	**피하기 위한 원칙:**
	1. **최소한으로 작성**: 에이전트가 추측할 수 없는 정보(빌드 명령, 금지 사항)만 포함한다. 코드를 읽으면 알 수 있는 정보는 제외한다.
	2. **도구 지시에 집중**: "어떤 도구를 어떻게 사용하라"는 지시가 가장 효과적이다. ETH Zurich 연구에서 도구 지시가 에이전트의 도구 사용률을 1.6배 높였다.
</details>

## 핵심 정리

- **프로젝트 규칙(AGENTS.md)**: 인간이 직접, 최소한으로 작성한다. LLM 자동 생성은 비추천 (ETH Zurich 연구: 자동 생성 -2% vs 인간 작성 +4%)
- **"순응의 역설"**: 에이전트는 나쁜 규칙도 충실히 따른다. 도구 지시와 금지 사항에 집중하라
- **커스텀 에이전트**: 사용자가 호출하여 **역할을 강제**한다 (도구 제한으로 리뷰어는 수정 불가)
- **서브에이전트**: 에이전트가 자율적으로 호출하여 **작업을 위임**한다 (독립 컨텍스트로 메인 보호)
- **스킬**: 필요할 때만 로드하는 재사용 절차. `npx skills add`로 커뮤니티 스킬 설치
- **확장 기능**: 멀티에이전트, 훅, 메모리, Plan 모드로 에이전트의 활용 범위를 넓힌다
