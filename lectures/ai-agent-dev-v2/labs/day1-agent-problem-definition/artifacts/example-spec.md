## Agent 후보 #1 — 고객 문의 자동 분류 및 답변

**Pain**:
매일 고객 문의 이메일 50-80건을 수동으로 읽고 담당팀에 라우팅한다.
팀당 평균 15분, 하루 3-4시간을 단순 분류 작업에 소비한다.
문의 내용이 유사함에도 매번 사람이 판단해야 한다.

**Task**:
수신 고객 문의를 자동으로 분류하고 표준 답변을 생성하여 담당자에게 전달한다.

**Skills**:
- 문의 내용 분류 (카테고리, 긴급도 판단)
- 유사 문의 과거 케이스 검색
- 표준 답변 초안 생성
- 담당자 이메일 발송

**Tools**:
- `email_reader`: 수신 이메일 읽기 (IMAP API)
- `vector_search`: 유사 문의 과거 케이스 검색 (벡터 DB)
- `llm_classify`: 카테고리 및 긴급도 분류 (LLM + Structured Output)
- `llm_draft`: 표준 답변 초안 생성 (LLM)
- `email_send`: 담당자에게 라우팅 이메일 발송 (SMTP API)

**패턴**: 자동화형
- 트리거: 새 이메일 수신
- 흐름이 비교적 고정적 (분류 → 검색 → 초안 → 발송)
- 예외는 있지만 정상 흐름이 90% 이상

**구조**: Hybrid (RAG + MCP)
**이유**:
유사 과거 케이스 검색은 RAG가 적합하고 (정적 지식 기반),
이메일 수신·발송은 외부 API 호출이 필요하므로 MCP Tool이 필요하다.
두 요소가 모두 필요해 Hybrid 구조를 선택했다.

## Agent 후보 #2 — 코드 리뷰 인사이트 분석

**Pain**:
PR(Pull Request) 리뷰 코멘트가 쌓여 있지만 팀 전체의 패턴을 분석하는 사람이 없다.
어떤 종류의 실수가 반복되는지, 리뷰 속도가 개선되고 있는지 파악하기 어렵다.
분기마다 수동으로 데이터를 취합하는 데 이틀이 걸린다.

**Task**:
GitHub PR 리뷰 데이터를 수집하고 분석하여 팀 코드 품질 인사이트를 생성한다.

**Skills**:
- PR 및 리뷰 코멘트 데이터 수집
- 코멘트 카테고리 분류 (버그, 스타일, 성능, 설계 등)
- 트렌드 분석 및 패턴 파악
- 인사이트 보고서 생성

**Tools**:
- `github_api`: PR 목록 및 리뷰 코멘트 수집
- `llm_classify`: 코멘트 카테고리 분류
- `data_aggregate`: 통계 집계 (Python 코드)
- `llm_summarize`: 인사이트 요약 생성

**패턴**: 분석형
- 입력: 분석 기간 (예: "지난 3개월")
- 데이터 소스: GitHub API (다양한 엔드포인트)
- 결과를 자연어로 설명해야 함

**구조**: MCP (Tool 중심)
**이유**:
모든 데이터가 GitHub API를 통해 실시간으로 수집된다.
정적 문서 기반이 아니므로 RAG가 불필요하다.
Tool (GitHub API + 집계 함수 + LLM)만으로 충분히 구현 가능하다.
