# 문체 및 서술 가이드

가이드 본문 작성 시 참조하는 문체, 구조화, 비유, 금지 패턴 레퍼런스.
SKILL.md의 "개념 설명은 이 파일의 규칙을 따른다"에서 참조된다.

---

## 문장 규칙

1. **한 문장 = 한 가지 정보**. 두 가지 이상의 정보를 한 문장에 담지 않는다.
2. 문장 길이는 **40자 이내**를 기본으로 한다. 60자를 초과하면 반드시 분리한다.
3. 접속사로 이어붙인 복문보다 **단문 나열**을 선호한다.
4. "~하고, ~하며, ~하는" 식의 연쇄 연결은 금지한다. 각각 독립 문장으로 쓴다.

### 어미 규칙: 명사형 종결 우선

**대화체(`~합니다`, `~입니다`)보다 명사형 어미(`~임`, `~함`, `~가능`)를 기본으로 사용한다.**
간결하고 정보 밀도가 높은 교안을 만들기 위함이다. 단, 비유/스토리텔링 등 자연스러운 흐름이 필요한 맥락에서는 대화체도 허용한다.

| 상황 | 어미 스타일 | 예시 |
|------|-----------|------|
| 개념 설명, 비교, 나열 | **명사형** (기본) | "변경 이력을 자동 기록. 언제든 과거로 복구 가능" |
| 비유, 스토리텔링, 질문 | 대화체 (허용) | "게임의 세이브 포인트를 떠올려 보자" |
| 콜아웃, 주의사항 | **명사형** | "커밋은 로컬 저장소에만 저장됨. push는 별도 단계" |
| Q&A 답변 | 대화체 (자연스러움) | "가능하다. `git commit -a`를 사용하면 된다" |

**금지 패턴:**
- `~합니다`, `~됩니다`가 3문장 이상 연속되면 명사형으로 전환
- 표/불릿 안에서 `~합니다` 종결 금지 → 명사형으로 통일

## 이모지 활용 규칙

**맥락에 맞는 이모지를 적극 활용하여 시각적 스캔을 돕는다.** 이모지는 장식이 아니라 정보 구분 도구다.

### 용도별 이모지 가이드

| 용도 | 이모지 | 사용 위치 |
|------|--------|----------|
| 문제/경고 | ⚠️ | 문제 상황 나열 앞 |
| 번호 나열 | 1️⃣ 2️⃣ 3️⃣ | 순서가 있는 포인트 |
| 올바른 패턴 | ✅ | 좋은 예시, 정답 |
| 잘못된 패턴 | ❌ | 나쁜 예시, 금지 |
| 핵심 인사이트 | 💡 | 팁, 핵심 설명 |
| 흐름/결과 | ➡️ | 인과관계, 결론 |
| 코드/기술 | 🔧 ⚙️ | 명령어, 설정 |
| 실습 | 🏋️ | 실습 섹션 도입 |
| 비유 | 🎮 📦 🍽️ | 비유 대상에 맞는 이모지 |
| 증가/감소 | ⬆️ ⬇️ | 통계, 트렌드 |
| 참고 | 📝 📖 | 부연 설명 |

### 이모지 금지 사항

- 한 문장에 이모지 2개 초과 금지 (과도하면 오히려 산만)
- 코드 블록 안에 이모지 삽입 금지
- 의미 없는 장식용 이모지 금지 (모든 이모지는 정보 구분 목적)
- 번호 이모지(1️⃣2️⃣3️⃣)를 독립 블록으로 사용 금지 → 노션 번호 리스트(1. 2. 3.)를 사용

## 노션 구성요소 활용 규칙

**가장 중요한 원칙: 매 문장을 별도 블록으로 만들지 않는다.**
노션에서 마크다운의 빈 줄은 새 블록을 만든다. 관련 있는 문장은 같은 블록 안에서 `<br>`로 줄바꿈하거나, 불릿/번호 리스트로 묶는다.

### 헤딩 역할 정의

| 레벨 | 역할 | 예시 | 사용 빈도 |
|------|------|------|-----------|
| **H1** (`#`) | 세션 제목 (페이지 최상위) | `# Git 기초 — 로컬 워크플로우` | 페이지당 1개 |
| **H2** (`##`) | 대주제 구분 (개념, 퀴즈, 실습) | `## 1. 버전 관리란 무엇인가` | 세션당 3~6개 |
| **H3** (`###`) | 소주제 (6단계 구조의 각 단계) | `### 왜 이것이 중요한가` | 개념당 6개 |

> H2 사이에는 `---` 구분선을 삽입하여 시각적으로 분리한다. H3 사이에는 구분선을 넣지 않는다.

### 불릿 리스트 (`-`)

**관련 항목을 나열할 때 반드시 불릿 리스트를 사용한다.** 각 항목을 별도 텍스트 블록으로 나열하면 노션에서 산만하게 보인다.

```markdown
❌ 나쁜 예 (모든 것이 별도 블록)
⚠️ 문제 3가지

1️⃣ **최신본 구분 불가**

→ app_final.py, app_final2.py 중 어느 것이 맞는지 불분명

2️⃣ **변경 기록 없음**

→ 어제까지 됐는데 오늘 갑자기 안 되는 상황

✅ 좋은 예 (콜아웃 + 불릿 리스트로 그룹화)
<callout icon="⚠️" color="yellow_bg">
	**버전 관리 없이 겪는 문제 3가지**
	- **최신본 구분 불가** — `app_final.py`, `app_final2.py` 중 어느 것이 맞는지 불분명
	- **변경 기록 없음** — "어제까지 됐는데 오늘 갑자기 안 된다" 원인 추적 불가
	- **복구 불가** — 실수로 삭제하거나 잘못 고쳤을 때 방법이 없음
</callout>
```

### 번호 리스트 (`1. 2. 3.`)

**순서가 중요한 단계, 절차는 번호 리스트를 사용한다.** 이모지 번호(1️⃣2️⃣)로 대체하지 않는다.

```markdown
❌ 나쁜 예 (이모지 번호 + 별도 블록)
1️⃣ 파일 수정

2️⃣ 스테이징

3️⃣ 커밋

✅ 좋은 예 (번호 리스트)
1. 파일 수정 — Working Directory에서 코드 변경
2. 스테이징 — `git add`로 변경사항을 Staging Area에 등록
3. 커밋 — `git commit`으로 스냅샷 저장
```

### 콜아웃 (`<callout>`)

**콜아웃은 "그룹화 컨테이너"로 활용한다.** 단일 문장이 아니라, 관련 정보를 하나의 박스에 묶는 용도다.

| 용도 | icon | color | 예시 상황 |
|------|------|-------|----------|
| 학습 목표 | 📖 | blue_bg | 세션/실습 시작부 |
| 핵심 인사이트 | 💡 | blue_bg | 개념의 핵심 결론 |
| 팁/참고 | 💡 | gray_bg | 보충 설명, 선택 정보 |
| 경고/문제 | ⚠️ | yellow_bg | 흔한 실수, 오해 목록 |
| 성공/정리 | ✅ | green_bg | 세션 정리, 올바른 패턴 |
| 과제 조건 | 📋 | yellow_bg | YOU DO 요구사항 |
| macOS 안내 | 🍎 | gray_bg | OS별 차이 |

**콜아웃 안에서는 불릿 리스트, 코드 블록, 줄바꿈을 자유롭게 사용한다:**

```markdown
<callout icon="💡" color="blue_bg">
	**Git은 이 세 가지 문제를 모두 해결하는 도구**
	- ✅ 변경 이력 자동 기록
	- ⏪ 언제든 과거 상태로 복구 가능
	- 👥 여러 명이 동시에 작업해도 충돌 관리
</callout>
```

### 인용블록 (`>`)

**핵심 한 줄 메시지, 기억할 원칙에 사용한다.**

⚠️ **Notion에서 `>` 줄이 여러 개이면 각각 별도 인용블록이 된다.** 표준 마크다운과 다르게 동작함.

```markdown
❌ 나쁜 예 (별도 인용블록 2개 생성)
> 💡 **Git의 커밋 = 게임의 세이브 포인트**
> 중요한 시점에 세이브하고, 실패하면 그 지점으로 돌아가는 것

✅ 좋은 예 (한 물리적 줄에 <br>로 연결 — 하나의 인용블록)
> 💡 **Git의 커밋 = 게임의 세이브 포인트** — 중요한 시점에 세이브하고, 실패하면 그 지점으로 돌아가는 것

✅ 좋은 예 (길면 <br> 사용 — 반드시 한 줄)
> 💡 **Git의 커밋 = 게임의 세이브 포인트**<br>중요한 시점에 세이브하고, 실패하면 그 지점으로 돌아가는 것
```

**규칙:**
- 인용블록은 **한 줄로 완결**하는 것이 원칙
- 길어지면 `<br>`을 사용하되 **한 물리적 줄에 모두 작성** (`<br>` 뒤에 줄바꿈 금지)
- `<br>` 뒤에 `\n>`을 쓰면 별도 인용블록이 됨 — 절대 금지

### 줄바꿈 규칙 (Notion 블록 모델 이해)

**핵심 원칙: Notion에서 마크다운의 빈 줄(`\n\n`)은 새 블록을 생성한다. 이것은 바꿀 수 없다.**
Notion의 블록 모델에 맞춰 설계하되, `<br>`로 억지로 합치려 하지 않는다.

**`<br>` 사용 규칙:**
- **일반 텍스트에서 `<br>` 사용 금지** — Notion이 예측 불가능하게 렌더링함
- **인용블록(`>`)에서만 `<br>` 사용** — 반드시 한 물리적 줄에 작성: `> 줄1<br>줄2<br>줄3`
- **콜아웃 안에서도 `<br>` 지양** — 대신 불릿 리스트나 자식 블록으로 구성

```markdown
❌ 나쁜 예 (일반 텍스트에서 <br> 사용 — 렌더링 깨짐)
Git은 **스냅샷** 방식으로 버전을 저장.<br>
파일을 수정할 때마다 스냅샷을 기록.<br>
스냅샷들이 쌓이면 전체 히스토리 완성.

❌ 나쁜 예 (인용블록 여러 줄 — 각각 별도 quote 블록)
> 💡 **핵심 원리**
> Git은 스냅샷을 기록한다.

✅ 좋은 예 (명사형 + 불릿 리스트로 구조화)
Git의 버전 저장 방식:
- **스냅샷(snapshot)** 방식 — 파일 수정 시 전체 프로젝트 상태를 기록
- 스냅샷이 쌓이면 프로젝트의 **전체 변경 히스토리** 완성
- 각 스냅샷에는 **누가, 언제, 무엇을** 바꿨는지 정보 포함

✅ 좋은 예 (인용블록 한 줄 완결)
> 💡 **핵심 원리** — Git은 스냅샷을 기록하는 도구. 파일 변경 시 전체 상태를 저장하고, 이 기록이 쌓여 프로젝트 히스토리가 됨.
```

**문단 작성 전략:**
- 한 문단(블록) = 2~3문장을 빈 줄 없이 명사형 어미로 이어씀
- 맥락이 바뀌면 빈 줄로 분리하여 새 블록 생성 (의도적)
- **여러 항목 나열은 반드시 불릿/번호 리스트 사용** (텍스트로 나열 금지)
- 관련 정보 그룹화는 콜아웃 사용 (텍스트 블록 합치기 시도 금지)
- 산문 설명보다 **불릿 포인트 + 볼드 키워드**로 스캔 가능하게 작성

### 테이블

**2개 이상의 항목을 비교하거나 정보를 정리할 때 테이블을 적극 사용한다.** 산문으로 비교를 나열하지 않는다.

### 토글 (`<details>`, `{toggle="true"}`)

- **퀴즈 힌트/정답**: `<details><summary>💡 힌트</summary>` 사용
- **실습 단계 (I DO/WE DO/YOU DO)**: `### 🔍 I DO: 제목 {toggle="true"}` 사용
- **color 속성 사용 금지** — 전체 내용에 색상이 적용되는 Notion 버그 발생

## 단락 구성 규칙

- 한 단락은 **최대 3문장**으로 구성한다. 4문장 이상이면 단락을 나눈다.
- 단락 사이에 **빈 줄**을 삽입하여 시각적 여백을 확보한다.
- 새로운 소주제로 넘어갈 때는 **소제목(###)**으로 전환한다.

## 개념 설명 구조 (6단계)

각 "개념 설명" 섹션은 다음 순서로 작성하되, **각 단계를 소제목으로 분리**한다:

1. **왜 이것이 중요한가** — 배경과 동기.
   - **이전에 배운 개념과의 연결**로 시작한다 ("앞서 배운 X를 기억하십시오. 그런데...")
   - 이 개념이 없으면 어떤 문제가 생기는지 **구체적 시나리오**를 보여준다
   - 수강생이 "이걸 왜 배워야 하지?"라는 의문을 가질 수 없을 정도로 동기를 명확히 한다

2. **핵심 원리** — 개념의 작동 원리.
   - 적절한 비유/아날로지가 있으면 활용한다 (비유 작성 규칙 참조)
   - 비유는 수강 대상(audience)이 이미 아는 영역에서 가져온다
   - 대비, 도식, 화살표(→)를 적극 활용한다

3. **실무에서의 의미** — 현업 맥락.
   - "실제 프로젝트에서 이 개념이 어떻게 쓰이는지" 구체적 사례 1~2개
   - 가능하면 실제 오픈소스 프로젝트나 유명 서비스의 사례를 언급한다

4. **다른 접근법과의 비교** — 대안과의 장단점을 **비교 테이블**로 정리한다.

5. **주의사항과 흔한 오해** — 흔한 실수, 함정을 인용블록(`>`)으로 강조한다.
   - **수강생 관점에서 혼란스러울 수 있는 포인트**를 선제적으로 짚는다
   - "여기서 많은 분이 X라고 생각하시는데, 실은 Y입니다" 패턴으로 오해를 교정한다
   - **논쟁적이거나 의견이 갈리는 부분**이 있으면 양쪽 시각을 모두 소개한다
   - 최소 **2개 이상**의 흔한 오해를 다룬다

6. **코드 예제** — "이를 코드로 표현하면:" 전환 후 코드 제시.
   - 올바른 코드와 **잘못된 코드를 대비**하여 차이를 명확히 한다
   - 코드 뒤에 **다음 개념 예고** 문장을 포함한다

### 분량 가이드

| 단계 | 최소 분량 | 포함 필수 요소 |
|------|-----------|---------------|
| 1. 왜 중요한가 | **5~10문장** | 이전 개념 연결, 문제 시나리오 |
| 2. 핵심 원리 | **5~10문장** | 원리 설명, 도식/화살표, 적절한 비유 |
| 3. 실무 의미 | **3~5문장** | 구체적 사례 1~2개 |
| 4. 비교 | 비교 테이블 1개+ | 대안 최소 1개 |
| 5. 주의사항 | **4~8문장** | 흔한 오해 2개+, 혼란 포인트 |
| 6. 코드 예제 | 코드 30줄 이내 | 다음 개념 연결 1~2문장 |

> **분량이 부족하면 slide-master가 빈약한 발표 스크립트를 생성한다.** 가이드는 Source of Truth이므로 "너무 상세한 것"은 없다. 강사가 가이드만 읽어도 수업을 진행할 수 있을 정도로 충분한 내용을 담는다.

## 비유/아날로지 작성 규칙

비유는 추상적 개념을 수강생의 기존 경험에 연결하는 강력한 도구다. 자연스러운 비유가 있을 때 활용한다. 억지스러운 비유는 오히려 혼란을 줄 수 있으므로 무리하게 넣지 않는다.

### 비유 소스 우선순위

1. **수강생 배경(audience)의 기술 경험** — 가장 효과적. audience가 "Java 개발자"면 Java의 유사 기능과 비교
2. **일상생활** — 기술과 무관한 보편적 경험 (식당 주문, 택배 배송, 레고 블록 등)
3. **타 분야 기술** — 프론트엔드/백엔드, 웹/모바일 등 다른 기술 영역

### 비유 작성 패턴

```markdown
### 핵심 원리

**비유로 이해하기**: {비유 대상}을 떠올려 보십시오.
→ {비유 대상의 특징 A}는 {개념의 특징 A}와 같습니다.
→ {비유 대상의 특징 B}는 {개념의 특징 B}와 같습니다.

**실제 동작**: {비유에서 실제로 전환}
→ {개념이 실제로 어떻게 작동하는지}

**비유의 한계**: 다만 {비유 대상}과 달리 {개념}은 {차이점}입니다.
```

> 비유의 한계를 반드시 명시한다. 비유를 100% 동일시하면 오히려 잘못된 이해를 심어준다.

### 좋은 비유 예시

| 개념 | 비유 | 왜 효과적인가 |
|------|------|--------------|
| Props vs State | 식당: Props=주문서(외부), State=재고(내부) | 단방향성과 소유권이 직관적 |
| TypeScript 타입 | 계약서: 함수가 받는 값의 "계약" | 컴파일 타임 검증 개념이 명확 |
| 컴포넌트 | 레고 블록: 조립/분리 가능한 독립 단위 | 재사용성과 조합 개념 전달 |

## 개념 간 연결 규칙

학생이 개별 개념을 고립된 지식으로 암기하지 않도록, **모든 개념을 전후 맥락에 연결**한다.

### 도입부 연결 (Backward Link)

각 개념 설명의 **첫 문장**에서 이전에 배운 개념과의 관계를 명시한다.

패턴:
- "앞서 배운 {이전 개념}에서 {한계/질문}이 있었습니다. {현재 개념}이 바로 그 해답입니다."
- "{이전 개념}으로 {X}를 할 수 있었습니다. 이번에는 한 단계 더 나아가 {Y}를 다룹니다."
- "직전에 {이전 개념}을 봤는데, 여기서 자연스럽게 드는 질문이 있습니다: {질문}. 그 답이 {현재 개념}입니다."

### 마무리 예고 (Forward Link)

각 개념 설명의 **마지막 문장**에서 다음에 배울 개념을 예고한다.

패턴:
- "이 {현재 개념}은 다음에 배울 {다음 개념}의 기반이 됩니다."
- "여기서 사용한 {요소}가 어딘가 불편하게 느껴졌다면, {다음 개념}에서 그 해결책을 보게 됩니다."
- "지금은 {간단한 방식}으로 했지만, {다음 개념}을 배우면 더 우아한 방법이 있다는 걸 알게 됩니다."

> 단, 첫 번째 개념(도입부 연결 불가)과 마지막 개념(예고 불필요)은 예외로 한다.

## 수강생 배경 활용 규칙

`audience` 파라미터는 단순 참고가 아니라 **전체 설명의 톤과 비유를 결정**하는 핵심 변수다.

### 활용 지점

| 지점 | 방법 |
|------|------|
| 개념 도입 | 수강생이 이미 아는 유사 기능과 비교하며 시작 |
| 비유 선택 | 수강생의 기술 스택에서 비유 소재를 가져온다 |
| 주의사항 | 수강생 배경에서 **특히** 빠지기 쉬운 함정을 강조 |
| Q&A | 수강생 배경에서 나올 법한 질문을 선제 포함 |
| 코드 예제 | 수강생이 익숙한 패턴으로 먼저 보여주고, 새로운 패턴으로 전환 |

### 배경별 설명 전략 예시

```markdown
<!-- audience: "Java 개발자" -->
"Java의 interface를 아시죠? TypeScript의 interface도 이름은 같지만 의미가 다릅니다.
Java interface는 메서드 시그니처만 정의하지만, TypeScript interface는 속성까지 정의합니다.
오히려 Java의 record나 POJO에 더 가깝습니다."

<!-- audience: "Python 개발자" -->
"Python에서 dict에 아무 키나 넣을 수 있는 것 기억하시죠?
TypeScript의 interface는 '이 dict에는 이 키만 이 타입으로 넣을 수 있다'고 미리 선언하는 겁니다.
Python의 TypedDict와 가장 비슷합니다."
```

> audience가 "개발 경험 없음"이면 일상생활 비유를 우선 사용하고, 코드 비유는 최소화한다.

## 금지 패턴

다음 패턴이 발견되면 즉시 수정한다.

### 문장 수준

| 금지 패턴 | 이유 | 교정 방법 |
|-----------|------|-----------|
| "~하고, ~하며, ~한다" | 복문 연쇄 → 가독성 저하 | 각각 독립 문장으로 분리 |
| "~인데, ~이므로, ~하기 때문에" | 종속절이 길어짐 | 원인/결과를 별도 문장으로 |
| 60자 초과 문장 | 한눈에 안 들어옴 | 분리하거나 불릿으로 전환 |
| 부연 설명이 2줄 이상 | 산문이 됨 | 테이블 또는 불릿으로 구조화 |

### 구조 수준

| 금지 패턴 | 이유 | 교정 방법 |
|-----------|------|-----------|
| 4문장 이상 연속 단락 | 텍스트 벽 | 3문장마다 단락 분리 |
| 소제목 없이 500자 이상 | 스캔 불가 | 소제목(###) 삽입 |
| 비교 설명을 산문으로 | 대비가 안 됨 | 비교 테이블로 전환 |
| 나열을 문장 안에 포함 | 항목이 묻힘 | 불릿 리스트로 분리 |

## 좋은 예 vs 나쁜 예

### ❌ 나쁜 예

```markdown
⚠️ 버전 관리 없이 겪는 문제 3가지

1️⃣ **최신본 구분 불가**

→ app_final.py, app_final2.py 중 어느 것이 맞는지 불분명

2️⃣ **변경 기록 없음**

→ "어제까지 됐는데 오늘 갑자기 안 된다" — 원인 추적 불가

Git은 **스냅샷(snapshot)** 방식으로 버전을 저장합니다.<br>
파일을 수정할 때마다 스냅샷을 기록합니다.<br>
스냅샷들이 쌓이면 전체 히스토리가 완성됩니다.
```

**문제점:**
- 모든 문장이 별도 블록 → 노션에서 산만하게 표시
- 이모지 번호(1️⃣2️⃣)가 네이티브 리스트를 대체 → 부자연스러움
- `<br>` 일반 텍스트에서 사용 → 렌더링 깨짐
- 대화체(`~합니다`) 연속 → 정보 밀도 낮음
- 콜아웃 미사용 → 정보 그룹화 부재

### ✅ 좋은 예 (v5 확정 스타일)

```markdown
코드를 처음 작성하면 파일 하나로 시작. 그런데 며칠이 지나면 `app_final.py`, `app_final2.py`, `app_진짜최종.py` 같은 파일이 쌓이기 시작.

<callout icon="⚠️" color="yellow_bg">
	**버전 관리 없이 겪는 문제 3가지**
	1. **최신본 구분 불가** — 여러 복사본 중 어느 것이 진짜 최신인지 알 수 없음
	2. **변경 기록 부재** — "어제까지 됐는데 오늘 갑자기 안 된다" 상황에서 원인 추적 불가
	3. **복구 불가** — 실수로 삭제/잘못 고쳤을 때 되돌릴 방법 없음
</callout>

특히 세 번째 문제가 치명적임. 한 번의 실수로 며칠치 작업이 사라지는 경험을 하고 나서야 버전 관리의 필요성을 절감하게 됨.

> 💡 버전 관리 없이 협업하면? — 두 명이 같은 파일 수정 시 한 사람의 작업이 덮어씌워짐. 규모가 커질수록 심각해짐.

<callout icon="💡" color="blue_bg">
	**Git = 이 세 가지 문제를 모두 해결하는 도구**
	- ✅ 변경 이력 자동 기록 — 언제, 무엇이, 왜 바뀌었는지 추적 가능
	- ⏪ 과거 상태로 즉시 복구 가능 — 실수를 되돌리는 안전망
	- 👥 동시 작업 시 충돌 관리 — 협업의 필수 인프라
</callout>

Git의 버전 저장 방식:
- **스냅샷(snapshot)** 방식 — 파일 수정 시 전체 프로젝트 상태를 기록
- 스냅샷이 쌓이면 프로젝트의 **전체 변경 히스토리** 완성
- 각 스냅샷에는 **누가, 언제, 무엇을** 바꿨는지 정보 포함

![Git 스냅샷 방식](https://git-scm.com/book/en/v2/images/snapshots.png)
```

**핵심 차이 (v5 확정 스타일의 원칙):**
- **명사형 어미** — "~시작.", "~불가", "~가능", "~포함" (대화체 `~합니다` 금지)
- **콜아웃으로 그룹화** — 경고(⚠️), 인사이트(💡), 정리(✅) 등 용도별 사용
- **불릿 리스트 적극 활용** — 나열 항목은 항상 리스트로 구조화
- **인용블록 한 줄 완결** — `>` 줄이 여러 개이면 별도 블록이 되므로 반드시 한 줄로
- **`<br>` 일반 텍스트 금지** — 인용블록 안에서만 사용, 한 물리적 줄에 작성
- **자연스러운 문단** — 2~3문장을 빈 줄 없이 이어쓰기, 맥락 변경 시에만 빈 줄

## 코드 및 실행 인용 규칙

개념을 충분히 설명한 후, **코드, 실행 결과, UI 화면으로 보강**한다.

### 원칙

1. **설명 → 개념 → 코드**: 충분한 텍스트 설명 후 코드를 제시한다. 코드만으로 개념을 전달하지 않는다.
2. **코드 → 실행 결과**: 코드를 보여준 후 실행 결과(터미널 출력, 로그, UI 스크린샷)를 함께 제시한다.
3. **비교 예시**: 올바른 코드와 잘못된 코드를 대비하여 차이를 명확히 한다.
4. **점진적 빌드업**: 작은 코드에서 시작하여 점점 기능을 추가하는 방식으로 구성한다.
5. **코드 간결성**: 하나의 코드 블록은 30줄 이내를 권장. 긴 코드는 단계별로 나눈다.
