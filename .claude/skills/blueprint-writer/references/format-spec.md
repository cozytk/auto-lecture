# Blueprint 포맷 스펙

Blueprint는 수업의 **모든 구조적 결정**을 한 곳에 담는 축약 문서다. 산문이 아닌 불릿 포인트로 구성하며, 1일 과정 기준 200~400줄을 목표로 한다.

---

## 문서 구조

### 헤더 + META

```markdown
# {과정 제목} Blueprint
<!-- blueprint-version: 1.0 -->
<!-- status: draft | approved | locked -->

## META
- **주제**: {topic}
- **수강 대상**: {audience description, 사전 지식 수준 포함}
- **사전 요구사항**: {필수 설치/지식}
- **기간**: {N}일 ({hours}시간)
- **비율**: 강의 {X}% / 실습 {Y}%
- **실행 환경**: {Colab | Local(Linux) | WSL2 | Docker}
```

**status 값:**
- `draft`: 작성/수정 중. 산출물 생성 불가.
- `approved`: 사용자 승인 완료. 산출물 생성 가능.
- `locked`: 산출물 생성 완료 후 잠금. 수정 시 명시적 unlock 필요.

---

### 학습 목표

```markdown
## 학습 목표
1. {동사로 시작하는 구체적 목표 — 블룸 택소노미 기반}
2. ...
3. ...
```

- 3~5개. 각 목표는 "~할 수 있다" 형태.
- 모든 목표에 대응하는 실습 블록이 최소 1개 필요.

---

### DAY 구조

```markdown
## DAY {N}
```

하루는 여러 **세션**으로 구성. 각 세션은 여러 **블록**으로 구성.

---

### 세션

```markdown
### [{HH:MM}-{HH:MM}] {세션 제목} ({duration}분)
<!-- session: d{N}s{M} -->
**학습 목표**: #{번호}, #{번호}
```

- 세션 ID: `d{day}s{session}` (예: `d1s1`, `d1s2`, `d2s3`)
- 시간은 24시간 형식
- 세션당 최대 120분. 초과하면 분리.
- 학습 목표 번호를 참조하여 세션-목표 매핑을 명시

---

### 블록 (핵심 단위)

블록은 세션 안의 개별 활동 단위. **강의 블록**과 **실습 블록** 두 가지 유형.

#### 강의 블록

```markdown
#### 블록 {K}: {제목} ({duration}분) [강의]
<!-- block: d{N}s{M}b{K} -->
- **핵심 메시지**: {반드시 1문장. 슬라이드 섹션 제목의 기반}
- **개념**: {개념A}, {개념B}, {개념C}
- **교수법**: {비유→원리→실무 사례 | 데모→분석→정리 | 코드 워크스루 | ...}
- **슬라이드**: {N}장, 시각: {Mermaid 흐름도 | HTML 다이어그램 | 이미지 | 코드 블록 | 비교표}
- **핵심 비유**: "{비유 한 줄}" {비유가 적절하지 않은 개념은 생략 가능}
- **혼동 주의**: {흔히 헷갈리는 포인트 — 없으면 생략}
```

**필드 규칙:**
- `핵심 메시지`: 1문장 필수. "X는 Y다" 형태. 학생이 이 블록에서 기억해야 할 1가지.
- `개념`: 이 블록에서 다루는 개념 목록. guide-writer가 각 개념을 6단계 구조로 확장.
- `교수법`: 이 개념을 어떻게 가르칠지의 전략. slide-master가 이를 기반으로 슬라이드 구성.
- `슬라이드`: 장수는 가이드라인. slide-master가 ±2장 범위에서 조정 가능. 시각 요소는 반드시 포함.
- `핵심 비유`: 학생의 기존 경험에 연결하는 비유. guide-writer가 산문으로 확장.
- `혼동 주의`: 학생이 틀리기 쉬운 포인트. Q&A와 별도 — 블록 내 개념 간 혼동.

#### 실습 블록

```markdown
#### 블록 {K}: {실습 제목} ({duration}분) [실습]
<!-- block: d{N}s{M}b{K} -->
- **실습 유형**: 코드 | README 중심
- **난이도**: 기초 | 중급 | 심화
- **연관 학습 목표**: #{N}
- **I DO** ({M}분): {강사가 시연할 내용 — 1줄}
- **WE DO** ({M}분): {함께 할 내용 — 1줄}
- **YOU DO** ({M}분): {독립 과제 — 1줄}
- **기술 스택**: {Python, Docker, Node.js, ...}
- **검증 방법**: {테스트 실행 | 산출물 확인 | 발표 | 코드 리뷰}
```

**필드 규칙:**
- `실습 유형`: 코드 실습은 `Justfile + src/ + solution/`, README 중심은 `README + artifacts/`
- `I DO / WE DO / YOU DO`: 각각의 시간 합 = 블록 전체 시간
- `YOU DO`는 `WE DO`의 단순 반복이 아닌 새로운 요구사항/제약이 추가된 과제
- `검증 방법`: lab-manager가 이를 기반으로 검증 방식 결정

---

### 쉬는 시간

```markdown
### [{HH:MM}-{HH:MM}] 쉬는 시간 (15분)
```

- 2시간마다 최소 15분 휴식
- block-id 불필요

---

### Q&A Bank (세션별)

```markdown
#### Q&A Bank
| # | 유형 | 질문 | 핵심 답변 (1줄) |
|---|------|------|-----------------|
| 1 | 흔한 오해 | {question} | {answer} |
| 2 | 트러블슈팅 | {question} | {answer} |
| 3 | 논쟁적 | {question} | {answer} |
```

**유형:**
- `흔한 오해`: 학생이 잘못 이해하기 쉬운 포인트
- `트러블슈팅`: 실습 중 자주 발생하는 에러/문제
- `논쟁적`: 커뮤니티에서 의견이 갈리는 주제
- `배경별`: 수강생의 기존 경험에서 비롯되는 질문
- `심화`: 고급 학생을 위한 추가 질문

**규칙:**
- 세션당 최소 3개
- 핵심 답변은 1줄 (guide-writer가 3~5문장으로 확장)
- script-writer가 Q&A를 슬라이드 대본에 자연스럽게 포함

---

### 혼동 포인트 (세션별)

```markdown
#### 혼동 포인트
- **{개념 A vs 개념 B}**: {왜 혼동하는지} → {어떻게 구분하는지}
- **{용어 X vs 용어 Y}**: {차이점} → {기억하는 팁}
```

- 세션에서 다루는 개념 중 학생이 혼동하기 쉬운 쌍을 명시
- 없으면 섹션 자체를 생략

---

### 슬라이드 설계 노트 (전체)

```markdown
## 슬라이드 설계 노트
- 총 예상 슬라이드: ~{N}장
- 텍스트 전용 3장 연속 금지 — 시각 요소 배치 확인
- 커버 슬라이드: "{제목 문구}"
- 섹션 구분 슬라이드: {각 세션 시작에 배치}
- 특별 슬라이드: {있다면 — 라이브 데모, 참여형 퀴즈 등}
```

- slide-master가 이 노트를 슬라이드 전체 구성의 가이드라인으로 사용
- 블록별 슬라이드 수의 합 ≈ 총 슬라이드 수

---

### 실습 총괄 (전체)

```markdown
## 실습 총괄
| # | 실습명 | 세션 | 형식 | 난이도 | 시간 | block |
|---|--------|------|------|--------|------|-------|
| 1 | {name} | d1s2 | 코드 | 기초 | 40분 | d1s2b3 |
| 2 | {name} | d1s3 | README | 중급 | 30분 | d1s3b2 |
총 실습: {X}분 / 전체: {Y}분 ({Z}%)
```

- 모든 실습 블록을 한눈에 볼 수 있는 요약 테이블
- lab-manager가 이 테이블을 실습 제작의 입력으로 사용
- 비율은 반드시 65~75% 범위

### 리서치 요약 (선택)

```markdown
## 리서치 요약
- **공식 문서 핵심**: {key finding}
- **트렌드**: {key industry trend}
- **반영 포인트**: {what to emphasize}
```

- `research-brief.md`의 핵심만 축약
- 상세 내용은 `research-brief.md` 참조

## Block-ID 규칙

### ID 형식

`d{day}s{session}b{block}` — 모든 소문자, 구분자 없음.

| 예시 | 의미 |
|------|------|
| `d1s1b1` | Day 1, Session 1, Block 1 |
| `d1s2b3` | Day 1, Session 2, Block 3 |
| `d3s1b2` | Day 3, Session 1, Block 2 |

### 산출물에서의 참조

하위 에이전트(guide-writer, slide-master, lab-manager)가 생성하는 산출물에 source 코멘트를 삽입:

**slides.md:**
```markdown
---
layout: default
---
<!-- source: d1s1b1 -->
# 슬라이드 제목
```

**guide session file:**
```markdown
## 2. {개념 제목}
<!-- source: d1s1b1 -->
```

**lab README:**
```markdown
# {실습 제목}
<!-- source: d1s2b3 -->
```

이 추적 코멘트로 피드백 시 blueprint의 어떤 블록을 수정해야 하는지 즉시 파악.

## 예시: 1일 Git 기초 과정

```markdown
# Git 기초 Blueprint
<!-- blueprint-version: 1.0 -->
<!-- status: approved -->

## META
- **주제**: Git 기초
- **수강 대상**: 주니어 개발자 — 코딩 경험 있으나 버전관리 초보. CLI 기본 가능 (cd, ls, mkdir)
- **사전 요구사항**: Git 설치 완료, 터미널 기본 조작, 텍스트 에디터
- **기간**: 1일 (1시간)
- **비율**: 강의 30% / 실습 70%
- **실행 환경**: Local (Linux/macOS)

## 학습 목표
1. 버전관리의 필요성을 설명하고 Git의 3영역(Working Directory, Staging, Repository)을 구분할 수 있다
2. git init, add, commit, status, log 명령어로 로컬 저장소를 관리할 수 있다
3. 브랜치를 생성·전환하고 merge로 병합할 수 있다

## DAY 1

### [09:00-10:00] Git 기초: 버전관리부터 브랜치까지 (60분)
<!-- session: d1s1 -->
**학습 목표**: #1, #2, #3

#### 블록 1: 왜 버전관리인가 (5분) [강의]
<!-- block: d1s1b1 -->
- **핵심 메시지**: 버전관리 없이 협업하면 "최종_진짜최종_v3" 비극이 반복된다
- **개념**: 버전관리 필요성, Git의 탄생 배경
- **교수법**: 문제 상황 제시 → Git이 해결하는 방법
- **슬라이드**: 3장, 시각: 파일명 혼란 이미지 + Git 타임라인
- **핵심 비유**: "게임 세이브 포인트 — 언제든 과거로 돌아갈 수 있다"

#### 블록 2: Git 3영역과 기본 명령어 (10분) [강의]
<!-- block: d1s1b2 -->
- **핵심 메시지**: Working Directory → Staging Area → Repository, 3단계를 이해하면 Git의 80%를 안다
- **개념**: Working Directory, Staging Area, Repository, git init, git add, git commit, git status, git log
- **교수법**: 다이어그램 → 각 영역 설명 → 명령어 데모
- **슬라이드**: 5장, 시각: Mermaid 흐름도 (3영역) + 코드 블록 (명령어)
- **핵심 비유**: "택배 포장 — 물건 준비(WD) → 박스에 담기(Staging) → 발송(Commit)"
- **혼동 주의**: add는 "파일 추가"가 아니라 "변경사항을 스테이징"하는 것

#### 블록 3: Git 기본 실습 (25분) [실습]
<!-- block: d1s1b3 -->
- **실습 유형**: 코드
- **난이도**: 기초
- **연관 학습 목표**: #1, #2
- **I DO** (5분): 빈 디렉토리에서 git init → 파일 생성 → add → commit → log 시연
- **WE DO** (10분): 학생이 따라하며 3개 파일을 순차 커밋. 각 단계에서 git status 확인
- **YOU DO** (10분): README.md 작성 + .gitignore 설정 + 3회 이상 커밋 이력 생성
- **기술 스택**: Git CLI
- **검증 방법**: git log로 커밋 이력 3개 이상 확인

#### 블록 4: 브랜치와 병합 (5분) [강의]
<!-- block: d1s1b4 -->
- **핵심 메시지**: 브랜치는 독립된 작업 공간 — 실험하고 실패해도 main은 안전하다
- **개념**: branch, checkout/switch, merge, 충돌 해결
- **교수법**: 시각화 → 명령어 → 충돌 시나리오
- **슬라이드**: 3장, 시각: Mermaid 브랜치 다이어그램 + 충돌 코드 예시
- **핵심 비유**: "포스트잇 — 여러 아이디어를 동시에 메모하고 나중에 합치기"

#### 블록 5: 브랜치 실습 (15분) [실습]
<!-- block: d1s1b5 -->
- **실습 유형**: 코드
- **난이도**: 기초~중급
- **연관 학습 목표**: #3
- **I DO** (3분): feature 브랜치 생성 → 수정 → main으로 merge 시연
- **WE DO** (5분): 함께 브랜치 만들고 수정 후 merge. 충돌 없는 케이스
- **YOU DO** (7분): 의도적 충돌 생성 → 수동 해결 → merge 완료
- **기술 스택**: Git CLI
- **검증 방법**: git log --oneline --graph로 브랜치 히스토리 확인

#### Q&A Bank
| # | 유형 | 질문 | 핵심 답변 |
|---|------|------|-----------|
| 1 | 흔한 오해 | git add .은 모든 파일을 추가하니까 항상 써도 되나요? | .gitignore에 없는 파일도 모두 추가되므로, 개별 파일 지정이 안전. 민감 파일 유출 위험 |
| 2 | 트러블슈팅 | merge 충돌이 나면 어떻게 해요? | <<<< ==== >>>> 마커 사이에서 원하는 코드를 남기고 마커 삭제 후 add+commit |
| 3 | 배경별 | SVN 경험이 있는데 Git과 뭐가 다른가요? | Git은 분산형 — 로컬에 전체 이력이 있어 오프라인 작업 가능. 브랜치가 가볍고 빠름 |

#### 혼동 포인트
- **git add vs git commit**: add는 스테이징(준비), commit은 확정(저장). add 없이 commit하면 변경사항이 반영되지 않음
- **main vs master**: Git 2.28+ 기본 브랜치명은 main. 구버전은 master. 동일한 역할

## 슬라이드 설계 노트
- 총 예상 슬라이드: ~15장
- 텍스트 전용 3장 연속 금지
- 커버 슬라이드: "Git 기초 — 버전관리의 시작"
- 섹션 구분: 개요 | 3영역 | 기본 명령어 | 브랜치

## 실습 총괄
| # | 실습명 | 세션 | 형식 | 난이도 | 시간 | block |
|---|--------|------|------|--------|------|-------|
| 1 | Git 기본 워크플로우 | d1s1 | 코드 | 기초 | 25분 | d1s1b3 |
| 2 | 브랜치와 병합 | d1s1 | 코드 | 기초~중급 | 15분 | d1s1b5 |
총 실습: 40분 / 전체: 60분 (67%)
```

## 분량 가이드라인

| 과정 기간 | Blueprint 목표 줄 수 | 세션 수 | 블록 수 |
|-----------|---------------------|---------|---------|
| 1시간 | 80~120줄 | 1 | 3~5 |
| 1일 (8시간) | 200~400줄 | 3~5 | 15~25 |
| 3일 (24시간) | 500~900줄 | 9~15 | 40~70 |
| 5일 (40시간) | 800~1500줄 | 15~25 | 60~120 |

블록당 평균 5~8줄. 이보다 길어지면 블록을 분리하거나 내용을 축약한다.
