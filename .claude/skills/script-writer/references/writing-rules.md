# 발표자 대본 작성 규칙

이 문서는 script-writer가 대본을 작성할 때 따라야 하는 모든 규칙을 담고 있다.
병렬 워커(Agent subagent 또는 Codex)에게 태스크를 분배할 때, 이 문서의 **전문**을 태스크에 포함해야 한다.

---

## 대본 형식

모든 슬라이드에 HTML 주석(`<!-- -->`)으로 대본을 작성한다.
반드시 영어 `[click]`을 사용한다 (`[클릭]` 금지).

```markdown
<!--
[스크립트]
{도입부: 이전 슬라이드/개념과의 연결}

{본문: 슬라이드에 보이는 모든 시각 요소를 빠짐없이 설명.
완성된 문장으로, 구어체로 작성.
코드 블록은 줄 단위 해설 포함.}

[click] {v-click 단계 1에서 새로 나타나는 내용의 완전한 설명. 최소 3문장.}

💡 여기서 잠깐 — {헷갈릴 수 있는 포인트를 선제적으로 짚는다}
"이 부분에서 많은 분이 ~라고 생각하시는데, 실은 ~입니다."

[click] {v-click 단계 2 설명. 최소 3문장.}

[Q&A 대비]
Q: {예상 질문 1 — 자주 묻는 질문}
A: {3~5문장 답변. 이유와 맥락 포함}

Q: {예상 질문 2 — 수강생 배경에서 나올 질문}
A: {3~5문장 답변. 익숙한 개념과 비교}

전환: {다음 슬라이드로의 연결 멘트. 다음 개념이 왜 필요한지 1문장으로.}
시간: X분
-->
```

---

## 대본 작성 원칙

### 1. 도입부 — 이전 슬라이드와의 연결

모든 슬라이드의 대본은 **이전에 다룬 내용과의 연결**로 시작한다.

| 슬라이드 위치 | 도입부 패턴 |
|---|---|
| 첫 번째 슬라이드 | 인사 + 수업 소개 |
| 개념 연속 | "방금 X를 배웠습니다. 그런데 여기서 한 가지 의문이 생깁니다..." |
| 새 섹션 시작 | "지금까지 X 파트를 마쳤습니다. 이제 Y로 넘어갑니다. Y가 왜 필요하냐면..." |
| 실습 전 | "개념은 충분히 다뤘습니다. 이제 직접 해봐야 진짜 이해가 됩니다." |
| 실습 후 | "실습을 하면서 ~를 경험하셨을 겁니다. 이제 그 원리를 정리합니다." |

- **금지**: "다음으로 넘어가겠습니다", "자, 이제..." 같은 단조로운 전환
- **필수**: 다음 개념이 **왜 필요한지**를 1문장으로 설명

### 2. 본문 — 슬라이드 시각 요소의 완전한 설명

슬라이드에 보이는 **모든 시각 요소**를 빠짐없이 설명한다.

| 시각 요소 | 설명 방법 |
|---|---|
| 코드 블록 | **줄 단위 해설**. "첫 번째 줄을 보시면...", "3번째 줄에서..." |
| 비교 테이블 | 각 행/열을 하나씩 짚으며 차이점 부각 |
| Mermaid 다이어그램 | 흐름/관계를 구두로 따라가며 설명 |
| 이미지 | 이미지가 보여주는 핵심 포인트 2~3개 설명 |
| 불릿 리스트 | 각 불릿의 의미와 맥락을 풀어서 설명 |
| 강조 박스 (box-*) | "화면 하단/오른쪽에 정리된 내용을 보시면..." |
| 수치/통계 | 수치의 의미, "이 숫자가 의미하는 것은..." |

#### 코드 설명 깊이

코드 블록은 단순 요약이 아니라 **강사가 코드를 한 줄씩 가리키며 설명하는 수준**으로 작성한다.

```
"자, 코드를 보겠습니다.
첫 번째 줄, `const userName: string = "홍길동"`입니다.
변수명 `userName` 뒤에 콜론(:)을 붙이고 `string`이라고 적었습니다.
이것이 타입 어노테이션입니다. '이 변수에는 문자열만 넣겠다'고 선언하는 겁니다.

그 아래 `const userAge: number = 28`을 보십시오.
Java에서는 int, float, double을 구분하지만, TypeScript는 정수와 실수를 모두 number 하나로 처리합니다.
Java 개발자분들은 이 점이 좀 어색할 수 있습니다."
```

### 3. [click] 설명 — v-click 단계별 충분한 설명

각 [click] 마커 뒤에는 **해당 단계에서 새로 나타나는 내용에 대한 완전한 설명**을 포함한다.

- [click] 수는 반드시 슬라이드의 v-click/v-clicks 단계 수와 일치한다
- 각 [click] 설명은 **최소 3문장** 이상
- 코드 라인 하이라이트 전환이면 새로 하이라이트되는 코드의 줄 단위 해설 포함

### 4. 헷갈림 포인트 — 대본 흐름에 자연스럽게 삽입

가이드의 "주의사항과 흔한 오해" 섹션 + Q&A 섹션의 내용을 **대본 본문 안에** 녹인다.
별도 섹션으로 분리하지 않고, 관련 개념을 설명하는 도중에 자연스럽게 끼워넣는다.

**삽입 화법**:

```
"여기서 많은 분이 헷갈려하시는 부분이 있습니다.
[혼란 포인트를 설명]
직관적으로는 ~라고 생각하시겠지만, 실제로는 ~입니다.
왜냐하면 [이유]."
```

**유형별 삽입 패턴**:

| 유형 | 삽입 위치 | 화법 |
|---|---|---|
| 흔한 오해 | 개념 설명 직후 | "여기서 흔히 하는 오해가 있습니다..." |
| 용어 혼동 | 새 용어 도입 시 | "이름이 같지만 의미가 다릅니다. Java의 X와..." |
| 코드 실수 | 코드 설명 중 | "이렇게 하면 동작은 하지만, 문제가 될 수 있습니다..." |
| 수강생 배경 충돌 | 유사 개념 비교 시 | "{audience}에서는 ~했을 텐데, 여기서는 달라요..." |

> 헷갈림 포인트를 삽입할 때 `💡 여기서 잠깐 —` 마커를 사용한다. 이 마커는 강사가 대본을 읽다가 "여기서 청중의 혼란을 짚어줘야 한다"는 신호로 쓴다.

### 5. Q&A 대비 — 각 슬라이드별 최소 2개

가이드의 Q&A 항목 중 해당 슬라이드와 관련된 것을 선별하고, 추가로 보충한다.

**Q&A 유형 (우선순위)**:
1. **자주 묻는 질문** — 해당 개념을 가르칠 때 반복적으로 나오는 질문 (필수 1개)
2. **수강생 배경 질문** — audience의 기존 경험에서 비롯되는 질문 (필수 1개)
3. **논쟁적 포인트** — 커뮤니티에서 의견이 갈리는 주제 (해당 시)
4. **실무 적용 질문** — "실제 프로젝트에서는 어떻게 쓰나요?" 류 (해당 시)

**답변 작성 규칙**:
- 3~5문장. 이유와 맥락 포함
- "좋은 질문입니다" 같은 빈말 금지 — 바로 핵심 답변
- 가능하면 코드나 구체적 예시 포함
- 수강생이 이미 아는 개념과 비교하며 설명

**Q&A 대비는 대본의 부록이다**: 실제 발표에서 읽는 부분이 아니라, 질문이 들어왔을 때 참고하는 부분이다. 따라서 `[Q&A 대비]` 마커 아래에 위치하며, 시간 산정에 포함하지 않는다.

### 6. 전환 멘트 — 다음 개념의 필요성을 예고

| 금지 | 필수 |
|---|---|
| "다음으로 넘어가겠습니다" | "지금 X에서 ~한 불편함을 느꼈을 겁니다. 바로 그 해결책이 다음에 나옵니다" |
| "자, 이제 Y를 보겠습니다" | "X만으로는 ~한 한계가 있습니다. Y가 그 한계를 어떻게 극복하는지 보겠습니다" |
| "그럼 계속하겠습니다" | "방금 배운 X를 기반으로, 이제 한 단계 더 들어갑니다" |

### 7. 시간 산정 — 현실적 기준

**기본 공식**:

| 요소 | 시간 기준 |
|---|---|
| 대본 텍스트 | 1분당 약 200자 (한국어 기준) |
| 코드 설명 | 코드 5줄당 약 2분 |
| v-click 단계 | 단계당 최소 30초 |
| Q&A 대비 | **산정에 포함하지 않음** (참고용) |
| 섹션 전환 슬라이드 | 30초 |
| 퀴즈 슬라이드 | 문제당 2~3분 |

**보정 규칙**:
- 계산된 시간의 **1.3배**를 적용한다 (청중 반응, 부연 설명, 즉석 질문 대응)
- 각 세션의 슬라이드 시간 합계가 세션 할당 시간의 ±10% 이내인지 검증
- 초과 시 대본을 줄이거나 슬라이드 분할을 제안 (직접 슬라이드를 수정하지 않는다)

## 슬라이드 유형별 대본 가이드

### 커버/섹션 슬라이드
- 짧고 에너지 있게. 30초~1분
- 이번 섹션에서 다룰 내용을 예고
- 이전 섹션 마무리 + 연결

### 개념 설명 슬라이드
- 가이드의 6단계 설명(왜 중요한가, 핵심 원리, 실무 의미, 비교, 주의사항, 코드)을 구어체로 변환
- 비유/아날로지를 적극 활용 — 가이드에 비유가 있으면 반드시 대본에 포함
- 💡 헷갈림 포인트를 본문에 삽입
- Q&A 최소 2개

### 코드 슬라이드
- **줄 단위 해설** 필수 — 가장 중요한 슬라이드 유형
- 라인 하이라이트 전환에 맞춰 [click]으로 단계 구분
- 설명 패턴: "이 코드가 하는 일은..." → "한 줄씩 보겠습니다..." → "실행하면..."
- 실행 결과 예측 → 실제 결과 확인 패턴 활용
- 잘못된 코드 예시가 있으면 **왜 잘못되었는지** 구체적으로 설명

### 비교 슬라이드
- 좌우(또는 행) 대비를 구두로 명확히
- "왼쪽을 보시면... 반면 오른쪽은..."
- 각 항목의 차이가 **왜 중요한지** 설명
- 어떤 상황에서 어느 쪽을 선택해야 하는지 실무 가이드라인 포함

### 퀴즈 슬라이드
- 문제 읽기 → "30초 생각해 보십시오" → [click]으로 정답 공개
- 정답 공개 후 **왜 그 답인지** 충분히 설명
- 오답이 왜 오답인지도 설명 (오답을 고른 수강생의 사고 흐름을 존중)

### 실습 안내 슬라이드
- 실습 목표와 예상 소요 시간 안내
- "지금부터 N분간 실습합니다. 완료하신 분은 ~를 추가로 해보세요"
- 흔히 막히는 부분 사전 안내: "실습 중 가장 자주 막히는 부분은 ~입니다. 이런 오류가 나면 ~를 확인해 보세요"

### 휴식/전환 슬라이드
- 10~15초. 간결하게
- "N분 쉬겠습니다. 돌아오시면 바로 ~를 진행합니다."

## 가이드 활용 규칙

가이드는 **대본의 원재료**다. 가이드의 내용을 그대로 옮기는 것이 아니라, 발표용 구어체로 **변환**한다.

| 가이드 요소 | 대본에서의 활용 |
|---|---|
| 왜 이것이 중요한가 | 도입부에서 동기 부여 멘트로 변환 |
| 핵심 원리 + 비유 | 본문의 핵심 설명. 비유를 구어체로 풀어서 전달 |
| 실무에서의 의미 | "실제 프로젝트에서는..." 사례 삽입 |
| 다른 접근법과의 비교 | 비교 슬라이드 또는 코드 슬라이드의 보충 설명 |
| 주의사항과 흔한 오해 | 💡 헷갈림 포인트로 본문에 삽입 |
| Q&A | [Q&A 대비] 섹션에 선별 포함 + 일부는 본문에 삽입 |
| 코드 예제 | 코드 슬라이드의 줄 단위 해설 원재료 |
| 개념 간 연결 (Forward/Backward Link) | 도입부와 전환 멘트에 활용 |
| **리서치 브리프** (research-brief.md) | 현재 트렌드, 찬반 논거, 실제 사례를 대본에 삽입. "최근 업계에서는...", "흥미로운 논쟁이 있는데요..." 화법으로 활용. [Q&A 대비]에 트렌드 관련 질문 추가 |

### 리서치 브리프 활용

`lectures/{topic}/guide/research-brief.md`가 존재하면 적극 활용한다.

| 리서치 항목 | 대본 활용 방법 |
|---|---|
| 찬반 논거 | 💡 헷갈림 포인트에서 "업계에서도 의견이 갈리는데요..." 패턴으로 삽입 |
| 최근 동향 | 도입부 또는 본문에 "최근 ~에서 ~를 발표했는데..." 패턴으로 삽입 |
| 채택 사례 | "실제로 ~회사에서는..." 패턴으로 실무 설명 보강 |
| 비판/회의론 | [Q&A 대비]에 "~라는 비판도 있는데 어떻게 생각하나요?" 형태로 추가 |

### 가이드에 없는 내용

가이드에 충분한 설명이 없는 슬라이드가 있을 경우:
1. 가이드의 전체 맥락에서 추론하여 대본을 작성한다
2. 추론이 어려우면 WebSearch로 보충 자료를 찾는다
3. 보충한 내용은 대본에 자연스럽게 통합한다

## 구어체 변환 규칙

가이드의 문어체를 **강의실에서 강사가 실제로 하는 말투**로 변환한다.

| 문어체 (가이드) | 구어체 (대본) |
|---|---|
| "~이다", "~한다" | "~입니다", "~합니다" |
| "다음과 같다" | "이렇게 생겼습니다", "한번 보시죠" |
| "주의해야 한다" | "여기서 조심하셔야 합니다" |
| "~라고 할 수 있다" | "~라고 보시면 됩니다" |
| "비교하면 다음과 같다" | "둘을 비교해 보겠습니다" |

### 금지 패턴

| 금지 | 이유 |
|---|---|
| "이 슬라이드에서는..." | 메타 발화 — 청중은 슬라이드를 보고 있다 |
| "~에 대해 설명하겠습니다" | 예고만 하고 설명을 미루는 느낌 → 바로 설명 시작 |
| "아시다시피" | 모른다고 가정해야 한다 |
| "간단히 말하면" | 간단하지 않은 내용을 이 말로 축소하면 안 된다 |
| "당연히" | 수강생에게 당연한 것은 없다 |
